终章 · 为什么是这套架构

一条贯穿全系统的分工线：
LLM 决主观，算法决客观。

LLM 负责 · 主观品味
出蓝图 / 段集合 / 权重 / 语义打分——"这家亲子餐厅适不适合 5 岁娃"这类没有标准答案、要靠品味判断的事。
算法 / critic 负责 · 客观事实
验时序 / 时长 / 距离 / 营业时间 / 约束——"19:00 闭馆你却排了 20:00"这类有唯一对错、LLM 一本正经也会算错的事。

这不是一句口号，而是逼着整个系统从 rule 一路演进到 LangGraph Plan-and-Execute 的设计哲学。 每当 LLM 越界去做客观计算（写死 5 段、自己算距离），就翻车；每当算法越界去替 LLM 拍板品味（用 if 枚举段集合），就接不住开放输入。 下面这条时间线，记的就是每一次"越界翻车 → 把活儿还给对的一方"。

理论依据 · LLM-Modulo（Kambhampati, NeurIPS 2024, arXiv 2402.01817）：LLM 当 idea generator，外部 critic 做 sound verifier，硬违规 backprompt 重生成。

§1 · 范式演进

五次重写，每一次都是被一句用户反馈逼出来的

点开每一站，看"上一版什么反模式被戳穿、这一版怎么把活儿还给对的一方"。这条线只有一个方向：把客观计算交还算法、把主观决策交还 LLM。

01 · rule → 纯规则编排：5 段写死的模板系统 起点
被什么逼的
最快闭环——上午景点 / 午餐 / 下午景点 / 晚餐 / 夜间，五段笛卡尔积一拼就出方案。但它本质是"模板系统"不是"Agent"：评委一眼看穿。
保留价值：rule 至今仍是最底层兜底。任何上层 LLM 失败，最终都能 fallback 到它出一份"不出彩但不翻车"的方案。
02 · hybrid → LLM 出权重 + ILS 启发式搜索 + critic D-segments
被什么逼的（用户："我只有一个小时"）
5 段写死的反模式被戳穿——refiner 改了 duration，下游却不消费段维度变化，硬塞 5 段。于是把"段集合"从模板常量改成 IntentExtraction 的函数（segment_decider）。
这一版让 LLM 出 POI/餐厅权重，ILS（迭代局部搜索）做客观寻优，critic 验硬约束。但段集合仍是算法层用 if 启发式决定——主观/客观的线还没划干净。
03 · LLM-First → LLM 出 PlanBlueprint + critic backprompt + 拼装 D-llm-first
被什么逼的（用户："耦合还是有点严重"）
hybrid 仍违反 LLM-Modulo：段决策是算法 if 启发式做的。可"只想吃饭""夜宵""24h 营业""先吃饭再看展""独处沉浸"这些反 5 段需求，用 if 枚举不完。
引入 PlanBlueprint 中间数据结构作为"LLM 主观决策的契约"：段集合 / 段顺序 / 每段时长 / target_id 全交 LLM；critic 验时序/时长/营业时间，硬违规 backprompt 重生成 ≤2 次。加新段类型（夜跑 / city walk）只改 prompt 不改代码。
04 · ReAct → 单一 Agent：LLM 看全 8 工具自主 Reason+Act D-react-unified
被什么逼的（用户在 dock 直接打"太远了 3 公里以内"）
老的"router 双路径"耦合——LLM 看不到"上次方案"上下文，把反馈当成新需求重解析。也没必要上多 Agent（KidAgent / ElderlyAgent）：8 Tool + 双 mode + 关键词 fallback 三层防御已够。
让 LLM 在 ReAct 循环里 Reason+Act+Observe，跨 turn 历史用 ConversationRepository 持久化，critics_v2 当 output_validator。可它很快暴露新问题：手写 ReAct = 在重新发明 LangGraph 的子集。
05 · LangGraph Plan-and-Execute → 业界标配主架构（合体所有创新） D-langgraph · 终点
被什么逼的（用户三连问）
"业界是否有成熟范式" + "先按业界标配，创新之后再加" + "ILS 应该结合现有算法"——三问戳穿手写 ReAct 是偷懒。学术界 + LangChain 官方共识：旅行规划属"多步可预测 + 多约束"，Plan-and-Execute 优于 ReAct。
选 D 合体形态：把 ILS / blueprint / weights_llm / critics_v2 / narrator 全嵌进 graph 节点，0 代码废弃。PlanBlueprint 天然就是 P-and-E 的 plan 形态。execute 阶段 3 worker 并行约 ReAct 串行 3 倍速；四级 fallback（LLM 重试 → ILS → rule → 三层架构降级）每层都推 agent_thought 让评委可见。
§2 · 关键决策

五张牌：候选 / 选择 / 为什么不是另一个 / 影响

不是"本系统包含 A、B、C"，而是每一处都有被否决的候选。点开看那条没走的路为什么走不通。

▸ D1 LLM 选型：DeepSeek-V3 主 + Qwen 备 + OpenAI 兼容 SDK ✓ 已拍板 2026-05-08
候选
DeepSeek-V3 / 通义 Qwen-Plus / GPT-4o-mini（海外） / Claude Sonnet（贵 + 海外） / 混合 → DeepSeek 主 + 通义备
为什么
Function Calling 稳定性 ⭐⭐⭐⭐⭐（hackathon 不能因 hallucinate Tool 名浪费时间）；中文意图抽取 母语级；国内直连无需 VPN（现场网络下海外 API 不稳）；首 token < 2s；价格 ¥1/¥2 每百万 token——但价格不是首要，现场可达性 + 延迟才是。
为什么不是另一个
GPT/Claude 海外延迟 + 断连风险在国内 hackathon 现场是致命的；qwen-max 太贵（¥40/¥80）。
影响
统一用 openai Python SDK，切模型只换 base_url——这道抽象让"多 LLM 对比演示"成了几乎零成本的加分项。

▸ D2 前端栈：Next.js 14 + FastAPI + SSE 流式 ✓ 已拍板 2026-05-08
候选
Streamlit / Gradio / 纯 CLI / Next.js 14 + FastAPI + SSE
为什么
团队至少 1 人有 React 实战经验，1 个月时间够磨精致 UI；Tool 调用链路可视化最能出视觉冲击——直接对应评委"5 分钟哇时刻"清单；SSE 流式（vs 等 5s 转圈）多给评委一维体验。
为什么不是另一个
Streamlit/Gradio 是"Python 一把梭"给无前端团队的，省时但天花板低；CLI 演示效果 ⭐⭐ 不配这套规划链路。有 React 人手就没理由屈就。
影响
级联定下 FastAPI + Pydantic v2 + sse-starlette（D6）、目录结构（D7）；ToolTracePanel / ThoughtPanel 把 Agent 决策过程做成可见的"过程区"。

▸ D9 场景策略：全开放底层 + 演示形态，而非枚举关系 ✓ 已拍板 2026-05-07
候选
A 扩关系枚举 / B 同类细分 / C 环境约束轴 / D 全开放 / D + A 演示组合
为什么
评委看到"并行跑 N 个场景"是模板系统，看到"处理任意输入"才是 Agent——两者评分颗粒度不同。D 底层让"场景理解准确度 20%"评分项拿满 + 合赛题原意；A 演示选 6-8 个压测过的快捷按钮，保现场 5 分钟不翻车。
为什么不是另一个
枚举式（A/B/C）无论怎么加关系类型，本质还是"模板系统"，评分上限封在中段；纯 D 又有现场翻车风险。D+A 是"高分 + 低翻车"唯一解。
影响
场景 = LLM 动态抽取的约束集，联动改 7 份文档 + 新增演示场景集；开发量可控（3 人 +25h）。这也正是 §1 时间线一路反"5 段写死"的根因。
docs/01-requirements/架构选型.md :334-366 docs/00-overview/progress.md :346-350
▸ D-biz 商业三层抽象：Protocol + Stub，让"换数据源"不改业务代码 ✓ 已拍板 2026-05-17
候选
A 维持具体实现（商业演进时全重写） / B Protocol 抽象 + 默认实现 + Stub / C 直接接真高德/真 Redis（赛题禁止） → B
为什么
用户决议"这是参赛作品 + 希望真做成产品"。商业大奖维度要让评委看到"demo 不止于 demo"：评委只需 grep 一处签名就能验证"换数据源真不需要改业务代码"。Protocol + Stub 是最低成本的可信演示路径。
为什么不是另一个
A 看着 demo 简洁，但商业演进时全部重写——等于告诉评委"这架构是死路"；C 赛题明令禁止接真 API。
影响
落地为 ToolProvider / ConversationRepository / observability 三层（每层 Demo→MVP→真产品 三阶段路径，见 §4），全靠 .env 一行切换。
docs/00-overview/progress.md :409-419 backend/agent/runtime/tool_provider.py
▸ D-confirm fast-finalize：把 LLM/记忆甩后台，治 31s 卡顿 ✓ 已拍板 2026-06-03
痛点
首字节超时修复后，/chat/confirm 仍约 31s 才吐 tool_call。根因：Tool 执行完成后被 LLM confirm-narrator / memory-writer 串行挡住，评委会误以为"预约很慢"。
选择
确认成功 只同步等执行类 Tool（餐厅预约 / 门票 / 附加服务 / 转发文案 + 必要行程回填）；LLM narrator、memory_writer、ConversationStore 记录全部后置为后台副作用。
为什么这样划线
沿用全局哲学：执行类 Tool 是"客观必须等的结果"，叙事/记忆是"主观锦上添花"——后者没理由阻塞前者的可见反馈。
影响
人为让后台 memory sleep 5s，tool_call / itinerary_ready / done 仍在 0.010s 内输出；默认 direct-node 保留 memory_status 维持语义测试。
docs/00-overview/progress.md :578-582
§3 · 主架构落地

LangGraph 三大范式，对应真实 graph 节点

终点架构不是凭空选的——LangChain 官方三大编排范式，在 backend/agent/graph/build.py 里各对应一段真实的 graph 拓扑。

Routing
先分流，再对症下药
START → router
→ {chitchat | intent | refiner}
一句话进来先判类型：闲聊直接 END，新规划走 intent，反馈走 refiner——不让所有输入都吃整条重规划链路。注入检测也在这层最前置（不调 LLM）。
graph/nodes/router.py graph/build.py :186-194
Plan-and-Execute
先并行取数，再统一规划
intent/refiner
→ 3 worker 并行
→ execute_collect → planner → assemble
search_pois / search_restaurants / get_user_profile 三 worker 并行取数（LangGraph 多边并发），汇聚后 LLM 出 PlanBlueprint，纯函数 assemble 成 Itinerary。约 ReAct 串行 3 倍速。
graph/build.py :199-213
Evaluator-Optimizer
critic 验、不过就 backprompt
critic → {narrate | replan_router}
replan → {planner | ils_replan | give_up}
critic 验客观约束，硬违规走 双层 backprompt：llm_backprompt 回 planner 重生成 / ils_fallback 交 ILS 算法兜底 / give_up 也出方案。这正是 LLM-Modulo 的 critic→regenerate 闭环。
graph/build.py :216-244
⚠ 关键工程坑 · thinking 模型第二轮
MiMo 是 thinking 模型，第二轮调用要求回传 reasoning_content，但 LangGraph 默认不携带 → 第二轮直接报错。 标准解法：extra_body={"enable_thinking": False} 关掉 thinking。 这是 DeepSeek-R1 / Kimi K2 thinking / o1 同类问题的通用解——任何"思考模型 + 多轮编排框架"组合都会撞上，记下来省下一个团队半天。 docs/00-overview/progress.md :460 · 参考 MiMo 官方 vllm recipe + LiteLLM #23828

§5 · 学术依据

不是拍脑袋，每个范式都有论文背书

核心
