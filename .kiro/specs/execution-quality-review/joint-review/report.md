### 3.3 隐藏冲突（≥3 条，每条带 file:line + 独立第二意见取舍方案）

> 隐藏冲突 = 4 报告各自有合理逻辑，但结论互相**真实矛盾**——必须独立第二意见拍板。

#### 隐藏冲突 1：「TOOL_RESPONSE_INCONSISTENCY 主路径是否真的接通了」

- **A §3.3 / §6 失败模式 2**：「**布线覆盖只到 ILS 路径 + 单测 + verify_spec_c_demo.py**，主链路（LangGraph critic_node 与 ReAct output_validator）实际是否激活，需要 §6 失败模式 2 的关注」（A 主张：可能没接通）
- **B §五 5.2**：spec C demo Demo 3 「编造 P999 必识别为 hallucination」+ Demo 8 「validate_itinerary 集成 tool_results」全过 → 隐含「已接通」
- **C §4 ViolationCode 表**：第 11 类 TOOL_RESPONSE_INCONSISTENCY 「critic 拦」，引 critics_v2.py:953-1024 → 隐含「已接通」
- **D §五 5.4 messages 持久化测试缺**：「没有自动化测试覆盖『refine 后 trace 字段重置』这个场景」——指向**集成测试覆盖度真有缺**

**真实矛盾**：A 说「布线覆盖只到 ILS + 单测，主链路实际是否激活需要确认」；B 直接给「Demo 8/8 全过」当作已接通证据。这两个说法**只有一个对**——如果 LangGraph critic_node 没传 tool_results，spec C task 3 的算法在主链路上**默默失效**，但 verify_spec_c_demo.py 的 Demo 3 仍然过（因为它直接调 critic 函数而非走 graph）。

**联合审查的取舍**：**站在 A 一边，B 不充分**。理由：B 的 8/8 证据来自 verify_spec_c_demo.py 单测脚本，**不能代表 LangGraph 主路径运行时是否真注入 tool_results**——这正是 A §6 失败模式 2 的关注点。**修复方案**：30 分钟内做一次 LangGraph 端到端 + critic_node 是否 emit `TOOL_RESPONSE_INCONSISTENCY` 的集成测试，预期是「测一个含 P999 编造 ID 的 LLM 蓝图，能不能在 graph 主路径上 触发 violation」。如果触发就 B 对、如果不触发就 A 对（A 已给了具体修复 patch：`agent/graph/nodes/critic.py` 调 `validate_itinerary` 处补 `tool_results={"pois": state.get("pois") or [], "restaurants": state.get("restaurants") or []}`）。

#### 隐藏冲突 2：「Pass@1 估值是 80-85% 还是 40-55%」

- **B §一**：「项目首次跑会真跑通的 Pass@1 估计 = 中位 80-85%」+ 引用 verify_planning_quality 24/24 + verify_spec_c_demo 8/8 + test_8_scenarios 17 项
- **D §三 GAP 矩阵**：「Pass@1 阈值 85%（spec C 自定）→ Planner-R1 56.9% / TravelPlanner 0.6% → 数字看似领先 100×，实因小样本 + 闭世界 mock」
- **C §3** 整链失败率 7.2e-6 = 几乎不可能失败 → 与 B 一致
- **A §1** 「Recall 受 Demo 时间盒压制至中位水准」 → 隐含 Pass@1 只是中位

**真实矛盾**：B 数字（80-85%）针对的是「我们自己的 8 场景 + 自定 spec C 阈值」；如果换成 TravelPlanner 1225 query benchmark（这是论文 Pass@1 的真实定义），数字会显著下修——D 已警告小样本错觉。**A 的 Recall 中位说**与 **B 的 Pass@1 80-85%**也有内在矛盾——Recall 中位但 Pass@1 高，意味着「召回率不够时 critic 会救场让 Pass@1 仍然高」，这与 LLM-Modulo 范式吻合，但前提是 critic 真的在主链路运行（见冲突 1）。

**联合审查的取舍**：**两套口径都对，但用错场合就是误导**。**B 数字应该叫「8 演示场景闭世界 Pass@1 = 80-85%」**，不叫「Pass@1 = 80-85%」；**真正可对标 TravelPlanner benchmark 的开放域 Pass@1 估 40-55%**（理由：闭世界 80% × 开放域降级因子 0.5-0.7 = 40-55%）。在路演讲台上，应该用「8 演示场景全部稳定通过 + 自动化 22 项 spec C 断言通过」这一表述，**不要说 80-85% 这个数字**——评委如果是论文背景会用 TravelPlanner 标尺看你。

#### 隐藏冲突 3：「评委是否真会即兴攻击边角」

- **D §一翻车点**：3 个翻车点全部假设评委会即兴扔词典外输入 / 多重约束 / 极端时长 → **乐观估计：高概率攻击**
- **B §四 4.4**：「评委即兴扔词典外输入」概率估 30%、影响低；§六风险 4 给 30% × 低
- **C §5 触发难度**：「E3 评委需要做什么 = 输入想去市外距家 50km 的项目 + 强制让 grounding 放宽机制不生效；可见概率 ≈ 0%」 → 隐含「评委大概率不会做这种边角输入」
- **A 全文**未直接讨论评委攻击概率，但 §5 暗示主路径 8 场景全过 → 隐含「评委按推荐路径走」

**真实矛盾**：D 全文假设评委会主动攻击；B/C 估算评委有 70% 概率按推荐路径走、不会主动构造极端输入。**这是 hackathon 评委行为模型的两种极端假设**——D 偏悲观，B/C 偏乐观。

**联合审查的取舍**：**D 偏紧，B/C 偏松，真实在中间**。理由：① 评委一定会按推荐路径走完一次（演示文档 + 团队引导）；② 在主路径完成后，评委的剩余 1-2 分钟会**至少**做一次「即兴尝试」——这是评委专业素养。所以 D 的攻击概率不能按 100% 算（评委一定会按时间盒走），但也不能按 30% 算（B 的估值偏松）。**实战策略**：把 D §九「demo 现场 5 句话」严格背下来——不防住所有攻击向量，但保证遇到时能 30 秒回应到位 + 把镜头切回评分项。这种「认怂 + 转向」是 hackathon 答辩的必备技能。

#### 隐藏冲突 4：「Recall 弱是 bug 还是产品决策」

- **A §1 / §4.2**：默认 `PLANNER_LLM_STRATEGY=llm_first` 漏调 search_restaurants 8-15%（Recall 弱），「这是产品决策（Demo 评委要看 LLM 的『思考过程』而非死板规则），不是缺陷」（A 自己定调）
- **B §3.2**：「ReAct 路径主要损耗：MiMo Pro Function Calling 把 list[T] 序列化成 JSON 字符串」「失败模式 5 多重约束叠加 Pass@1 估 72-80%」 → 暗示 Recall 弱是 bug
- **C §3 LLM-First 失败 ≈ 8%**：与 A 数字一致，但归因到「LLM JSON 解析」而非「Recall 弱」
- **D §三 GAP**：「召回率维度差业界 2-3 个量级」 → 直接定调 Recall 弱

**真实矛盾**：A 说「Recall 弱是产品决策不是缺陷」；D 说「Recall 弱是真差距」。**这是建设性 vs 攻击性立场最尖锐的冲突**。

**联合审查的取舍**：**A 的「产品决策」说法部分成立但不完全成立**。「让 LLM 自主决策工具调用」是 hackathon 评分项「Agent 行为可见性 25%」的体现，这是 A 的合理逻辑；但「Recall 不够时不主动追问 / 不主动放宽 / 不主动告诉评委」是真缺陷——它让「主动质疑」从可选项变成强制项。**修复**：在 LLM-First Planner 入口加一句 system prompt「若用户输入提到任何吃/喝/餐 关键词，segments 必须含『用餐』，否则按 decide_nodes 推」（A §6 失败模式 3 给的修复，3 行代码）。这是 A 自己的提案，但 A 给它定级为「失败模式」——隐含承认这不是产品决策而是缺陷。

#### 隐藏冲突 5：「Pass@1 数字在 8 场景上的 91% vs Planner-R1 论文 56.9%——领先还是错觉」

- **B §三 3.1**：「LangGraph 主路径 80-85%」 + 业界对标行 LLM-Modulo 93.9% → 我们 vs SAT 求解器接近
- **A §1**：「与 LLM-Modulo + TravelPlanner 对比，已达 SOTA 水准」
- **C §8.1**：「LLM-Modulo 5/5 全覆盖，且 critic 数量超出论文 baseline 50%+」
- **D §三 GAP**：「Pass@1 数字看似领先 100×，实因小样本 + 闭世界 mock」 + 「8 场景上 0.9（小样本）」

**真实矛盾**：A/B/C 都把项目数字直接放在 Planner-R1 / TravelPlanner 一档；D 警告这是样本量错觉。

**联合审查的取舍**：**D 对，A/B/C 错位**。理由：8 演示场景 + 团队预设的 mock 数据是闭世界 N=8 的「场景验收」；TravelPlanner 是 1225 query 的开放域 benchmark；二者**统计学上不可比**。**正确的表述**：「我们在 LLM-Modulo + ItiNera + TravelAgent 三个范式上做了完整的工程拼装；在 8 演示场景 + spec C 22 项断言上稳定通过；如果未来要做开放域评测，可以接入 TravelPlanner benchmark」。**绝不在路演说「我们超过 Planner-R1 56.9%」**——这会让有论文背景的评委立刻判定团队不专业。

## 四、每个 sub-agent 的盲点 / 偏见识别

> 4 个 sub-agent 各自有「视角孤岛」，识别这些盲点能让联合审查得到独立第二意见。每个 agent 各 2-3 个具体盲点带证据。

### 4.1 Agent A（建设性 + 工具精确率）的盲点

**盲点 A-1：建设性视角天然忽略「评委可见性」攻击面**

- 证据：A §3 / §4 / §5 全是「正确性」分析（精确率 99% / 召回率漂移率 / 多级 fallback 链）；§6 三个「失败模式」都是技术级 bug，没一条是 demo 现场可见性问题
- 反例：D §六 6.4 指出「ChatDock + ToolTracePanel 默认收起 → 评委不点 = 看不到 → Tool 编排 25% 实质性扣分」，A 完全没提
- 联合判断：A 的视角对算法严谨度 100% 覆盖，但对 hackathon 评分维度的 30% 看不见。**A 给出的「8 工具 SOTA 拼装」是技术结论，不是评分结论**

**盲点 A-2：把「PLANNER_LLM_STRATEGY=llm_first 是产品决策」过早定调，错过「主动追问」加分点**

- 证据：A §1 「这是产品决策（Demo 评委要看 LLM 的『思考过程』而非死板规则），不是缺陷」+ §6 失败模式 3 同样的归因
- 反问：如果 LLM 漏调 search_restaurants 但**主动告诉评委「我没看到吃饭意图，需要补充吗」**，这就是 D 攻击 1 修复方案的「降级文案」——既体现 LLM 决策又体现 Agent 主动质疑
- 联合判断：A 把「LLM 自主决策」=「Recall 弱是合理的」这两件事混为一谈。**「自主决策」是 demo 加分项，「不主动追问」是减分项；二者不冲突，A 错把它们绑定**

**盲点 A-3：对 8 工具数量「8/8 行 Input 校验过」的报告口径过于精炼，忽略 8 个工具内部不均匀**

- 证据：A 矩阵显示 6/8 行有 hallucination 防护，reserve_restaurant / buy_ticket 仅靠 NOT_FOUND 间接拦
- 缺失分析：A §6 失败模式 1 已经识别这个问题，但归类为「失败模式」而非「评分项扣分」；评委如果看到执行类工具没有 ID 白名单校验，会认为是工程缺陷
- 联合判断：A 自身已识别该 gap，但应该升级为「评分项 5 异常韧性扣分」而不仅是「30 分钟修复提案」

### 4.2 Agent B（建设性 + Pass@1）的盲点

**盲点 B-1：Pass@1 估算 80-85% 的乐观偏差**

- 证据：B §一直接给「Pass@1 中位 80-85%」，§三对标表 LLM-Modulo 93.9% / Planner-R1 56.9% / SAT 93.9%，把项目数字直接放在 SAT 求解器一档
- 反问：项目 N=8 闭世界 + Planner-R1 N=1225 开放域 + SAT N=180 开放域，统计学上不可比
- D §三已经指出这是「数字看似领先 100×，实因小样本」
- 联合判断：B 数字应改写为「**8 演示场景闭世界 Pass@1 = 80-85%**」加限定语；裸数字与 Planner-R1 56.9% 直接比是误导

**盲点 B-2：B §四「边界场景识别」没覆盖词典外社交意图（D 攻击 1 的核心）**

- 证据：B §四覆盖了「反 5 段 / 极端时长 / 多重约束 / 词典外输入」4 类，但「词典外输入」用的例子是「带个老师 / 请客户」，估值「Pass@1 65-75%（首轮规划失败但能给暖回话不算 stream_error）」
- 反问：D 攻击 1 估值「**LLM 把『带老师』填成『同学重聚』后，候选错池**」 → 不是「stream_error」也不是「暖回话」，而是「方案表面 OK 但语义错误」
- B 把它归到「不算 stream_error」就过了，D 把它归到「翻车点 1」
- 联合判断：B 漏看了「intent 错填后下游静默走完」的隐式失败模式。这是 B 视角孤岛——只看「跑通 vs 没跑通」，不看「跑通了但语义错」

**盲点 B-3：B §五真实跑通证据链全部基于 verify 脚本 / 单测 / 自动化测试，没考虑「评委手动操作的输入空间」**

- 证据：B §五引 verify_planning_quality 24/24 + verify_spec_c_demo 8/8 + test_8_scenarios 17 项 + verify_react_agent 5 场景
- 反问：所有这些都是**预设输入 + 预设期望输出**的测试；评委手动输入是无界自由输入空间
- D §一三个翻车点都是「评委手动输入」触发的边角
- 联合判断：B 的「Pass@1 80-85%」是「自动化测试 Pass@1」，不是「评委手动 Pass@1」。后者通常比前者低 15-25%

### 4.3 Agent C（建设性 + 异常韧性）的盲点

**盲点 C-1：5 重防御链评估只看「静态代码是否存在」不看「动态触发是否可见」**

- 证据：C §2 5 类异常 × 5 重防御层矩阵，每格写「拦/不拦/部分」+ file:line。**全部是静态代码分析**
- 反问：C §5 自己已经承认「E3 触发难度 ★★★★★ 最高 / 可见概率 ≈ 0%」、「E4 触发难度 ★★★☆☆ 中 / 可见概率 ≈ 5%」 → 静态防御满分但动态可见性极差
- C §1 「13/15 = 87%」 但 §5 自己说「扣 2 分主要来自 demo 触发难度 + 4 级 fallback 链在评委 5 分钟时间盒内最多看到 2 级」 → C 自己已经指出 13/15 是静态分，动态分应该更低
- 联合判断：C 13/15 评分是「算法韧性满分 - 2 分 demo 可见性扣」的算术结果，但**评分项 5 是 demo 评分项，应该按动态可见性主导，13 分偏乐观**。独立打分按 §一矩阵：12 ± 1（80% ± 7%）

**盲点 C-2：「业界没有 5 重防御链的完整范式 → 超出业界 SOTA」论断过于自信**

- 证据：C §8.4 「业界没有『5 重防御链』的完整范式。本仓库是 LLM-Modulo + ItiNera + TOPTW + 自创三轴评分的工程拼装，作为 hackathon 项目超出业界 SOTA 的工程深度」
- 反问：「业界没有完整范式」≠「我们超出 SOTA」——可能是因为业界场景需要这种拼装，不需要 5 重防御。**5 重防御对 N=8 场景 demo 是合适的，对 N=1225 benchmark 可能过工程**
- 联合判断：5 重防御链是「工程深度的展示道具」而非「真实业界 SOTA」。在路演讲台上可以说「我们做了 LLM-Modulo + ItiNera + TravelAgent 三联混合范式」，**不要说「超出业界 SOTA」**——这种自我评价会让评委警觉

**盲点 C-3：「OPENING_HOURS_VIOLATION 镜像缺失」概率估值过低**

- 证据：C §7 漏洞 3「概率 低 × 影响 极高 = MEDIUM」
- 反问：D 攻击 1 词典外社交意图 + 攻击 4 LLM 主观判断翻车都是「评委即兴扔多义词触发」的高概率场景；如果评委输入「夜宵局」→ ILS 路径 21:30 闭店 POI，这个概率不低
- 联合判断：C 把 opening_hours 缺失归到「低概率」，但「评委即兴扔时间敏感词」是高概率攻击。建议升级为「中概率 × 极高影响」

### 4.4 Agent D（攻击性 + 攻击面）的盲点

**盲点 D-1：评委攻击概率估值过紧**

- 证据：D §一翻车点 1「概率 高 × 触发条件零难度」「带个老师 / 请客户」
- 反问：评委首轮 100% 走团队推荐路径（家庭主线 S1）；剩余时间评委更可能扔「业界已知 hard case」（如多重约束）而不是冷僻多义词
- B §四 4.4 估「评委即兴扔词典外输入」概率 30%、影响低
- 联合判断：D 把所有 9 个攻击向量都拉到「评委一定会做」的紧度，实际评委时间盒只够做 1-2 次即兴攻击。9 个攻击向量中真正需要修的是 Top 3-4

**盲点 D-2：D §三业界 GAP 矩阵的「我们的位置」一栏数据来源不一致**

- 证据：D §三表第 6 行「我们 Demo MVP」一栏：精确率 A≥0.9（spec planning-quality-deep-review）、召回率 R≥3（演示场景集 §4.1）、Pass@1≥85%（spec C 阈值）、异常韧性 4 类、可见性 中-高
- 反问：A≥0.9 来自 8 场景 verify；R≥3 是「最低候选数」而非召回率指标；Pass@1≥85% 是 spec 自定阈值而非测试通过率；这三个数字**不是同一个统计概念**，放在同一个矩阵里横比
- 联合判断：D 自己警告了 B 数字小样本错觉，但自己也在 §三 GAP 表中犯了相似错误。GAP 矩阵只有「评委可见性」一栏是真实的横向对比；其他三栏需要进一步澄清统计口径

**盲点 D-3：「评委开口攻击 5 句话」过于强调「转向评分项」，可能让团队过度依赖话术而非真修漏洞**

- 证据：D §九 5 句话全部是「承认小缺陷 + 指向 spec 论证 + 把镜头切回评分项」
- 反问：话术只能在评委已经识别问题之后用；如果评委尚未识别，话术起不了作用——只有真修漏洞才能避免评委识别
- D §八 必做高 ROI 3 条已经覆盖了大部分攻击向量的真修方案，但 §九话术给团队一个「不修也能答辩」的错觉
- 联合判断：路演脚本 + 真修漏洞两手都要硬，D §九的话术只是 backup plan，不能替代 §八的修复工时

## 五、评委 demo 现场最容易被攻破的 3 个点（综合 4 份报告 + 独立判断）

> 综合 4 报告 + 独立判断，按「真实概率 × 系统损伤 × 修复成本」排序的 Top 3 攻击点。

### 5.1 Top 1：fallback 链不可见（综合 C §7 漏洞 1 + D §六 6.4）

- **场景**：评委按 S1 家庭主线快捷按钮 → 主路径成功 → 评委追问「如果 LLM 挂了你们怎么办」
- **评委动作**：要求演示 LLM 失败的兜底链路 / 要求看 ToolTracePanel
- **当前后果**：① 演示者打开 ToolTracePanel 给评委看，但 ChatDock + ToolTracePanel 默认收起；② 即使打开，rule planner 兜底没有 SSE 事件透出，演示者只能口头说「rule planner 兜底了」；③ 评委如果点开 ToolTracePanel 看到「主路径直接成功，没看到 fallback 触发」，质疑「fallback 链是不是没用上」
- **当前防线是否足够**：**不足**——C §7 漏洞 1 已指出 rule planner 入口没有 emit；D §六 6.4 已指出双层折叠默认收起
- **预防方案**：① ToolTracePanel 默认展开（30 min 工时；前端 localStorage 默认值改 true）；② rule planner 入口加 `tracer.emit("agent_thought", "已切换 rule planner 安全兜底")`（C §9 加分 1 已给方案，1 小时工时）；③ replan_router 在 fallback_chain 改变时同步推一条 `replan_triggered` SSE（C §7 漏洞 1 给方案）
- **总工时**：≤2 小时

### 5.2 Top 2：词典外社交意图被静默错填（综合 D 攻击 1 + B 盲点 4.4）

- **场景**：评委即兴扔「带个老师」/「请客户」/「陪宠物」
- **评委动作**：评委复制粘贴自己想到的多义词输入
- **当前后果**：LLM 强行映射到 9 个 SOCIAL_CONTEXTS 之一，UI 上 IntentSummary 显示「同学重聚」与「老师」无关；评委追问「你怎么把老师理解成同学聚会」→ 第 1 评分项「场景理解 20%」实质性扣分

### 8.1 沉默风险 1：mock_data 西溪银泰 9 处共址不仅是 marker 叠加，更是「评委判断 demo 可信度」的硬伤

**4 报告各自怎么看**：
- A：完全没提（A 不读 mock_data）
- B：§二 8 场景表给「mock 候选 ≥3」即认为达标，没看坐标分布
- C：完全没提（C 看异常韧性，不看数据形态）
- D：§四 4.1 看到了 9 处共址 + MapOverlay 圆弧微扰，但归类为「低优先级 polish 不做」

**真实风险**：评委如果在 demo 后**点开任意 marker 看 InfoWindow**，看到「西溪银泰」三个字出现 9 次（地图上只看到 1 个 marker 因为微扰半径 50m 还是会重叠）：
- 评委立刻判定「mock 数据是 demo 等级」 → 「项目可信度」整体下修 5-10%

## 九、独立第二意见：评委可见性 vs 工程深度的 tradeoff

### 9.1 现状描述

我们做了 LLM-Modulo + ItiNera + TravelAgent 三联混合范式（A 反复强调 critic + Pydantic AI ReAct；C §8 明确把 5 重防御链对标到三个论文）：

- **LLM-Modulo（NeurIPS'24）**：critic Bank（11 类 ViolationCode）+ backprompt 反馈环 + outer loop（_MAX_LLM_RETRIES=2 + _MAX_TOTAL_RETRIES=4）+ dense scalar reward（compute_reward 占位预留）
- **ItiNera（EMNLP'24）**：preference_scorer.py LLM 语义打分（user-owned POIs filtered，ItiNera EMNLP'24 范式）+ ILS `_utility` 末尾 `+ 0.3 * semantic_scores.get(poi.id, 0.5)`
- **TravelAgent（NeurIPS'24）**：bounded retry ≤8 + give_up 兜底（与 _MAX_TOTAL_RETRIES=4 同构）+ 三层 schema 标配（IntentExtraction → PlanBlueprint → Itinerary）

**工程深度评估**：A/B/C 三 agent 给的「业界 SOTA 拼装」是真实的——本仓库**业界首次**把 LLM-Modulo + ItiNera 在工程上拼起来（spec C joint-review report.md 已论证）。

### 9.2 评委可见性危机

但**评委 5 分钟看到的 UI 元素只有 3 个**：
1. **ChatDock**（主对话流）：默认收起或仅展示最新一条
2. **ToolTracePanel**（工具调用轨迹）：默认收起，需要主动点开
3. **ItineraryCard**（行程卡片）：唯一默认可见的工程产物

**工程深度评委可能根本看不到**：
- 5 重防御链的「层 1 grounding-first」工作 → 没 SSE 事件透出（C §7 漏洞 2）
- 11 类 critic 的工作 → 仅在违规触发时 emit `replan_triggered`，主路径成功时静默
