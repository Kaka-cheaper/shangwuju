# Planning 策略过程记录

## 1. 记录范围

本记录描述后端如何把一句自然语言需求收敛成可执行行程。过程关注点包括：意图是否清楚、候选是否足够、模型输出是否可用、行程是否满足硬约束、失败时是否能解释。

## 2. 策略推进顺序

```text
输入接收
  -> 路线判断
  -> 意图抽取
  -> 候选准备
  -> 蓝图规划
  -> 行程拼装
  -> critic 检查
  -> 输出或重规划
```

这条顺序把模型擅长的主观偏好判断和系统擅长的结构化约束分开处理。模型负责理解需求、给出候选权重和中间节点选择；系统负责 home、hop、schedule、容量、距离、时长等可计算约束。

## 3. 意图抽取过程

| 观察项 | 处理方式 | 记录结果 |
| --- | --- | --- |
| 时间表达 | 解析今天、周末、下午、晚上等相对时间 | 写入标准时间窗口 |
| 同行人 | 抽取人数、关系、年龄段 | 影响餐厅、活动和节奏 |
| 距离限制 | 识别几公里以内、不要太远等描述 | 进入通勤和候选过滤 |
| 饮食偏好 | 抽取口味、忌口、健康诉求 | 影响餐厅查询和排序 |
| 体验偏好 | 抽取亲子、安静、社交、预算等标签 | 影响 POI 和路线权重 |
| 不确定项 | 低置信或缺失信息进入 `quality_issues` | 后续解释时提示用户 |

意图结果不会直接等同于最终方案。它只是规划输入，后续还要经过候选准备、蓝图规划和 critic 检查。

## 4. 候选准备过程

| 候选类型 | 来源 | 使用方式 |
| --- | --- | --- |
| POI | `search_pois` | 提供活动节点和标签 |
| 餐厅 | `search_restaurants` | 提供用餐节点、价位、口味和可订信息 |
| 用户画像 | `get_user_profile` | 提供 home、交通偏好、历史偏好 |
| 路线估算 | `estimate_route_time`、`lookup_hop` | 计算节点间通勤和缓冲 |

候选不足时不会立即中断，而是记录放宽信号，例如降低距离过滤、保留更宽泛标签或进入规则路径。这样能让系统优先给出可说明的方案，而不是直接返回空结果。

## 5. 蓝图与拼装过程

| 层级 | 输入 | 输出 | 作用 |
| --- | --- | --- | --- |
| PlanBlueprint | 候选池、意图、画像、反馈 | 中间节点、顺序、停留理由 | 让模型只选择“去哪里”和“为什么” |
| Assemble | 蓝图、路线、home、餐厅可用性 | 完整 Itinerary | 补齐时间、交通、用餐和节点说明 |
| DecisionTrace | 权重、critic、候选、fallback | 可展示的决策痕迹 | 说明方案来源和调整原因 |

这个拆分避免模型同时承担路线估算、时刻表计算和业务执行。蓝图可以重试，拼装可以规则校验，最终解释可以从 trace 中提取。

## 6. critic 与重规划过程

critic 检查集中处理硬约束和明显体验问题。

| 检查项 | 触发条件 | 处理动作 |
| --- | --- | --- |
| 距离过远 | hop 超出距离限制或通勤时间过长 | 回写 feedback 后重排节点 |
| 时间冲突 | 停留时长、营业时段或交通缓冲不成立 | 调整 schedule 或降低方案置信 |
| 餐厅不可用 | 满座、无候选或不符合人数 | 替换餐厅或给出无法预约说明 |
| 候选质量低 | 标签命中弱、画像冲突或缺少亲子/社交条件 | 调整权重，进入替代候选 |
| 多轮失败 | 重试达到上限 | 进入 ILS、规则路径或明确放弃 |

重规划不是简单再次调用模型，而是把 critic 发现的问题转换成可读 feedback，再带着问题重新进入规划。这样可以减少同类错误重复出现。

## 7. 过程结论

- Planning 主体由“理解、准备、蓝图、拼装、检查”五步组成。
- 模型输出只作为蓝图，不直接代表最终行程。
- critic 和 DecisionTrace 是解释能力的来源，能说明为什么重试、替换或降级。
- 规划失败时必须有明确 reason，不能只返回一段自由文本。

## 8. 待关注项

- 真实地图和餐厅服务接入后，需要重新校准距离、营业时段和满座场景。
- 多人协作反馈可能同时出现，需要继续明确同一时刻只处理哪一条约束。
- LLM 输出质量受 prompt 和模型配置影响，需要保留规则路径作为稳定边界。
