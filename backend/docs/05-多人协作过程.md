# 多人协作过程记录

## 1. 记录范围

本记录跟踪协作房间从创建、成员加入、状态同步、规划广播到确认权限控制的过程。重点关注两个问题：新成员能否补齐过程，非发起者能否被阻止执行确认。

## 2. 房间创建过程

| 步骤 | 输入 | 处理 | 输出 |
| --- | --- | --- | --- |
| 创建房间 | owner、昵称、初始行程或会话快照 | 初始化 Room、成员表、chat_state | `room_id`、owner、当前状态 |
| 绑定会话 | session_id 或协作专用 session | 保存规划事件和当前 itinerary | 后续规划可回填房间 |
| 返回状态 | room snapshot | 返回成员、消息、行程、planning_events | 前端进入房间页 |

房间状态由 `backend/collab/room.py` 管理，接口层由 `backend/api/collab.py` 负责接收 HTTP 和 WebSocket 请求。

## 3. 成员加入过程

| 节点 | 处理内容 | 结果 |
| --- | --- | --- |
| WebSocket 连接 | 读取 `room_id`、`user_id`、`nickname` | 建立成员连接 |
| 状态补齐 | 发送 room snapshot、chat_state、planning_events | 新成员看到已有过程 |
| 在线广播 | 成员加入、离开、消息变化 | 房间内用户同步感知 |
| 断开处理 | WebSocketDisconnect | 清理连接但保留房间过程 |

加入过程不是只同步最终行程，而是同步 planning_events。这样后来加入的人能看到前面为什么选择某个方案、是否经历过重试和替换。

## 4. 协作消息过程

| 消息类型 | 处理方式 | 影响范围 |
| --- | --- | --- |
| `chat` | 写入房间消息并广播 | 所有成员可见 |
| `constraint` | 转为规划约束或反馈 | 影响后续规划 |
| `planning_event` | 写入事件历史并广播 | 补齐过程可见性 |
| `confirm` | 校验 owner 权限后进入确认 | 只有发起者可执行 |
| `ping` | 返回 `pong` | 保持连接健康 |

协作消息只负责同步过程和约束，不直接越过后端规划入口。需要影响方案时，仍要进入规划或反馈链路。

## 5. 确认权限处理过程

确认预约是多人协作中的关键边界。

| 检查点 | 处理 | 结果 |
| --- | --- | --- |
| 前端按钮 | 非 owner 不展示或禁用确认入口 | 减少误操作 |
| WebSocket 消息 | 收到 `confirm` 后检查发送者是否 owner | 非 owner 返回错误事件 |
| 后端执行 | 只有 owner 通过校验才调用确认流 | 预约和购票只触发一次 |
| 广播结果 | 确认完成后广播订单和最终文案 | 所有成员看到结果 |

权限必须前后端同时控制。前端控制体验，后端控制安全边界；即使协作者手动发 WebSocket 消息，也不能触发预约。

## 6. 规划事件同步过程

| 事件来源 | 房间处理 | 用户看到的结果 |
| --- | --- | --- |
| 意图抽取完成 | 写入 planning_events_history | 展示时间、距离、同行人等标签 |
| 工具调用完成 | 写入工具摘要 | 展示候选搜索、路线估算、餐厅状态 |
| critic 重试 | 写入重试原因 | 展示为什么换方案 |
| 行程完成 | 更新当前 itinerary | 所有成员同步地图和方案 |
| 确认完成 | 写入 orders 和 share message | 所有成员看到最终结果 |

这部分曾重点验证“部分组件不同步”的问题。处理方向是让房间同步的不只是聊天消息，还包括规划阶段事件、当前行程快照和确认结果。

## 7. 过程结论

- 房间状态要包含成员、消息、当前行程和规划事件。
- 新成员加入后应补齐过程，而不是只看到末尾消息。
- 规划可以多人参与，确认执行只能由 owner 发起。
- 权限校验必须在后端完成，前端禁用只是体验层控制。

## 8. 待关注项

- 房间长期存在时，planning_events_history 需要限制长度或做分页。
- 多名成员同时提交约束时，需要明确队列顺序和覆盖规则。
- 确认结果应继续补充幂等键，避免 owner 重复点击造成重复执行。
