# 异常处理与兜底过程记录

## 1. 记录范围

本记录跟踪后端在输入、模型、工具、流式输出和确认执行中遇到异常时的处理过程。目标是让每类异常都有明确出口，而不是把错误留给前端猜测。

## 2. 输入异常处理过程

| 场景 | 观察 | 处理 |
| --- | --- | --- |
| prompt injection | 文本要求忽略规则、泄露系统信息或越权执行 | 进入安全拒答，不进入规划 |
| 无效输入 | 缺少必要字段、格式不合法、文本为空 | 返回可读错误或 `INVALID_INPUT` |
| 反馈不清楚 | 已有方案后输入模糊表达 | 保守识别为反馈并降低置信 |
| 需求过宽 | 时间、地点、偏好不足 | 继续规划，但在质量提示中说明不确定项 |

输入异常优先在入口和 router 阶段处理，避免污染后续候选准备和模型调用。

## 3. 模型异常处理过程

| 场景 | 处理机制 | 结果 |
| --- | --- | --- |
| 分类低置信 | route 结果进入保守分支 | 避免误把反馈当成新需求 |
| 意图抽取失败 | 构造 fallback intent，记录 `quality_issues` | 规划不中断，但解释会提示限制 |
| 蓝图格式不合格 | schema 校验失败后重试 | 模型输出不会直接进入拼装 |
| 蓝图违反约束 | critic 生成 feedback 并重规划 | 降低同类错误重复概率 |
| 多轮仍失败 | 进入 ILS、规则路径或 give_up | 给出明确可解释结果 |

模型异常不直接暴露原始输出。上层只接收结构化对象、失败 reason 和可读解释。

## 4. 工具异常处理过程

| FailureReason | 常见来源 | 处理动作 |
| --- | --- | --- |
| `EMPTY_CANDIDATES` | POI 或餐厅搜索为空 | 放宽标签、扩大候选或进入规则说明 |
| `RESTAURANT_FULL` | 餐厅时段不可订 | 替换餐厅或提示无法预约 |
| `INVALID_INPUT` | 工具入参不合法 | 停止该工具调用并回到上层修正 |
| `NOT_FOUND` | 用户画像、目标或资源不存在 | 使用默认值或提示缺失 |
| `UPSTREAM_FAILURE` | provider 超时、网络或服务失败 | 重试、降级或返回解释 |

工具异常统一收敛为 reason，规划层不解析自由文本。这样能保持失败路径可测试、可展示、可复盘。

## 5. SSE 异常处理过程

```text
内部事件
  -> to_sse 转换
  -> EventSourceResponse 输出
  -> 异常时 stream_error
  -> done 收口
```

处理原则：

- 内部异常必须转成 `stream_error`，不能让连接无声中断。
- 无论成功或失败，都尽量发送 `done`。
- 客户端断开不再继续推送，避免后台堆积。
- `done` 中保留总耗时、是否有行程、重试次数等摘要。

## 6. 确认执行异常处理过程

确认阶段的主要风险是越权、重复执行和目标篡改。

| 风险 | 处理方式 | 结果 |
| --- | --- | --- |
| 非法 target | 确认请求中的目标必须来自当前 Itinerary | 阻止无关预约或购票 |
| 重复确认 | 会话和订单结果写回状态 | 后续可识别已处理结果 |
| 执行失败 | 预约、购票、转发分别记录失败 reason | 返回部分成功或明确失败说明 |
| 文案失败 | 执行结果保留，文案走默认说明 | 不影响已经完成的动作 |

## 7. 过程结论

- 异常处理分入口、模型、工具、SSE、确认五个层级。
- 每层只处理自己能判断的问题，不能把模糊错误继续向后传。
- 业务失败必须结构化，技术异常必须转成前端可见摘要。
- 兜底不是隐藏失败，而是把失败变成可解释、可继续的状态。

## 8. 待关注项

- provider 接入真实服务后，需要补充更细的重试次数和退避策略。
- 多人协作场景下，同一房间多条异常事件需要保持广播顺序。
- 确认执行需要继续补充幂等验证，避免网络重试导致重复下单。
