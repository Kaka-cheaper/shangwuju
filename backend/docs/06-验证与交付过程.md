# 验证与交付过程记录

## 1. 记录范围

本记录跟踪后端功能从代码检查、接口自测、协作验证到文档交付的过程。它不是结论清单，而是保留每类验证的触发点、观察点和通过标准。

## 2. 接口验证过程

| 验证项 | 触发方式 | 观察点 | 通过标准 |
| --- | --- | --- | --- |
| 健康检查 | `/health`、`/ready` | 服务是否启动、依赖是否可用 | 返回正常状态 |
| 首次规划 | `/chat/turn` 发送完整需求 | 是否有 intent、工具事件、itinerary | 最终收到 `itinerary_ready` 和 `done` |
| 反馈重规划 | 同一 session 发送“太远了”等反馈 | 是否识别为 feedback | 出现重规划事件和新方案 |
| 确认执行 | `/chat/confirm` | 是否调用预约、购票、转发 | 返回订单或明确失败说明 |
| 异常输入 | 空文本、非法字段、高风险文本 | 是否提前拒绝或结构化失败 | 不进入无意义规划 |

接口验证先看主路径，再看反馈、确认和异常路径。只跑一次正常规划不能代表系统稳定。

## 3. SSE 验证过程

| 阶段 | 必看事件 | 检查原因 |
| --- | --- | --- |
| 开始 | `agent_thought` | 前端能进入处理态 |
| 理解 | `intent_parsed` | 用户需求被结构化 |
| 工具 | `tool_call_start`、`tool_call_end` | 事实查询有过程可见性 |
| 调整 | `replan_triggered` | 失败或约束冲突能解释 |
| 成果 | `itinerary_ready` | 前端能渲染卡片和地图 |
| 收口 | `done` | loading 能结束，统计能写入 |

异常场景下至少要看到 `stream_error` 或明确的失败事件，并尽量以 `done` 收口。

## 4. 协作验证过程

| 场景 | 操作 | 通过标准 |
| --- | --- | --- |
| 创建房间 | owner 创建并进入房间 | 返回 room_id 和初始状态 |
| 成员加入 | 新用户连接 WebSocket | 收到成员列表、chat_state、planning_events |
| 过程同步 | owner 发起规划或约束 | 成员端同步阶段事件和方案 |
| 非 owner 确认 | 协作者发送 confirm | 返回错误或被拦截，不触发预约 |
| owner 确认 | 发起者发送 confirm | 广播确认结果和最终文案 |

协作验证要同时看前端表现和后端日志。前端不显示按钮只能证明体验正确，后端拦截才证明权限边界正确。

## 5. 回归验证过程

| 回归点 | 关注原因 | 检查方式 |
| --- | --- | --- |
| 地图节点 | 行程节点需要在前端地图中同步 | 检查 itinerary 中坐标和节点顺序 |
| 餐厅满座 | 容易触发替换或失败解释 | 使用不可用餐厅场景 |
| 空候选 | POI 或餐厅不足时容易中断 | 检查 FailureReason 和 fallback |
| 模型不可用 | LLM 配置或 provider 可能失败 | 检查规则路径和错误解释 |
| 多 turn 状态 | 连续反馈依赖 session | 同一 session 连续发送请求 |

回归验证按风险排序，优先覆盖影响用户操作闭环的场景。

## 6. 交付记录过程

交付时保留三类材料：

| 材料 | 内容 | 用途 |
| --- | --- | --- |
| 过程文档 | 入口、Planning、工具、异常、协作、验证过程 | 说明系统推进和问题处理脉络 |
| 系统说明文档 | 两页 Word 说明 | 用于整体汇报和评审阅读 |
| 验证记录 | 启动日志、截图、接口或浏览器验证结果 | 支撑功能已经跑通 |

文档交付前需要检查：文件名清晰、内容没有明显占位、过程记录能串起来、验证项和系统现状一致。

## 7. 过程结论

- 验证不能只看正常回答，要覆盖反馈、确认、异常和协作权限。
- SSE 的 `done` 是前端和自动化验证共同依赖的结束信号。
- 协作功能必须验证“同步过程”和“确认权限”两条线。
- 文档材料应保留过程脉络，避免只写最终架构结论。

## 8. 待关注项

- 后续真实 provider 接入后，需要补充外部服务失败和限流验证。
- 自动化验证脚本应继续沉淀到固定命令，减少人工复测成本。
- 交付材料更新后要同步检查 Word 文档和后端过程文档是否口径一致。
