# 入口与会话过程记录

## 1. 记录范围

本记录跟踪后端请求从 HTTP 入口进入，到会话状态识别、规划事件输出、确认执行完成的过程。重点关注三个问题：

- 前端是否只需要面对稳定入口。
- 同一 `session_id` 下的连续输入能否保持上下文。
- 流式输出异常时能否给前端明确收口事件。

## 2. 入口确认过程

| 节点 | 观察内容 | 处理结果 |
| --- | --- | --- |
| `/chat/turn` | 前端发起自然语言需求、反馈和闲聊都可进入同一端点 | 作为会话主入口，内部完成路线判断 |
| `/chat/confirm` | 只有用户确认方案后才进入执行动作 | 触发预约、购票、转发文案等执行类工具 |
| `/chat/stream` | 可直接跑一次完整规划流 | 保留为接口自检和独立规划调试入口 |
| `/chat/refine` | 可在已有方案基础上提交调整意见 | 保留为反馈链路调试入口 |

推进时先确认入口职责，再检查前端调用是否需要自行判断“新需求还是反馈”。结果是把判断放在 `/chat/turn` 内部，前端只传 `session_id`、`message`、`user_id` 等基础字段。

## 3. 会话识别过程

| 步骤 | 输入 | 处理 | 输出 |
| --- | --- | --- | --- |
| 读取请求 | body、header、环境变量 | 解析 `planner_mode`、`user_id`、`session_id` | 本轮请求上下文 |
| 查询状态 | `session_id` | 读取轻量 session、ConversationRepository、LangGraph thread | 最近方案和阶段快照 |
| 判断路线 | 用户文本、最近方案、房间状态 | 识别 planning、feedback、chitchat、安全拒答 | `route_kind` |
| 写回状态 | 事件、方案、最终结果 | 更新 session store 和会话仓库 | 下轮可复用的上下文 |

过程里重点排查过“用户已有方案后继续说太远了”这类输入。处理方式是先看是否存在行程快照，再结合反馈词、模型分类和保守兜底判断，避免把反馈误当成新需求。

## 4. 流式输出过程

SSE 输出按阶段推进，前端可以在每个阶段同步更新界面。

| 事件 | 出现场景 | 前端用途 |
| --- | --- | --- |
| `agent_thought` | 进入规划、准备工具、重试前 | 展示正在处理的阶段 |
| `intent_parsed` | 意图抽取完成 | 展示时间、距离、同行人、偏好 |
| `tool_call_start` / `tool_call_end` | 调用 POI、餐厅、画像、路线等工具 | 展示事实查询过程 |
| `replan_triggered` | critic 或工具结果要求调整 | 展示为什么重试 |
| `itinerary_ready` | 行程拼装完成 | 渲染方案、地图节点和确认入口 |
| `agent_narration` | 输出自然语言说明 | 给用户解释方案 |
| `done` | 本轮结束 | 解除 loading，写入最终统计 |

`safe_stream` 包装所有 SSE 流。内部异常会转换为 `stream_error` 和 `done`，客户端断开时静默退出，避免前端无限等待。

## 5. 过程结论

- `/chat/turn` 承担主入口职责，前端不需要重复实现路线判断。
- `session_id` 是跨 turn 状态主键，也是 LangGraph thread 和协作房间补齐状态的基础。
- `/chat/confirm` 与规划入口分离，执行类工具不会在规划阶段提前触发。
- SSE 每轮必须以 `done` 收口，这是前端 loading、日志和自动化验证的共同边界。

## 6. 待关注项

- session store 仍偏轻量，后续接入 Redis 或 Postgres 后需要验证恢复一致性。
- `/chat/stream` 和 `/chat/refine` 作为调试入口存在，接口说明里需要持续标注推荐调用方式。
- 前端如果并发发送同一 `session_id` 的请求，需要明确 UI 层的打断或排队策略。
