# 工具调用链路过程记录

## 1. 记录范围

本记录跟踪后端工具从注册、入参校验、调用、失败收敛到结果进入规划的过程。目标是让工具承担事实查询和执行动作，而不在工具内部做跨步骤编排。

## 2. 工具分组过程

| 分组 | 工具 | 进入时机 | 返回用途 |
| --- | --- | --- | --- |
| 候选搜索 | `search_pois`、`search_restaurants` | 意图抽取后 | 提供活动和餐厅候选 |
| 画像与路线 | `get_user_profile`、`estimate_route_time`、`lookup_hop` | 候选准备和拼装阶段 | 确定 home、交通偏好、节点通勤 |
| 可用性 | `check_restaurant_availability` | 餐厅入选后或确认前 | 判断时段是否可订 |
| 执行动作 | `reserve_restaurant`、`buy_ticket`、`generate_share_message` | 用户确认后 | 生成预约、购票和转发结果 |

推进过程中将工具分成查询类和执行类。查询类可以在规划阶段使用，执行类只能在确认阶段使用，避免“只是在规划”时提前预约。

## 3. 注册与 schema 过程

| 节点 | 处理内容 | 结果 |
| --- | --- | --- |
| registry | 维护工具名、入参模型、出参模型和调用函数 | 上层按工具名统一调用 |
| Pydantic schema | 校验输入和输出结构 | 避免自由文本穿透到业务层 |
| FailureReason | 收敛常见失败类型 | 上层按类型重试、降级或解释 |
| Provider | mock、gaode、dianping 等服务实现位 | 替换真实服务时不改主流程 |

工具失败时不直接抛业务异常给规划层，而是返回结构化失败。这样 Planning 可以根据 `EMPTY_CANDIDATES`、`RESTAURANT_FULL`、`INVALID_INPUT`、`UPSTREAM_FAILURE` 等 reason 做决策。

## 4. 查询类工具过程

| 过程节点 | 观察 | 处理 |
| --- | --- | --- |
| POI 搜索 | 距离、标签、同行人会影响候选数量 | 候选不足时保留放宽标签 |
| 餐厅搜索 | 价格、口味、人数和时段影响匹配 | 返回候选和失败 reason |
| 用户画像 | home、交通偏好、历史接受标签影响排序 | 缺失时使用默认画像 |
| 路线估算 | 节点间通勤决定 schedule 是否成立 | 失败时保留低置信估算 |

查询类工具的结果进入候选池，不直接决定最终方案。最终是否采用由 planner、assemble 和 critic 共同判断。

## 5. 执行类工具过程

| 过程节点 | 输入边界 | 输出边界 |
| --- | --- | --- |
| 预约餐厅 | 只能选择当前 Itinerary 中合法餐厅目标 | 返回确认时间、人数、订单状态 |
| 购票 | 只能选择当前 Itinerary 中可购票目标 | 返回票务结果或失败 reason |
| 转发文案 | 使用最终行程、订单和用户可读说明 | 返回可复制或分享的文本 |

执行类工具必须在用户确认后触发。确认入口会带上合法目标集合，后端再次校验目标是否来自当前方案，避免前端篡改 target_id 后执行无关预约。

## 6. 工具事件记录过程

| 事件 | 记录内容 | 用途 |
| --- | --- | --- |
| `tool_call_start` | 工具名、入参摘要、阶段 | 告知前端正在查询 |
| `tool_call_end` | success、reason、耗时、结果摘要 | 支持界面展示和问题复盘 |
| `replan_triggered` | 失败 reason、触发工具 | 解释为什么放宽或重排 |
| `done` | 本轮工具统计和最终策略 | 自动化验证和交付展示 |

工具事件不能泄露完整敏感入参，只保留摘要和业务需要的状态。

## 7. 过程结论

- 工具层只做事实查询和执行动作，不承担整体规划。
- 查询类工具可以参与规划，执行类工具只在确认后触发。
- 所有失败都应落到结构化 reason，方便重试、降级和解释。
- Provider 替换应保持工具 schema 不变，避免影响上层编排。

## 8. 待关注项

- 接入真实服务后，需要补充 provider 级别的超时、限流和鉴权记录。
- 工具结果摘要需要继续控制长度，避免 SSE 事件过大。
- 执行类工具需要和订单系统的幂等键对齐，避免重复确认。
