# 晌午局 · 设计文档

> 美团 AI Hackathon 命题赛道 06 · 本地探索 · 周末闲时活动规划
> 对应 `验收标准.md` A16 硬性交付项；目标 ≤ 2 页

---

## 一、项目定位（一句话）

用户输入一句自然语言 → AI Agent 编排 Tool → 输出可执行 + 可转发的下午半日行程，并通过 Mock API 完成关键预订/购票/转发动作。

---

## 二、四层架构

```text
┌── Web UI（Next.js 14 + SSE 消费）            聊天框 / 行程卡 / Tool 调用链路 / 转发文案
│
├── HTTP/SSE 网关（FastAPI + sse-starlette）   /chat/stream  /chat/confirm  /scenarios  /health
│
├── Agent 编排层（intent_parser + planner）    意图解析 → ReAct 循环 → 异常重规划 → 行程组装 → 执行
│
├── Tool 层（@register_tool 注册表，8 个）     查询类 4 + 执行类 3 + 画像 1
│
└── Mock 数据层（JSON）                        21 POI / 19 餐厅 / 71 路线 / 1 用户画像
```

四层硬纪律（AGENTS.md §3.3）：UI 不直连 LLM；Tool 不互调；Tool 不做规划决策；Mock 是只读快照。

## 三、Planning 策略

**规则化 ReAct**（避免 LLM 在循环里漂移；MVP-1 评分项可控）。

```text
1. 意图解析（LLM Function Calling，response_format=json_object）
   ↓ 输出 IntentExtraction（§5.7 D-SoT，三类 tag 词典出口约束）
2. get_user_profile          取家位置 / 默认预算 / 交通偏好
3. search_pois                按 distance / physical / experience / suitable_for 过滤
   ↓ 命中 0 条 → 距离 +2 km 重试 1 次（loosen_distance）
4. search_restaurants         按 distance / dietary / capacity / suitable_for 过滤
   ↓ 同上
5. 餐厅可用性协商             对前 3 家候选 × {17:00, 17:30, 18:00} 顺序尝试
   ↓ 命中第一个 available=true 即停
6. estimate_route_time × 3    home→POI / POI→Rest / Rest→home（取打车 > 步行 > 公交）
7. 组装六段 Itinerary          出发 / 主活动 / 转场 / 用餐 / 返回（附加可选）
8. （用户确认后）execute_plan  reserve_restaurant + buy_ticket（可选）+ generate_share_message
```

**为什么规则化而非 LLM 自由 ReAct**：

- Hackathon 评分函数：Demo 闭环 > Agent 行为可见 > 异常韧性 > Mock 真实 > 工程整洁。规则化让前 3 项可控、可重放、可调试。
- LLM 仅用于意图抽取（输入域开放）+ 转发文案润色（output 多样性），其余链路确定性优先。
- 接口位保留：planner 暴露 plan_itinerary(intent) 单一入口，将来切 LLM ReAct 不动外层。

**D9 硬条款**：Tool / Agent 代码对场景类型完全无感——8 场景共享同一套规则，仅靠 IntentExtraction 字段（physical_constraints / dietary_constraints / experience_tags / social_context / capacity_requirement）驱动。grep gate：`scene_type` / `relation_type` / `if scene ==` 在代码中必须为 0 命中。

## 四、Tool 调用链路

```text
查询类（4）：search_pois / search_restaurants / check_restaurant_availability / estimate_route_time
执行类（3）：reserve_restaurant / buy_ticket / generate_share_message
画像  （1）：get_user_profile
```

**统一入口**：`tools.registry.invoke_tool(name, raw_args)`

- 输入用 Pydantic v2 二次校验（拦截 LLM 漂移字段 → reason=invalid_input）
- 输出再校验一次（防 Tool 实现漂移 → reason=upstream_failure）
- 计时返 duration_ms（给 SSE TOOL_CALL_END 事件）
- 单 Tool 调用上限 3 次 / 总调用上限 12 次（防过度规划）

**主路径事件序列**（从 SSE 视角）：

```text
seq 0  intent_parsed
seq 1-2  tool_call_start/end (get_user_profile)
seq 3-4  tool_call_start/end (search_pois)
seq 5-6  tool_call_start/end (search_restaurants)
seq 7-8  tool_call_start/end (check_restaurant_availability time=17:00)  ← 失败时
seq 9    replan_triggered (reason=restaurant_full)
seq 10-11 tool_call_start/end (check_restaurant_availability time=17:30) ← 备选成功
seq 12-17 tool_call_start/end (estimate_route_time × 3)
seq 18   itinerary_ready (Itinerary)
seq 19   done
```

## 五、异常处理机制

四类异常 + 显式恢复策略：

```text
| reason            | 触发条件                          | 恢复策略                              |
|-------------------|-----------------------------------|---------------------------------------|
| empty_candidates  | search_pois/restaurants 命中 0    | distance_max_km + 2 km 重试 1 次       |
| restaurant_full   | check_availability available=false| 同店换时段 → 全满则切下一家备选餐厅   |
| ticket_sold_out   | buy_ticket capacity=0             | 替换同类 POI（执行类失败不阻塞主流）  |
| upstream_failure  | Tool 内部异常 / 输出 schema 漂移   | 调用计数器 +1，超限走 abort           |
```

**E1 主路径已端到端验证**（test_e1_restaurant_full_recovery_in_family_scene）：mock R001 17:00 埋 available=false → planner 触发 replan_triggered 切到 17:30 → 行程正常组装。

**E2 已端到端验证**（test_e2_ticket_sold_out_recovery）：mock P_SOLD 埋 capacity.available_slots=0 → buy_ticket 返 reason=ticket_sold_out。

**降级原则**：Tool 失败回结构化 reason，Agent 层判断后续路径，**不抛业务异常**给 SSE 流；流中途若必须终止，推一条 stream_error 后立刻 done，HTTP 状态仍 200。

## 六、关键设计取舍

```text
| 取舍点               | 选择                          | 理由                              |
|----------------------|-------------------------------|-----------------------------------|
| LLM 编排 vs 规则编排 | 规则编排（MVP）+ LLM 抽取意图 | Demo 可控，评分硬指标可达          |
| 意图开放 vs 场景枚举 | 全开放（D9）                  | 评委可即兴扔输入；Tool 代码无感    |
| 一气呵成 vs 显式确认 | MVP-1 一气呵成；MVP-2 加确认  | Demo 紧凑 + 题目原意双轨           |
| 真 API vs Mock       | 全 Mock                       | 题目禁止 + 节省时间                |
| 单 LLM vs 多 LLM     | DeepSeek 主 + 通义备 + Stub   | OpenAI 兼容 SDK 仅切 base_url      |
```
