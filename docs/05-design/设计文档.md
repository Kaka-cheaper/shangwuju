# 晌午局 · 设计文档

> 美团 AI Hackathon 命题赛道 06 · 本地探索 · 周末闲时活动规划
> 对应 `验收标准.md` A16 硬性交付项；目标 ≤ 2 页

---

## 一、项目定位（一句话）

用户输入一句自然语言 → AI Agent 编排 Tool → 输出可执行 + 可转发的下午半日行程，并通过 Mock API 完成关键预订/购票/转发动作。

---

## 二、四层架构

```text
┌── Web UI（Next.js 14 + SSE 消费）            聊天框 / 行程卡 / Tool 调用链路 / 转发文案
│
├── HTTP/SSE 网关（FastAPI + sse-starlette）   /chat/stream  /chat/confirm  /scenarios  /health
│
├── Agent 编排层（intent_parser + planner）    意图解析 → ReAct 循环 → 异常重规划 → 行程组装 → 执行
│
├── Tool 层（@register_tool 注册表，8 个）     查询类 4 + 执行类 3 + 画像 1
│
└── Mock 数据层（JSON）                        21 POI / 19 餐厅 / 71 路线 / 1 用户画像
```

四层硬纪律（AGENTS.md §3.3）：UI 不直连 LLM；Tool 不互调；Tool 不做规划决策；Mock 是只读快照。

## 三、Planning 策略

**规则化 ReAct**（避免 LLM 在循环里漂移；MVP-1 评分项可控）。

```text
1. 意图解析（LLM Function Calling，response_format=json_object）
   ↓ 输出 IntentExtraction（§5.7 D-SoT，三类 tag 词典出口约束）
2. get_user_profile          取家位置 / 默认预算 / 交通偏好
3. search_pois                按 distance / physical / experience / suitable_for 过滤
   ↓ 命中 0 条 → 距离 +2 km 重试 1 次（loosen_distance）
4. search_restaurants         按 distance / dietary / capacity / suitable_for 过滤
   ↓ 同上
5. 餐厅可用性协商             对前 3 家候选 × {17:00, 17:30, 18:00} 顺序尝试
   ↓ 命中第一个 available=true 即停
6. estimate_route_time × 3    home→POI / POI→Rest / Rest→home（取打车 > 步行 > 公交）
7. 组装六段 Itinerary          出发 / 主活动 / 转场 / 用餐 / 返回（附加可选）
8. （用户确认后）execute_plan  reserve_restaurant + buy_ticket（可选）+ generate_share_message
```

**为什么规则化而非 LLM 自由 ReAct**：

- Hackathon 评分函数：Demo 闭环 > Agent 行为可见 > 异常韧性 > Mock 真实 > 工程整洁。规则化让前 3 项可控、可重放、可调试。
- LLM 仅用于意图抽取（输入域开放）+ 转发文案润色（output 多样性），其余链路确定性优先。
- 接口位保留：planner 暴露 plan_itinerary(intent) 单一入口，将来切 LLM ReAct 不动外层。

**D9 硬条款**：Tool / Agent 代码对场景类型完全无感——8 场景共享同一套规则，仅靠 IntentExtraction 字段（physical_constraints / dietary_constraints / experience_tags / social_context / capacity_requirement）驱动。grep gate：`scene_type` / `relation_type` / `if scene ==` 在代码中必须为 0 命中。

## 四、Tool 调用链路

```text
查询类（4）：search_pois / search_restaurants / check_restaurant_availability / estimate_route_time
执行类（3）：reserve_restaurant / buy_ticket / generate_share_message
画像  （1）：get_user_profile
```

**统一入口**：`tools.registry.invoke_tool(name, raw_args)`

- 输入用 Pydantic v2 二次校验（拦截 LLM 漂移字段 → reason=invalid_input）
- 输出再校验一次（防 Tool 实现漂移 → reason=upstream_failure）
- 计时返 duration_ms（给 SSE TOOL_CALL_END 事件）
- 单 Tool 调用上限 3 次 / 总调用上限 12 次（防过度规划）

**主路径事件序列**（从 SSE 视角）：

```text
seq 0  intent_parsed
seq 1-2  tool_call_start/end (get_user_profile)
seq 3-4  tool_call_start/end (search_pois)
seq 5-6  tool_call_start/end (search_restaurants)
seq 7-8  tool_call_start/end (check_restaurant_availability time=17:00)  ← 失败时
seq 9    replan_triggered (reason=restaurant_full)
seq 10-11 tool_call_start/end (check_restaurant_availability time=17:30) ← 备选成功
seq 12-17 tool_call_start/end (estimate_route_time × 3)
seq 18   itinerary_ready (Itinerary)
seq 19   done
```

## 五、异常处理机制

四类异常 + 显式恢复策略：

```text
| reason            | 触发条件                          | 恢复策略                              |
|-------------------|-----------------------------------|---------------------------------------|
| empty_candidates  | search_pois/restaurants 命中 0    | distance_max_km + 2 km 重试 1 次       |
| restaurant_full   | check_availability available=false| 同店换时段 → 全满则切下一家备选餐厅   |
| ticket_sold_out   | buy_ticket capacity=0             | 替换同类 POI（执行类失败不阻塞主流）  |
| upstream_failure  | Tool 内部异常 / 输出 schema 漂移   | 调用计数器 +1，超限走 abort           |
```

**E1 主路径已端到端验证**（test_e1_restaurant_full_recovery_in_family_scene）：mock R001 17:00 埋 available=false → planner 触发 replan_triggered 切到 17:30 → 行程正常组装。

**E2 已端到端验证**（test_e2_ticket_sold_out_recovery）：mock P_SOLD 埋 capacity.available_slots=0 → buy_ticket 返 reason=ticket_sold_out。

**降级原则**：Tool 失败回结构化 reason，Agent 层判断后续路径，**不抛业务异常**给 SSE 流；流中途若必须终止，推一条 stream_error 后立刻 done，HTTP 状态仍 200。

## 六、关键设计取舍

```text
| 取舍点               | 选择                          | 理由                              |
|----------------------|-------------------------------|-----------------------------------|
| LLM 编排 vs 规则编排 | 规则编排（MVP）+ LLM 抽取意图 | Demo 可控，评分硬指标可达          |

| 单 LLM vs 多 LLM     | DeepSeek 主 + 通义备 + Stub   | OpenAI 兼容 SDK 仅切 base_url      |
```

## 八、双范式 Planner（Phase 0.6）

### 设计动机

```text
评分项 1（场景理解）+ 项 2（规划链路）有两种实现路径：
- rule  ：规则代码写死 Tool 调用顺序，可调试 / 可重放 / Demo 不翻车
- llm   ：LLM 看 8 个 Tool spec 自主决策，更体现「Agent」而非「模板系统」
两者各有得分，**保留双轨**通过环境变量 / HTTP header 切换。
```

### 接口形态

```text
统一入口：plan_itinerary_with_mode(intent, mode)
分发：
  mode="rule" → planner.plan_itinerary（默认）
  mode="llm"  → llm_planner.plan_itinerary_llm
mode 优先级：HTTP header X-Planner-Mode > 环境变量 PLANNER_MODE > "rule"
```

### llm 范式纪律

```text
- 仅暴露查询类 5 个 Tool 给 LLM（执行类 reserve / buy_ticket / generate_share_message 在用户确认后由后端规则代码下发）
- 单 Tool 调用上限 3 次 / 总调用上限 12 次（防 LLM 死循环）
- 主循环最多 8 turn（finish_reason=stop 提前终止）
- 任何异常 / state 不齐 → 自动 fallback 到 rule 范式（沿用同一个 tracer 不丢事件）
- LLM 只决定「调哪些 Tool 拿哪些候选」；六段时间轴拼装由 _assemble_itinerary 规则代码完成（确定性优先）
```

### 失败 fallback 链路

```text
LLM 异常 / 决策不收敛 → llm_planner 内部 fallback：
  1. trace 推 agent_thought {"text": "LLM 自主规划异常，已切回规则规划"}
  2. plan_itinerary(intent, tracer=同一个) 复跑
  3. 前端 ToolTracePanel 看到完整事件链 + 一条提示
评委体验：即使真 LLM 抖动，Demo 仍能跑通；切换演示不需要重启后端
```

## 九、反馈重规划（Phase 0.6）

### 设计动机

```text
用户面对 Itinerary 不满意时，原方案只有「确认 / 取消」二选一，
缺失「拒绝 + 反馈说明 → Agent 据此调整」的闭环。
这是「Agent」与「聊天机器人」的核心区分（评分项 1+5 加分点）。
```

### 三态用户操作

```text
[确认]              → /chat/confirm 调执行类 Tool
[我说说哪不对]      → 弹反馈对话框（textarea 可空）→ /chat/refine
[取消方案]          → 前端 store reset，会话 ↑↓
```

### refiner 双层防御

```text
LLM 调用 1 次（response_format=json_object）+ Pydantic 二次校验
  ↓ 失败
错误回灌 LLM 重试 1 次
  ↓ 仍失败
_rule_fallback 关键词解析（启发式兜底）
  - "太远了" / "近一点"  → distance_max_km × 0.6 → 最低 2km
  - "便宜点"              → 去高人均 / 商务体面，加健康轻食
  - "时间紧" / "时间多"   → duration_hours 调整
  - 反馈空                → distance_max_km - 1km 让候选打散
关键：refine 端到端**必有**降级路径，Demo 不会因为 LLM 翻车而转圈
```

### SSE 事件序列

```text
seq 0  refinement_start    payload={"feedback_text": "..."}
seq 1  refinement_done     payload=RefinementOutput.model_dump()
                            （含 refined_intent / changed_fields / refiner_note）
seq 2-N  完整复用 /chat/stream 主路径事件序列
seq N+1 done
```

### 关键防御要点

```text
- raw_input 字段强制保留首次输入（refiner 不允许漂移）
- refined_intent 必须是合法 IntentExtraction（§5.7 D-SoT）
- D9 禁止字段（scene_type / relation_type）由 extra="forbid" 拦截
- 词典外 tag 由 Literal 拦截
- changed_fields 是中文人话（如「距离上限：5km → 3km」），前端直接展示给用户
```

## 十二、个性化（persona + memory，Phase 0.7）

### 设计动机

```text
评委观察：「这是怎么知道我喜欢的？」
回答：persona（我是谁，写死）+ memory（学到的偏好，累积）双驱动意图解析。
评分项 1（场景理解）+5-10 分：同句话不同 user 出不同方案。
```

### 5 个 mock persona

```text
| user_id     | 标签       | default_distance | suitable_for_priority    |
|-------------|-----------|------------------|--------------------------|
| u_dad       | 新手爸爸   |  5 km            | 家庭日常                  |
| u_biz       | 商务白领   |  8 km            | 商务接待 / 纪念日仪式感   |
| u_grandma   | 孝顺儿女   |  3 km            | 老人伴助 / 纪念日仪式感   |
| u_solo      | 独居青年   |  5 km            | 独处放空                  |
| u_couple    | 情侣党     |  6 km            | 情侣亲密 / 闺蜜聊天       |
```

### Memory 累积时机

```text
| 事件                        | 动作                                       |
|-----------------------------|-------------------------------------------|
| confirm 下单                | accepted_tags 累加 itinerary 命中的 tag    |
| confirm 下单                | distance_history 追加（保留最近 20）       |
| refine 反馈                 | refined.tags - original.tags = rejected   |
| 用户点「清空记忆」           | reset_memory(user_id)                     |
```

### 合并打分

```text
final_score(tag) =
    persona_default_base × persona_weight        # 3 × 0.3 = 0.9
  + memory.accepted_count × memory_weight        # 每次 +0.7
  - memory.rejected_count × memory_weight × 1.5  # 强惩罚
top_priors = sorted(score, desc)[:5]，保留 score>0
建议距离 = memory 中位数 || persona.default
```

### Prompt 注入纪律（保守补全，防候选过严）

```text
1. social_context 必注：用户没明示场景时用 persona.suitable_for_priority[0]
2. distance_max_km 必注：用户没明示距离时用建议默认距离
3. physical/dietary/experience 默认空：
   - 用户输入有暗示按用户的来
   - 用户完全无暗示**保持空**（避免双重过滤导致候选过严）
   - 例外：用户明确说"按我平常的来"才补 1 个最高优先 tag
4. 用户输入永远优先；冲突时 prior 让步并写入 ambiguous_fields
```

### Planner 五级降级（应对 prior 过严的兜底）

```text
1. 原约束（distance + tags + social_context + preferred_types）
2. 距离 +2km
3. 剥 preferred_types
4. 剥 physical + experience（prior 注入的偏好让步）
5. 仅 distance + social_context（最宽松，仍调性匹配）
search Tool quota 3→5 / 总 quota 12→16
```

### 浏览器实测（核心 Demo「哇时刻」）

```text
同一句话「今天下午想出去玩」：
  切「商务白领」→ 云岚商务雅集茶室 + 金樽商务日料（distance=8）
  切「新手爸爸」→ 悦读亲子绘本馆 + 鲸落健康简餐（distance=5，亲子+低脂注入）

确认下单 → 偏好画像「最近接受 top 5」立刻 +1：
  - 家庭日常 ×1 / 健康轻食 ×1 / 适合 5-10 岁 ×1 / 拍照友好 ×1 / 低强度 ×1
```

### 与 D9 的边界

```text
✓ persona 是 user 维度（"我是谁"）：5 个 mock 写死
✗ scene 是输入维度（"这次想做什么"）：仍由 LLM 动态抽取
两者**不冲突**：Tool 仍对 scene 类型无感（不写 if scene == "family"）；
persona prior 仅影响 social_context / 距离 / tag 的默认值。
```

## 十四、ReAct 单一 Agent + 商业演进抽象层（Phase 0.11/0.12）

### 设计动机

```text
| 旧痛点（截至 Phase 0.10.3）              | 新解法                                |
|------------------------------------------|---------------------------------------|
| dock 直接输入反馈被当成新需求重解析     | LLM 看 message_history 自主判断       |
| 段决策耦合主客观（hybrid 仍 if 启发式） | LLM 决主观 + critic 验客观（LLM-Modulo） |
| 18 .py + 5878 行手写 LLM SDK / retry    | 引入 Pydantic AI 业内标准框架         |
| 无数据源 / 持久化 / 观测性可演进路径   | Protocol 抽象 + Stub 实现三层骨架     |
```

### ReAct 单一 Agent 主体

`backend/agent/v2/react_agent.py`：unified_agent: Agent[AgentDeps, AgentOutput]

```text
output_type = Union[ChatResponse, ItineraryResponse]
  → LLM 自选输出形态：闲聊 / Q&A / 拒答 → ChatResponse；完整规划 → ItineraryResponse
  → 不需要外部分类器（router）

8 工具 @unified_agent.tool 装饰参数化展开
  → 不传整个 Input 模型给 LLM，参数 description 双语（中文 + 英文括号）
  → MiMo / GPT 都稳

@output_validator 接 critics_v2.validate_itinerary
  → critical 违规 → ModelRetry → LLM 自纠错
  → warning 不触发重做（避免 LLM 过度修复劣化方案）
  → retries=3 给 critic backprompt 留循环空间
```

### Critic 兜底层（LLM-Modulo 教科书级实现）

`backend/agent/v2/critics_v2.py` 7 类 ViolationCode + 2 级 Severity：

```text
| ViolationCode                | Severity | 触发                                 |
|------------------------------|----------|--------------------------------------|
| DURATION_OUT_OF_RANGE        | CRITICAL | 总时长越出 intent.duration_hours 容差 |
| DISTANCE_EXCEEDED            | WARNING  | stage 距家 > distance_max_km + 0.5km |
| STAGES_INCOMPLETE            | CRITICAL | 段集合缺主活动 / 用餐 / 返回         |
| RESTAURANT_FULL_UNRESOLVED   | CRITICAL | 用餐 17:00 触发（mock 满座埋点）     |
| TIMELINE_INCONSISTENT        | CRITICAL | 段间反序超 5min 容差                 |
| SOCIAL_CONTEXT_MISMATCH      | WARNING  | 独处场景预约多人位                   |
| DIETARY_VIOLATION            | WARNING  | 餐厅 tags 未覆盖 dietary_constraints |
```

## 附录 A · ReAct 单一 Agent 范式说明

### A.1 范式选择

晌午局采用 **ReAct 单一 Agent**（Reason + Act 循环 + Function Calling），而非「多 Agent 协作」。


```text
┌──────────────────────────────────────────────────┐
│  Single Agent (LLM + Function Calling)           │
│  ┌────────────────────────────────────────────┐  │
│  │  Reason  推理下一步该调哪个 Tool / 该停了   │  │
│  │  Act     调用 Tool 拿结果                   │  │
│  │  Observe 结果回灌上下文                     │  │
│  │  loop 直到 finish_reason=stop 或达上限      │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

- **Reason**：LLM 在每个 turn 看候选 Tool 集合 + 已收集的 observation，决定下一步行动（调 Tool 或终止）
- **Act**：调用选中的 Tool（OpenAI Function Calling 的 tool_calls 字段）
- **Observe**：Tool 返回结构化结果，写回 messages 作为下一轮 Reason 的输入

### A.2 为什么不做多 Agent

```text
| 方案              | 优点                       | 缺点（Hackathon 致命点）                |
|-------------------|----------------------------|----------------------------------------|
| 多 Agent 协作      | 职责清晰、可独立优化         | 跨 Agent 状态同步复杂、Demo 翻车面广   |
| 单 Agent + Tool 层 | Demo 可控、状态收敛于 messages | 单 LLM 抖动会传染整个链路（用 fallback 防）|
```

我们选「单 Agent + 8 个 Tool + 双 mode + 关键词 fallback」三层防御，覆盖了多 Agent 的鲁棒性，又规避了多 Agent 的 Demo 翻车面。

### A.3 关键防御

- **mode 切换**：`X-Planner-Mode: rule | llm` HTTP header 或 `PLANNER_MODE` 环境变量；rule 是 Demo 安全网，llm 是评分加分
- **call quota**：单 Tool 上限 3 次 / 总 Tool 上限 12 次 / 主循环上限 8 turn；防 LLM 死循环
- **schema 二次校验**：Tool 输入 / 输出在 Tool 入口都用 Pydantic v2 校验，拦截 LLM 漂移字段
- **fallback 链路**：LLM 异常 → fallback rule 范式 → 同一个 tracer 不丢事件 → 前端 ToolTracePanel 仍能完整呈现

详见 `backend/agent/llm_planner.py` 与 `docs/01-requirements/架构选型.md` D-SoT。

### A.4 与未来「多 Agent 演进」的边界

ReAct 单 Agent 在 v1 已足够。如果业务复杂到必须拆 Agent（如「家庭场景需要 KidAgent 专家 + 老人 Agent 专家协作」），届时拆分边界应是 **Tool 集合而非 Reason 链路**——单 Agent 仍是入口，专家 Agent 作为 Tool 暴露。

## 附录 B · 跨 turn 上下文管理

### B.1 上下文类型

```text
| 上下文类型      | 生命周期        | 存储位置                  | 持久化策略             |
|-----------------|-----------------|--------------------------|----------------------|
| persona         | 长期（用户级）   | mock_data/personas.json   | MVP 切 Redis           |
| memory          | 长期（用户级）   | InMemoryRepository（dict）| MVP 切 Redis 真产品 PG |
| conversation    | 中期（会话级）   | session 内 messages 列表  | MVP 切 Redis TTL=24h   |
| intent / itinerary | 短期（单 turn）| SSE 事件流 + Pydantic 模型 | 不持久化               |
| trace events    | 短期（单 turn）| Tracer 内存队列 + SSE     | 真产品切 OpenTelemetry |
```

### B.2 跨 turn 关键信息流

```text
turn N（首次规划）：
  user_input → router 分类（planning / chitchat / ...）
            → intent_parser（注入 persona prior + memory）
            → planner（rule / llm）→ Itinerary
            → 用户看 Itinerary

turn N+1（反馈重规划）：
  feedback_text + 上一 turn 的 intent / itinerary
                → refiner（合并约束）
                → 复用主路径事件序列
                → 新 Itinerary

turn N+2（确认）：
  itinerary_id + user_id
                → executor（reserve + buy_ticket + share_message）
                → memory.accepted_tags 累加
                → memory.distance_history 追加（保留最近 20）
```

### B.3 关键防御

- **raw_input 强制保留**：refiner 不允许漂移首次输入（D-SoT §5.7 IntentExtraction.raw_input 字段必填）
- **memory 衰减**：accepted_tags / rejected_tags 各保留最近 20 条，防长期偏好漂移
- **session ID 透传**：HTTP header `X-Session-Id` 在 stream / refine / confirm 三端点贯穿，refiner 不需要重传 itinerary 全文（仅传 session_id 拿上次结果）
- **D9 边界**：persona 是 user 维度（"我是谁"），scene 是输入维度（"这次想做什么"）。Tool / Agent 仍对 scene 类型无感；persona prior 仅影响 social_context / 距离 / tag 默认值，不影响 Tool 选择

### B.4 演进路径

`backend/agent/v2/conversation.py`（已实现）暴露 `ConversationRepository` 抽象，当前实现 `InMemoryRepository`（单进程 dict）。MVP 阶段切 `RedisRepository` 实现多实例共享 + 跨设备同步，真产品切 `PostgresRepository` 实现永久存档 + 用户行为 analytics。详见 `docs/06-business/02-持久化演进.md`。
