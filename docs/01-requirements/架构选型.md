## 二、决策项

### D1. LLM 提供商选型

**背景**：Agent 编排核心需要 LLM 做意图解析 + Tool Calling + 决策。需要选 1 个主用模型。

**候选**：

| 候选 | Function Calling | 中文 | 价格（百万 token） | 速度 | 备注 |
|---|---|---|---|---|---|
| DeepSeek | ✅ 好 | ✅ 母语 | 输入 ¥1 / 输出 ¥2 | 快 | `技术架构.md` 默认推荐 |
| 通义千问（qwen-max） | ✅ 好 | ✅ 母语 | 输入 ¥40 / 输出 ¥80 | 中 | `技术架构.md` 备选 |
| GPT-4o-mini | ✅ 优秀 | ✅ 流畅 | 输入 ~$0.15 / 输出 ~$0.6 | 快 | 海外 |
| Claude Sonnet 4 | ✅ 优秀 | ✅ 流畅 | 输入 ~$3 / 输出 ~$15 | 中 | 海外 |
| Hunyuan / 文心 / 智谱 | 各异 | ✅ 母语 | 中位 | 中 | 国内备选 |

**评估维度**：

- **Function Calling 稳定性**——hackathon 期间不能因为模型 hallucinate Tool 名 / 参数错位浪费时间
- **价格**——hackathon 演示期间多次 Demo 跑，但单次成本都不高，**价格不是首要因素**
- **网络可达性**——国内 hackathon 现场网络下，海外 API 可能不稳；国内模型更安全
- **Demo 现场延迟**——评委盯着 demo 跑时不能让他们等 5 秒以上

**选择**：✅ **DeepSeek-V3 主 + 通义 Qwen-Plus 备**（2026-05-08 已拍板）

**理由**：

- Function Calling 稳定性 ⭐⭐⭐⭐⭐；中文意图抽取质量 ⭐⭐⭐⭐⭐
- 国内直连无需 VPN；首 token < 2s
- 价格便宜（输入 ¥1/输出 ¥2 每百万 token）
- OpenAI 兼容 SDK 统一客户端——`openai` Python 包 + `base_url="https://api.deepseek.com/v1"`；切用通义只需换 base_url
- 1 个月时间盒够做「对比演示」（Week 4 加分项）

**影响**：

- 编码时写 `LLMClient` wrapper 支持 `provider` 参数在 deepseek/qwen 之间切换
- session 2 立即需 user 提供 DeepSeek API Key（写到 `.env`）
- 通义 Qwen-Plus API Key 延迟到 Week 4（多 LLM 对比阶段）再申请

**状态**：✅ **已决策 [2026-05-08]**

### D2. 是否做 Web UI

**背景**：`技术架构.md` 强烈建议做 Next.js + shadcn/ui Web UI；`比赛详情.md` 赛道 06 允许 CLI / Web UI。Web UI 是加分项但耗时。

**候选**：

| 选项 | 开发耗时 | 演示效果 | Demo 录屏便利 |
|---|---|---|---|
| 选项 A：Next.js + shadcn/ui + Tailwind | 6-10h | ⭐⭐⭐⭐⭐ | 容易 |
| 选项 B：纯 HTML + 原生 JS（vanilla） | 3-5h | ⭐⭐⭐ | 容易 |
| 选项 C：Streamlit / Gradio | 2-3h | ⭐⭐⭐⭐ | 容易 |
| 选项 D：纯 CLI + 录屏 | 1h | ⭐⭐ | 中 |

**评估维度**：

- 团队是否有前端经验？有 → 选项 A；无 → 选项 C（Python 一把梭）
- 时间盒预算多少给前端？
- 是否打算把 Tool 调用链路做成可视化（`技术架构.md` §2.1 加分项）？做 → 选项 A 最爽，C 次之

**选择**：✅ **选项 A：Next.js 14 App Router + FastAPI + SSE 流式**（2026-05-08 已拍板）

**理由**：

- 团队（D-team）至少 1 人有 React/Next.js 实战经验，无学习成本
- 1 个月时间够磨出精致 UI，视觉冲击直接对应评分项 §六 "评委 5 分钟观察清单"第 2 / 6 件哇时刻
- Tool 调用链路可视化——`技术架构.md` §2.1 加分项
- SSE 流式输出（vs 等 5s 按钮转圈）——评委体验多一维

**具体技术栈**：

- 前端：Next.js 14 App Router + TypeScript + Tailwind CSS + shadcn/ui
- 后端：FastAPI + Pydantic v2 + `sse-starlette` （见 D6）
- 通讯：SSE 流式 + REST（初始化 / 确认步）

**影响**：

- `frontend/` 目录立刻可创建（D9 文档纪律解锁）
- D6 / D7 自动级联为方案 A

**状态**：✅ **已决策 [2026-05-08]**

### D3. Tool 边界划分

**背景**：`需求分析.md` §M3 列了 8 个必有 + 2 个可选 Tool。需要确认精确数量与边界。

**候选**：

**方案 A（极简）**：6 个 Tool

```text
查询：search_pois / search_restaurants / check_restaurant_availability
执行：reserve_restaurant / generate_share_message
画像：get_user_profile
```

→ MVP-1 需求，最快闭环；缺门票 / 路线 / 异常分支 E2

**方案 B（标准，推荐）**：8 个 Tool

```text
查询：search_pois / search_restaurants / check_restaurant_availability / estimate_route_time
执行：reserve_restaurant / buy_ticket / generate_share_message
画像：get_user_profile
```

→ 覆盖 MVP-2 必做项；触发 E1 + E2 异常；评分项 100% 满足

**方案 C（丰满）**：10 个 Tool

```text
方案 B + order_extra_service + check_weather
```

→ 多 2 个加分点；但 check_weather 可能"花瓶"，order_extra_service 演示价值有限

**预选**：**方案 B**（8 个）

**理由**：

- `比赛详情.md` 评分项偏好"宁少勿滥"
- 8 个能覆盖所有必做评分点（A1-A15），再加只是冗余
- `技术架构.md` §2.3"Tool 数量控制在 8-10 个"——8 是下限，刚好

**影响**：MVP-1 仍是 6 个；MVP-2 补 2 个（`buy_ticket` + `estimate_route_time`）

**状态**：✅ **预决策**（session 1 user 确认即可锁定）

### D4. Mock 数据规模与字段密度

**背景**：`技术架构.md` §2.4 建议 ≥ 20 POI / ≥ 30 餐厅。需要确认字段密度。

**字段最低集**（必有，D9 后扩充）：

```yaml
POI:
  - id, name, type, location, distance_km
  - opening_hours, rating, age_range, price_range, tags[]
  - capacity, available_slots（用于异常 E2）
  - suitable_for[]: 适用社交上下文集合 (家庭日常/朋友热闹/情侣亲密/老人伴助/...)
Restaurant:
  - id, name, cuisine, location, distance_km
  - opening_hours, avg_price, rating
  - capacity: { 2: bool, 4: bool, 6: bool, 8: bool, private_room: bool }
  - reservation_slots: [{time, available, queue_minutes}]（用于异常 E1 + 排队逻辑）
  - tags[]: 饮食 + 体验 + 社交 全集
  - suitable_for[]: 同 POI
Route:
  - from_location, to_location
  - walking_minutes, taxi_minutes, bus_minutes
```

**Tag 词典**（必预定义，D9 后扩为三类，避免 Tool 调用时 tag 散乱）：

- **人物/物理 tags**：`亲子友好` `适合 5-10 岁` `适合青少年` `适合老人` `无台阶` `可休息` `无障碍` `高强度` `低强度`
- **饮食 tags**（仅 Restaurant）：`低脂` `健康轻食` `高蛋白` `日料` `粤菜` `不辣` `无牛肉` `有儿童餐` `高人均` `有包间`
- **体验/社交 tags**：`拍照友好` `网红打卡` `安静聊天` `热闹` `社交` `独处舒缓` `商务体面` `礼仪感` `亲密情侣` `学习成长` `室内` `户外`

**规模**（D9 后扩大，必覆盖 ≥ 6 种社交上下文）：

- **POI**：30-40 个，跨覆盖以下走向：
  - 亲子 ≥ 8（家庭主场景）
  - 社交 / 打卡 ≥ 8（朋友 / 闺蜜）
  - 老人友好 ≥ 4（带父母）
  - 情侣亲密 ≥ 4（看展 / 咖啡 / 安静）
  - 独处舒缓 ≥ 3（独自图书馆 / SPA / 书店）
  - 商务体面 ≥ 3（高端会所 / 商务会馆）
- **Restaurant**：50-60 个，跨覆盖以下走向：
  - 健康轻食 ≥ 12
  - 亲子餐 ≥ 8
  - 网红聚会 ≥ 10
  - 老人友好 / 软烂 ≥ 6
  - 安静情侣 ≥ 6
  - 商务高人均包间 ≥ 6
  - 独处友好 / 单人坐位 ≥ 4
- **Route**：50-80 条（POI ↔ Restaurant + POI ↔ POI）

**状态**：✅ **预决策**（随 D9 同步扩充，不需另行拍板）

**实施时检查**：用 jq / py 脚本统计字段覆盖率，确保每条数据 ≥ 8 个有效字段；且检查每个 `suitable_for` 走向都有 ≥ 1 条「可预订成功」与 ≥ 1 条「满/售罄」（跨场景都能触发异常）

### D5. 用户确认交互模式

**背景**：`比赛详情.md` 提到"用户确认 → 自动完成下单"。是否实现确认步骤？

**候选**：

- **A. 一气呵成**：LLM 决策完直接调执行类 Tool，输出已预约方案（MVP-1 默认）
- **B. 显式确认**：输出方案 → 等用户回 "确认" → 才调执行类 Tool（更贴题目原意）
- **C. 双输出**：Demo 同时演示两种模式（A 给评委看快、B 给评委看可控）

**预选**：**MVP-1 用 A，MVP-2 加 B**

**理由**：

- A 闭环更快、Demo 更紧凑（5 分钟限时下重点演示 A）
- B 更贴题目原意（"小明递给老婆"），但需要多一步用户输入，演示拖时间
- C 演示价值低、徒增复杂

**状态**：✅ **预决策**

- LLM 客户端用 `openai` Python SDK（兼容 DeepSeek + 通义，仅需切 base_url）

### D8. 部署 & 演示形态

**背景**：hackathon 演示是现场跑还是录屏？

**预选**：

- **首选**：现场跑（评委直接看实时输入输出）

### D9. 场景策略（D 全开放底层 + A 演示形态）

**背景**：赛题原文只举 2 个场景示例（家庭/朋友）。是否拿 2 个场景作为总体范围？

**背后问题**：评委看到“并行跑 N 个场景”是“模板系统”，看到“处理任意输入”才是“Agent”——两者评分颗粒度不同。

**候选**：

| 候选 | 说明 | 评分上限 | 开发量 |
|---|---|---|---|
| A. 扩关系类型枚举 | 在家庭/朋友之外加情侣/带父母/独处 等，仍然是枚举式 | 中（3 等 / 优秀） | +8h |
| B. 同类型内部细分 | 家庭拆 1大1孩 / 2大1孩 / 3代同堂 | 低-中 | +6h |
| C. 加环境约束轴 | 保留 2 场景，加雨天/高峰占用 | 中（与异常处理重叠） | +6h |
| **D. 全开放** | Agent 接任意自然语言，场景 = LLM 动态抽取的约束集 | 高（一等 / 商业星途） | +20h |
| **D + A 演示** | 底层走 D；Demo 选 6-8 个关系类型作为压力测压过的“快捷按钮” | **高 + 低翻车风险** | +25h |

**理由**：

1. D 底层保证　`场景理解准确度` 评分项拿满分；合题目原意
2. A 演示保证　现场 5 分钟 Demo 不翻车；评委可点按钮 · 可输入
3. 主线开发量可控（3 人 +25h 总量，每人 +8h）

## 四、Polisim 经验沉淀（可选参考）

如果你之前看过 Polisim AGENTS.md，注意 hackathon 项目与长期项目的取舍**完全不同**：

- Polisim 严守"6 层架构边界"——晌午局只有 4 层且不打算严格分层（Streamlit 路线甚至前后端不分）
- Polisim 拒绝 UI——晌午局 UI 是加分项必做或近似必做
- Polisim 全 Pydantic + ConfigDict(extra="forbid")——晌午局 Tool 输入用 Pydantic，但 Mock 数据用裸 dict 即可
- Polisim 测试覆盖 818 项——晌午局 8-15 个端到端测试足够

**核心区别**：长期项目优化"未来变更成本"；hackathon 优化"24-48h 内 Demo 完成度"。
