# 演示场景集

> 这份文档列出 Demo 现场用的 8 个开放场景。每个场景仅是「Mock 数据 tag 的不同组合 + 输入文案的不同表达」——**Tool 代码与 Agent 逻辑对场景类型完全无感**。
>
> 决策来源：`架构选型.md` D9（D 全开放底层 + A 演示形态）。
>
> 用法：见 [§五 用法](#五用法)。

## 一、设计原则

1. **8 个场景共享 1 套 Tool / 1 套 Agent / 1 套 Mock 数据**——只是 Mock 中的 `tags[]` 与 `suitable_for[]` 字段被不同输入命中
2. **Tool 代码不允许有 `if scene_type == "X"` 分支**（违反 `AGENTS.md` §4.2）
3. 每个场景至少**埋 1 个异常**（餐厅满 / 门票售罄 / 距离超限），Demo 现场可挑触发
4. 8 个场景的 Mock tag 覆盖率必须达到 `架构选型.md` D4 列出的最低数量

## 三、每个场景的详细期待

### S1. 家庭（主场景）

- **输入**：「今天下午想和老婆孩子出去玩几个小时，别离家太远，孩子 5 岁，老婆最近在减肥。」
- **期待抽取**（按 `需求分析.md` §5.7 schema）：
  ```json
  {
    "start_time": "today_afternoon",
    "duration_hours": [3, 5],
    "distance_max_km": 5,
    "companions": [{"role": "妻子", "count": 1}, {"role": "孩子", "age": 5, "count": 1}],
    "physical_constraints": ["亲子友好", "适合 5-10 岁"],
    "dietary_constraints": ["低脂", "健康轻食"],
    "experience_tags": [],
    "social_context": "家庭日常"
  }
  ```
- **期待主路径**：搜亲子 POI（≥ 4 候选） → 筛 5 岁适配 → 搜健康餐厅（≥ 6 候选）→ 查 17:00 是否有位 → 估路线
- **期待异常**：第一家健康餐厅 17:00 满 → 切到 17:30 / 备选餐厅
- **期待输出文案调性**：「下午带宝贝去 X 公园玩 2 小时，再去 Y 餐厅吃个低脂晚餐...」（家长口吻）

### S2. 朋友

- **输入**：「今天下午想和朋友出去玩几小时，4 个人 2 男 2 女，别离家太远。」
- **期待抽取**（按 `需求分析.md` §5.7 schema）：
  ```json
  {
    "start_time": "today_afternoon",
    "duration_hours": [3, 5],
    "distance_max_km": 5,
    "companions": [{"role": "朋友", "count": 4, "gender_mix": "2男2女"}],
    "physical_constraints": [],
    "dietary_constraints": [],
    "experience_tags": ["社交", "拍照友好"],
    "social_context": "朋友热闹",
    "capacity_requirement": 4
  }
  ```
- **期待主路径**：搜热闹 / 拍照 POI（密室 / 桌游 / citywalk / 网红咖啡）→ 搜 4 人桌餐厅 → 查座位 → 估路线
- **期待异常**：第一家热门密室满人 → 换 citywalk 或备选密室
- **期待输出文案调性**：「下午跟你们一起去 X 玩，记得带相机...」（朋友群聊口吻）

### S3. 情侣看展

- **输入**：「周日下午带着女朋友去看个展，顺便找个安静能聊天的地方吃饭。」

  ```json
    "distance_max_km": 5,

    "physical_constraints": [],
    "dietary_constraints": [],

    "social_context": "情侣亲密",
  ```

- **期待主路径**：搜展览 POI（须 `suitable_for: 情侣亲密`）→ 搜安静餐厅 → 查 18:00 / 18:30 位 → 估路线
- **期待异常**：热门展览门票售罄 → 换备选展览或私人画廊
- **期待输出文案调性**：「下午带你去看 X 展，看完去 Y 吃饭，环境很安静...」（情人间软语）

### S4. 带父母散步

  ```json
    "dietary_constraints": ["软烂", "适合老人"],
  ```

- **期待主路径**：搜公园 / 茶馆 / 戏曲（须 `适合老人 + 无台阶`） → 搜软烂餐厅 → 查老年友好评分 → 估**步行**为主路线
- **期待异常**：第一家茶馆当天闭店 → 换备选茶馆或江边公园
- **期待输出文案调性**：「下午陪您去 X 公园走走，那里有椅子可以歇...」（家人关切）

- **期待异常**：网红下午茶排队 60+ 分钟 → 换排队 < 20 分钟的备选
- **期待输出文案调性**：「亲爱的下午约 X，那里超出片...」（闺蜜亲昵）

- **期待异常**：第一家包间满 → 换备选高人均餐厅或调整时间
- **期待输出文案调性**：「X 总下午我安排在 Y 见面，那里环境合适...」（正式商务）

## 四、Mock 数据覆盖率自检表

8 个场景跑通的前提是 Mock 数据覆盖足够。在 Mock 数据准备完之后，必须用以下查询自检：

| 查询条件 | 期望候选数 | 用于场景 |
|---|---|---|
| `tags ⊇ {亲子友好, 适合 5-10 岁}` | ≥ 4 POI | S1 |
| `tags ⊇ {网红打卡}` AND `capacity ≥ 4` | ≥ 4 餐厅 | S2, S5 |
| `tags ⊇ {安静聊天, 亲密情侣}` | ≥ 3 餐厅 | S3 |
| `tags ⊇ {适合老人, 无台阶}` | ≥ 3 POI / ≥ 3 餐厅 | S4, S8 |
| `tags ⊇ {下午茶, 拍照友好}` | ≥ 3 餐厅 | S5 |
| `tags ⊇ {商务体面, 高人均, 有包间}` | ≥ 3 餐厅 | S6 |
| `tags ⊇ {独处舒缓}` | ≥ 3 POI / ≥ 2 餐厅 | S7 |
| `tags ⊇ {粤菜, 6 人桌}` | ≥ 2 餐厅 | S8 |
| 任意 `tags`：埋了 `available=false` 的失败案例 | ≥ 8 处（每个场景 ≥ 1） | 全部 |

如果某查询条件不满足，说明 Mock 数据需补充——这是 Mock 数据准备阶段的硬性 gate。

### 4.1 业务合理性反例自检（spec planning-quality-deep-review S9 加分项）

这一段不是 Mock 数据查询，而是「**AI 主动质疑反人性方案**」的端到端反例自检——
比 §三 的 8 个顺路场景多一类「**反例**」，演示 AI 真的在为用户考虑：

```text
| 场景 ID | 输入                                       | 期待 AI 行为                                                      |
|---------|--------------------------------------------|------------------------------------------------------------------|
| S9      | 「5 岁娃下午全天去博物馆」                   | 主活动建议 ≤ 75-90min（学龄前注意力上限），不是 2.5h               |
|         |                                            | 输出含质疑文案（"这么长宝贝可能会累" / "中途休息" / "拆短"）       |
|         |                                            | 给出拆分建议（如博物馆 75min + 公园 30min）                       |
|         |                                            | 命中 BlueprintLLM 主防（首轮 ≤ 90min）或 critic backprompt（重生成 60-75min） |
| S9.1    | 「带 78 岁奶奶下午爬山，安排 3 小时主活动」  | 主活动建议 ≤ 60min（高龄 cap）                                    |
|         |                                            | 触发 critic：「expected_range 45-60 min」                          |
|         |                                            | 文案含质疑（"奶奶腿脚 / 中途休息 / 缩短"）                         |
| S9.2    | 「商务接待客户，吃个轻食 30 分钟搞定」       | 用餐建议 ≥ 90min（商务接待 dwell prior 提示太短）                 |
|         |                                            | rationale 解释："商务接待用餐建议 90+ 分钟便于交流"                |
```

**演示路径（评委可点同 1 个按钮触发）**：
1. 评委点 "5 岁娃下午全天去博物馆" 按钮
2. 屏幕同时显示：
   - **左侧** ToolTracePanel：critic 触发 `AGE_DURATION_MISMATCH`（"建议范围 60-75 min"）
   - **中间** Itinerary：博物馆主活动 75min（不是 165min）+ 附近公园 30min 拆分
   - **右侧** Narrator 文案："5 岁宝贝注意力大约 60-75 分钟，安排 75 分钟博物馆 + 公园放放风刚好不会闹腾"
3. 评分项 1（场景理解 20%）+ 评分项 2（Tool 编排 25%）+ 评分项 5（异常韧性）三项命中

**为什么这个反例最有说服力**：S1-S8 都是「AI 一次过 + 用户满意」的顺路；S9 是「**用户提了反人性需求 → AI 主动质疑给出更合理方案**」——这是 AI 的产品价值，不是规划工具的产品价值。

## 五、用法

### 5.1 主线开发阶段（MVP-1）

只关注 S1（家庭主场景）。其余 7 个场景**留 Mock 数据 + 输入文案占位**，不投入 Tool 代码改动。

### 5.2 演示数据扩充阶段（MVP-2.2）

按本文件 Mock 覆盖率自检表逐项查 Mock，补到 ≥ 4 个场景能跑通（家庭 + 3 个跨关系，建议 S2/S3/S4 中选 3 个）。

### 5.3 加分项阶段（MVP-3）

凑齐 8 个场景全跑通，Demo 现场用：

- **快捷输入按钮**：8 个场景的输入文案做成按钮，评委一点即跑（已压力测试，零翻车）
- **输入框**：评委可即兴输入任意句子，体现开放性

### 5.4 现场预演 checklist

Demo 前必须本地连跑 8 个场景，每个场景**至少 1 次**完整跑通，记录：

- 输入抽取约束 JSON 是否与本文件期待一致（关键字段不能漏）
- Tool 调用链路是否齐全（出发 / 主活动 / 转场 / 用餐 / 返回 5 段）
- 输出方案的餐厅 / POI 标签是否与场景调性匹配（情侣场景不能选家庭餐 / 商务场景不能选街边小吃）
- 至少 1 次显式触发异常并恢复

如果 8 个场景里有 ≥ 2 个翻车，立即砍场景或扩 Mock 数据，**不要在演示当天才发现**。

## 七、persona × 同句对比演示路径（Phase 0.7 加分点）

> 评委「哇时刻」最强的一段：**同一句话切 user，方案完全不同**。
> 与 §二的 8 场景互补：8 场景考验"任意输入鲁棒性"；本节考验"同输入个性化"。

### 7.1 演示脚本（≤ 60 秒）

