# 验收标准

> 怎么证明做完了。任何"已完成"声明都必须能引用本文件的某条标准 + 给出输入 / 执行方式 / 实际输出 / 通过判定。
>
> 评委评分维度提取自 `chatgpt分析.md` §九 与 `比赛详情.md` 赛道 06 交付目标。

## 一、评分维度（评委视角）

按 `chatgpt分析.md` §九 与 `比赛详情.md` 赛道 06 提炼，**评委大概率**关注：

| 维度 | 占比估计 | 我们的应对策略 |
|---|---|---|
| 1. 场景理解准确度 | 20% | **D9 走开放意图解析**：任意自然语言 → LLM 动态抽取约束，不枚举场景类型；隐含约束在 Tool 调用中显式体现（孩子 5 岁 → 排除高强度 / 老婆减肥 → 推健康餐 / 老同学聚会 → 推安静餐厅 等） |
| 2. 规划链路完整度 | 25% | "出发 → 主活动 → 转场 → 用餐 → 附加 → 返回"6 段必齐全 |
| 3. Tool 设计合理度 | 20% | 8-10 个 Tool 按查询/执行清晰分类，JSON Schema 规范 |
| 4. Mock API 真实度 | 10% | 字段密度按 `技术架构.md` §二.4 标准；埋失败案例 |
| 5. 异常处理体现 Agent 能力 | 15% | Demo 显式触发 ≥ 2 个异常分支并恢复 |
| 6. Demo 闭环完整度 | 10% | 输入 → 规划 → 确认 → 执行 → 转发 5 步全跑通 |

**前 5 项 90% 是技术含量，第 6 项 10% 是工程闭环——但第 6 项是前 5 项的载体，不闭环前 5 项都白搭**。

## 二、内部验收清单（按 MVP 项映射）

> 每条标准给"通过判据 + 演示输入 + 期待输出"三件套。AI 完成对应工作时按此清单做证据。

### 2.1 输入解析（对应 MVP-1.1 / 需求分析 M1）

**A1. 自然语言意图解析（主场景）**

- **通过判据**：从「今天下午想和老婆孩子出去玩几个小时，别离家太远，孩子 5 岁，老婆最近在减肥」抽取出至少 6 个约束（时间 / 时长 / 距离 / 同行人 / `physical_constraints` / `dietary_constraints`）
- **演示输入**：上面那句话
- **期待输出**：JSON 形式的约束对象，字段名严格按 `需求分析.md` §5.7 schema（SoT）——至少含 `start_time`, `duration_hours`, `distance_max_km`, `companions`, `physical_constraints`, `dietary_constraints`, `social_context`
- **演示方式**：日志打印 / UI 中间过程显示
- **反向检查**：JSON 里**不能**出现 `scene_type` / `relation_type` / `is_family` 这种枚举字段（D9 硬条款）

### 2.2 Tool 实现（对应 MVP-1.2 / 需求分析 M3）

**A2. Tool 定义规范**

- **通过判据**：每个 Tool 都有 (1) JSON Schema 输入 (2) 结构化输出 (3) 失败分支
- **演示方式**：贴出 Tool 定义代码 + 至少 1 次成功调用 + 1 次失败调用的日志

**A3. Tool 数量与覆盖**

- **通过判据**：≥ 6 个 Tool（MVP-1）/ ≥ 8 个 Tool（MVP-2）；查询类 + 执行类齐全
- **演示方式**：列出 Tool 清单（名 / 类型 / 输入 / 输出 / 是否埋了失败）

### 2.3 Agent 规划（对应 MVP-1.3 / 需求分析 M2）

**A4. 多步骤规划成功**

- **通过判据**：规划输出包含「出发 / 主活动 / 转场 / 用餐 / 返回」≥ 5 段时间节点
- **演示输入**：MVP-1.1 的固定文案
- **期待输出**：含 ≥ 5 个 `(开始时间, 结束时间, 活动)` 的列表

**A5. LLM 基于 Tool 输出决策**

- **通过判据**：`search_pois` 返 8 个候选后，LLM 不是随便挑一个，而是按"5 岁孩"约束筛掉不适合的
- **演示方式**：贴 LLM 中间 prompt + 选定的 POI + 选择理由（System 1 直接选 / System 2 显式说"因为孩子 5 岁所以排除 X 选 Y"）

### 2.4 异常处理（对应 MVP-1.4 / 需求分析 M4）

**A6. 异常分支显式触发**

- **通过判据**：Demo 中可见至少 1 个异常分支被触发，且系统恢复（不报错退出）
- **演示输入**：使 Mock 数据让某餐厅 17:00 没位（提前埋）
- **期待输出**：日志显示「餐厅 X 17:00 满 → 切换到餐厅 Y / 时间 17:30」+ 最终方案不含失败的 X

**A7. 异常恢复策略合理**

- **通过判据**：恢复方案不是"瞎换"——餐厅没位换同距离同标签的备选，时段冲突优先调整时间不是直接换餐厅
- **演示方式**：贴备选餐厅与原餐厅的 tags / 距离对比

### 2.5 输出（对应 MVP-1.5）

**A8. 三段输出齐全**

- **通过判据**：(1) Tool 调用日志 (2) 行程方案 (3) 转发文案 三段都有
- **演示方式**：贴三段实际输出截图 / 文本

**A9. 转发文案口语化**

- **通过判据**：文案不含 `[1] 解析意图...` 这类技术日志格式；像家人间随口一说
- **演示方式**：贴文案 + 对照 `chatgpt分析.md` §五"转发文案"段评比

**A10. 已为你预留清单可见**

- **通过判据**：方案末尾有「已为你预留：餐厅 X 17:00 三人位（订单号 R...）」类清单
- **演示方式**：贴方案末尾截图

### 2.6 Mock 数据（对应 需求分析 S4）

**A11. POI 数量与字段密度**

- **通过判据**（D9 后分档）：
  - MVP-1：POI ≥ 20（亲子走向 ≥ 8）
  - MVP-2.2：POI 30-40，跨 ≥6 种社交上下文走向（按 `架构选型.md` D4）
  - 每个含 ≥ 8 个字段（名称 / 类型 / 距离 / 营业时间 / 评分 / `tags[]` / `suitable_for[]` / 价格）
- **演示方式**：贴 `pois.json` 头 5 条 + 跨走向字段覆盖率统计

**A12. 餐厅数量与字段密度**

- **通过判据**（D9 后分档）：
  - MVP-1：餐厅 ≥ 30（健康轻食 ≥ 12）
  - MVP-2.2：餐厅 50-60，跨 ≥6 种社交上下文走向
  - 含 `capacity` 多桌型（2/4/6/8/private_room） / `reservation_slots` / `tags[]` / `suitable_for[]`
- **演示方式**：贴 `restaurants.json` 头 5 条 + 跨走向字段覆盖率统计

**A13. 失败案例埋点**

- **通过判据**：Mock 数据中至少 2 处显式标记 `available=false` / `sold_out=true`
- **演示方式**：grep `available.*false` 或对应失败标记

### 2.7 Demo 闭环（对应 MVP-1.6）

**A14. 端到端闭环**

- **通过判据**：从用户输入到转发文案，全流程 ≤ 30 秒，无报错中断
- **演示方式**：录屏 / 现场跑

**A15. UI 中间过程可见**（如做 Web UI）

- **通过判据**：Tool 调用链路在 UI 中实时显示（流式 / 步进）
- **演示方式**：录屏看到中间步骤

### 2.8 开放意图鲁棒性（对应 D9 / MVP-2.2 / MVP-3）

**A17. 开放意图解析鲁棒性（D9 加分项）**

- **通过判据**：除主场景外再跑 ≥5 个开放场景（跨关系类型：朋友 / 情侣 / 带父母 / 闺蜜 / 商务 / 独处 中任选 5）全部跑主路径不报错，且抽取的约束与人工预期一致度 ≥ 80%
- **演示输入**：`演示场景集.md` 中任选 5 句
- **期待输出**：每句话走完整五段路径 + 输出与场景调性匹配的方案（例：情侣场景餐厅是安静不是家亲子餐；商务场景餐厅是高人均包间不是街边小吃）
- **演示方式**：录屏连跑5 个场景，每个场景 ≤ 30 秒
- **反向检查**：代码 grep 不出现 `if scene_type == "family"` / `if relation == "friends"` 这种枚举分支

**A18. 现场任意输入鲁棒压测（MVP-3 项）**

- **通过判据**：评委现场扔一句未在 `演示场景集.md` 中预留的输入，Agent 走主路径不报错（可能调用 Tool 后返空集，但应有优雅降级提示）
- **演示方式**：现场 Demo / 预设 ≥10 句未预演输入库随机抽 1 句

## 三、设计文档验收（赛题硬性交付）

`比赛详情.md` 赛道 06 要求"设计文档（≤ 2 页）"。

**A16. 设计文档**

- **通过判据**：≤ 2 页 PDF / Markdown，含 (1) Planning 策略 (2) 工具调用链路 (3) 异常处理机制 三段
- **来源**：本仓库 `docs/01-requirements/{需求分析,MVP定义}.md` 提炼即可
- **演示方式**：交付 PDF + 上交后链接

## 四、验收证据模板

任何"我已完成 Ax 项"的声明都按此格式记录在 `progress.md` 的"决策记录"或本文件附录：

```text
验收对象：
对应验收项：（引用 A1 / A2 / ... / A16）
输入：
执行方式：（CLI 命令 / Web 操作步骤 / 文件路径）
实际输出：（贴日志 / 截图链接 / 文件 head）
是否通过：是 / 否 / 部分
备注：（如部分通过，缺什么）
```

**示例**：

```text
验收对象：search_pois Tool 实现
对应验收项：A2 + A3
输入：query="亲子", distance_max_km=5
执行方式：python -m backend.tools.search_pois --query 亲子 --distance 5
实际输出：返回 12 条 POI（>= 8 满足要求），含字段 name/type/distance/age_range/tags
是否通过：是
备注：—
```

## 五、不接受的验收方式

- ❌ "我写好了，你跑一下看看"
- ❌ 不贴输入 / 输出就声称通过
- ❌ 仅靠人工读代码判断完成度
- ❌ Demo 跑通主路径就算 100%——必须显式触发异常分支（A6 + A7）
- ❌ "理论上可以"、"看起来没问题"

## 六、附：Hackathon 评委 5 分钟观察清单

如果评委只看 Demo 5 分钟，确保以下 5 件事可见：

1. **30 秒内**：主场景用户输入 → 完整方案输出
2. **能看到中间过程**：Tool 链路在屏幕上滚动 / 阶段卡片渐入 + **意图抽取后的结构化约束**在屏幕顶部可见
3. **触发 1 个异常**：餐厅没位 → 切换的过程要可见
4. **转发文案口语化**：评委能想象"这文案我能直接发给我老婆"
5. **设计文档 1 页能讲完故事**：Planning 策略 + 调用链 + 异常机制
6. **跨场景现场压测**（D9 加分哇时刻）：点一个「闺蜜 / 独处 / 商务」的快捷按钮或现场输入一句未预演的话，Agent 仍跑主路径

如果上述 6 件事都能看到，就是一等作品的量级。只能看到 1-5 是优秀作品，看不到 1-5 件是不合格。
