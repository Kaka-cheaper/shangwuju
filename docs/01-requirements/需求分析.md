# 需求分析

> 把 `比赛详情.md` 赛道 06 与 `chatgpt分析.md` 的拆解，提炼成**可执行、可验收**的条款。
>
> 这份文件是验收标准的上游——验收标准必须能追溯到本文件的某一条。

## 一、核心需求陈述

> **晌午局** 接受用户**一句自然语言模糊需求**，输出**可执行 + 可转发的下午半日（4-6 小时）行程方案**，并通过 Mock API **模拟完成关键预订/下单动作**。

关键词解释：

- **一句自然语言**：用户不需要分步操作，一次输入即可（vs 普通搜索的多次筛选）
- **可执行**：行程包含具体时间、地点、活动、预约状态（vs 推荐列表"自己挑")
- **可转发**：输出文案适合直接发给家人/朋友确认（vs 技术日志格式）
- **下午半日**：时间窗 4-6 小时，覆盖 1 个主活动 + 1 顿正餐 + 0-1 个附加活动
- **关键预订/下单**：餐厅预约 / 门票购买 / 路线规划 / 转发消息生成（最少这 4 项）

## 二、必做项（Must Have）

### M1. 自然语言意图解析（D9 开放路径）

> 根据 D9 决议、AGENTS.md §3.5，意图解析**不允许**采用「场景枚举」实现（`if scene == "family"`）。所有约束必须是 LLM 从输入中 **动态抽取** 的，不依赖预设关系类型。

系统必须能从**任意**一句话中抽取以下维度的约束（例子仅供参考，非枚举）：

| 维度 | 可能表达 | 抽取为 |
|---|---|---|
| **时间** | "今天下午" / "明天晚上" / "周日上午" | `start_time`, `start_weekday` |
| **时长** | "几个小时" / "一下午" / "半天" | `duration_hours` |
| **距离** | "别太远" / "在商圈附近" / "不要过江" | `distance_max_km` |
| **同行人** | "老婆孩子" / "4 个朋友" / "带上父母" / "女朋友" / "一个人" | `companions: [{role, age?, count?}]` |
| **隐含物理约束** | "5 岁孩" → 低强度、有儿童设施；"年迈" → 节奏慢、无台阶 | `physical_constraints` |
| **隐含饮食约束** | "减肥" → 低脂；"不吃辣" → 选菜系 | `dietary_constraints` |
| **隐含体验偏好** | "拍照好看" → 网红打卡；"安静聊天" → 排除喧闹；"生日惊喜" → 需附加服务 | `experience_tags` |
| **隐含社交场景** | "谈业务" → 安静；"老同学聚会" → 适合长时间坐 | `social_context` |
| **缺失项默认** | 用户家位置 / 预算 / 交通 → 用预设值（Demo 简化） | 调 `get_user_profile` 补 |

**鲁棒性要求**：对于以下表未枚举的表达仍必须能试图抽取：

- "三代人周日纪念日" → 老人+孩子+中青年 / 需预订 / 礼仪感
- "带着女朋友去看展" → 2 人 / 情侣场景 / 展览 + 饭
- "接个外地客户" → 商务场景 / 需体面 / 不能太随便

### M2. 多步骤行程规划

系统必须把下午时间拆成合理结构：

```text
出发 → 主活动 → 转场 → 用餐 → 附加活动（可选） → 返回
```

且规划要满足：

- M2.1 距离合理（活动 ↔ 餐厅步行/打车 ≤ 30 min）
- M2.2 时间衔接（活动结束 → 餐厅开始 之间 ≥ 转场时间）
- M2.3 餐厅有位（必须调 `check_restaurant_availability` 确认）
- M2.4 活动适合人群（5 岁孩 → 排除高强度 / 危险类目）
- M2.5 总时长在 4-6 小时内（超长须删附加活动）

### M3. Tool 调用链路

系统必须实现 **8-10 个 Tool**，覆盖以下能力：

**查询类（必有）**：

- T1 `search_pois`——按关键词 / 距离 / 标签查活动地点
- T2 `search_restaurants`——按距离 / 菜系 / 标签查餐厅
- T3 `check_restaurant_availability`——查餐厅指定时段是否有位
- T4 `estimate_route_time`——A → B 的预估通勤时间（步行 / 打车 / 公交）

**执行类（必有）**：

- T5 `reserve_restaurant`——预约餐厅
- T6 `buy_ticket`——购买活动门票
- T7 `generate_share_message`——生成可转发文案

**用户上下文（必有）**：

- T8 `get_user_profile`——获取用户画像（home 位置 / 默认预算 / 交通偏好；**不含「场景类型」字段**，场景仅从输入意图抽取）

**可选**（视时间盒决定，最多再加 2 个）：

- T9 `order_extra_service`——下单蛋糕 / 鲜花 / 饮品
- T10 `check_weather`——查天气（用于触发"下雨改室内"异常分支）

### M4. 异常处理（评分硬性项）

系统必须在 Demo 中**显式触发并恢复**至少 **2 个**异常分支。可选清单：

| 异常 | 触发方式（Mock 层） | 系统应对 |
|---|---|---|
| E1 餐厅没位 | `check_restaurant_availability` 返回 `available=false` | 切换备选餐厅 / 调整用餐时间 |
| E2 门票售罄 | `buy_ticket` 返回 `success=false, reason="sold_out"` | 替换同类型活动 |
| E3 距离超限 | `estimate_route_time` 返回 > 阈值 | 降级为附近商圈方案 |
| E4 总时长超 6h | 规划组装时检测 | 删附加活动，保留主活动 + 晚餐 |

**MVP 版**至少触发 E1 + E2 各 1 次。

### M5. 执行确认与转发

- M5.1 用户输入 → 系统输出**完整方案 + 显式的"已为你预约/购买"清单**（不只是建议）
- M5.2 系统生成**可一键复制的转发文案**（适合发给老婆/朋友确认）
- M5.3 文案风格**口语化**，不是技术日志（vs `[1] 解析意图 → [2] 查询...`——这部分作为中间过程显示，不是最终交付）

## 三、应做项（Should Have）

### S1. Web UI 而非纯 CLI

- 聊天框输入 + 行程卡片输出 + Tool 调用日志可视化
- `技术架构.md` §4 已强调"优先 Web UI 而非 CLI"——视觉冲击决定演示效果

### S2. 中间过程可见

让评委看到 Agent 的"思考过程"——Tool 调用链路 + 中间状态。
不能只输出最终结果，要凸显"Agent"而非"聊天机器人"。

### S3. 演示场景集扩充（主路径稳定后渐增，D9 加分项）

> 代替原「朋友场景」单独项。按 D9 双轨原则，意图解析代码**从一开始就开放**；Mock 数据及快捷输入按钮**依主路径稳定度渐增**。

- 按 `演示场景集.md` 准备 6-8 个跨关系类型场景（朋友 / 情侣 / 带父母 / 闺蜜 / 商务 / 独处 / 跨代际 等）
- **仅扩 Mock 数据 + 输入用例**，不动 Tool 代码 / 不写 if-else 分支
- Demo 现场提供 6-8 个快捷输入按钮 + 开放输入框并存

### S4. Mock 数据"像真的"

> D9 后分档：总量与跨走向覆盖两轴。详见 `架构选型.md` D4 、`演示场景集.md` §四 覆盖表、以及 `验收标准.md` A11/A12。

- **MVP-1 主路径（家庭场景）**：POI ≥ 20（亲子/低强度走向 ≥ 8） + 餐厅 ≥ 30（健康轻食走向 ≥ 12）
- **MVP-2.2 演示场景集**：POI 30-40 + 餐厅 50-60（跨 ≥6 种社交上下文走向）
- 字段密度：名称 / 类型 / 距离 / 营业时间 / 评分 / `tags[]` / `suitable_for[]` / `capacity` / `reservation_slots` / 库存状态
- 必埋 ≥2 个失败案例（对应 M4 的 E1 / E2）；D9 后额外要求每个 `suitable_for` 走向都 ≥1 条可预订成功 + ≥1 条满/售罄

## 四、不做项（Won't Have，第一版）

明确排除，不要"顺手帮忙"：

- ❌ 真实地图 API（高德 / 百度等）
- ❌ 真实美团接口
- ❌ 真实支付 / 真实预订
- ❌ 用户画像持久化（Mock 即用即返）
- ❌ 多用户 / 账号系统
- ❌ 跨日 / 多日行程（仅"下午半日"）
- ❌ 语音输入（仅文字）
- ❌ 个性化学习（每次都是冷启动；不记忆历史偏好）
- ❌ 第二个城市适配（默认一个 Mock 城市即可）

## 五、约束词典（跨场景通用，不枚举关系类型）

> D9 决议后，不再按「家庭场景 / 朋友场景」二分法预设约束。所有约束抽象为以下词典，由 LLM 动态从输入中抽取，再传给 Tool。

### 5.1 人物约束（`physical_constraints`）

| 起因 | 表现 | 应在 Mock 中的 tag |
|---|---|---|
| 幼儿（3-6 岁） | 低强度 / 不太晚 / 需儿童设施 | `亲子友好` `有儿童椅` `适合 5-10 岁` |
| 青少年 | 能中高强度 | `适合青少年` |
| 老人（60+） | 低强度 / 无高台阶 / 需休息点 / 节奏慢 | `适合老人` `无台阶` `可休息` |
| 孕妇 / 身障 | 平地无台阶 / 无剧烈运动 | `无障碍` |

### 5.2 饮食约束（`dietary_constraints`）

| 起因 | 排除 | 推荐 tag |
|---|---|---|
| 减肥 / 健身 | 油炸 / 高热量 | `低脂` `健康轻食` `高蛋白` `日料` `粤菜` |
| 忌口（不吃辣 / 不吃牛肉） | 达到忌口门槛的菜系 | `不辣` `无牛肉` |
| 亲子 | 需小份菜 / 儿童菜 / 子餐 | `有儿童餐` `亲子友好` |
| 商务 / 体面 | 隔间 / 安静 / 高人均 | `包间` `商务` `高人均` |

### 5.3 体验偏好（`experience_tags`）

- `拍照友好` `网红打卡` `安静聊天` `热闹` `社交` `独处身心舒缓` `商务体面` `礼仪感` `亲密情侣` `学习成长`

### 5.4 距离 / 时长（跨场景同一套）

- `distance_max_km`：默认 5，可被输入覆盖（“远一点也可以” → 10；“别过江” → 3）
- `duration_hours`：默认 4-6，可被输入覆盖（“只有 2 个小时” → 2；“一整天” → 8-10）

### 5.5 社交上下文（`social_context`）——决定餐厅/活动调性

- 家庭日常 / 老人伴助 / 闺蜜聊天 / 朋友热闹 / 情侣亲密 / 商务接待 / 同学重聚 / 独处放空 / 纪念日仪式感

### 5.6 举例：赛题原文的 2 个场景只是上表的特例化组合

| 原文场景 | 抽取后的约束集 |
|---|---|
| 家庭（5 岁孩 + 减肥老婆） | `physical: [亲子友好, 适合 5-10 岁]` + `dietary: [低脂]` + `social: 家庭日常` |
| 朋友（4 人 2 男 2 女） | `companion_count: 4` + `experience: [社交, 拍照友好]` + `social: 朋友热闹` |

任何其他输入（情侣 / 带父母 / 同事 / 独处 等）同样是上表的不同组合——**这才是 Agent，不是模板系统**。

详细场景输入文案 + 期待输出见 `演示场景集.md`。

### 5.7 意图抽取输出 schema（**全项目 source of truth**）

> ⚠️ **本节是意图解析输出格式的唯一权威定义**。其他文档（`MVP定义.md` Tool 日志示例 / `演示场景集.md` S1-S8 期待抽取 / `验收标准.md` A1）的 JSON 字段必须与此对齐。
> 任何字段增删先改本节，再 grep 同步其他文档。

```jsonc
// IntentExtraction —— 意图解析模块的唯一输出格式
{
  // ===== 时间维度（必有，缺则 LLM 用默认值"今天下午"）=====
  "start_time": "string, ISO-like 形如 2026-05-09T14:00 或 today_afternoon | tomorrow_evening",
  "start_weekday": "string?, 可选, weekday 标签如 saturday / sunday（若用户用『周日』则填，否则可空）",
  "duration_hours": [3, 5],     // [min, max]，默认 [4, 6]

  // ===== 空间维度（必有，缺则用 get_user_profile 兜底）=====
  "distance_max_km": 5,         // 数字，默认 5

  // ===== 同行人结构（必有）=====
  "companions": [
    {
      "role": "string",         // 自由文本：妻子 / 孩子 / 朋友 / 女朋友 / 外公 / 商务客户 / 闺蜜 / ...
      "age": 5,                 // optional, 整数岁数
      "count": 1,               // optional, 默认 1
      "gender_mix": "string?",  // optional, 仅多人时填，如 "2男2女"
      "is_birthday": false,     // optional, 是否当事人生日
      "is_special_role": false  // optional, 商务客户 / 长辈 等需特别尊重场合
    }
  ],

  // ===== 三类约束 tag（必有，可空数组）=====
  "physical_constraints": ["亲子友好", "适合 5-10 岁"],   // 来自 §5.1 词典，仅取预定 tag
  "dietary_constraints":  ["低脂", "健康轻食"],            // 来自 §5.2 词典
  "experience_tags":      ["拍照友好"],                    // 来自 §5.3 词典

  // ===== 社交上下文（必有单值，从 §5.5 9 选 1）=====
  "social_context": "家庭日常",  // enum: 家庭日常 / 老人伴助 / 闺蜜聊天 / 朋友热闹 / 情侣亲密 / 商务接待 / 同学重聚 / 独处放空 / 纪念日仪式感

  // ===== 容量与额外服务（可选）=====
  "capacity_requirement": 4,           // optional, 同行 ≥ 4 人时填
  "extra_services": ["蛋糕"],          // optional, 仪式场合需附加服务
  "preferred_poi_types": ["展览"],     // optional, 用户明示 POI 类型

  // ===== 元数据（必有）=====
  "raw_input": "原始用户输入字符串",
  "parse_confidence": 0.85,    // 0-1, LLM 对自身抽取的信心；< 0.6 时 Agent 应 ask back
  "ambiguous_fields": []       // 字符串数组，LLM 自报"哪些字段我不确定"
}
```

**字段约束硬条款**：

- ❌ **禁止**出现 `scene_type` / `relation_type` / `is_family` / `is_friends` 这种枚举型顶层字段（违反 D9）
- ✅ `social_context` 是 enum 但**不是场景枚举**——它是从约束反推的氛围标签，可以是任意输入命中的 9 选 1
- ✅ `physical_constraints` / `dietary_constraints` / `experience_tags` 仅接受 §5.1/5.2/5.3 词典里的 tag，LLM 不可发明（详见 `pitfalls.md` P1-预埋 词典出口防御）
- ✅ `companions[].role` 是**自由文本**而非枚举（这是 D9 开放性的体现）

**消费方契约**：

- `MVP定义.md` MVP-1.5 输出的"Tool 调用日志"行 `[1] 解析意图：...` 内容由本 schema 字段拼接
- `演示场景集.md` S1-S8 每个场景的"期待抽取"段对应本 schema 的具体值
- `验收标准.md` A1 的"通过判据"按本 schema 字段名验证

## 六、用户故事（评委演示参考）

### 6.1 主场景故事（家庭、题目原文）

> 用户：「今天下午想和老婆孩子出去玩几个小时，别离家太远，孩子 5 岁，老婆最近在减肥。」

期待系统行为（5 步可见）：

1. **解析约束**（屏幕显示）："3 人 / 时长 4-6h / 距离 ≤ 5km / physical: [亲子友好] / dietary: [低脂]"
2. **Tool 调用链路**（屏幕显示）：搜亲子 POI → 筛 5 岁适配 → 搜健康餐厅 → 查 17:00 是否有位 → 估路线时间
3. **触发异常**（演示亮点）：第一家餐厅 17:00 满 → 切换到 17:30 / 备选餐厅
4. **输出方案**：时间轴卡片 + "已为你预留"清单
5. **转发文案**：可复制的口语文案

详细文案见 `chatgpt分析.md` §五"系统最终输出" / "转发文案"段。

### 6.2 开放场景故事（评委现场即兴输入，鲁棒性压力测）

以下 3 句话都不在题目原文中，但 Agent 必须能处理（走主路径 + 可能触发异常）：

- 「周日下午带着女朋友去看个展，顺便找个安静能聊天的地方吃饭。」
  - 期待抽取：`companions: [女朋友 ×1]` + `social_context: 情侣亲密` + `experience_tags: [看展, 安静聊天]`
- 「周日下午想带外公外婆出去走走，别走太远他们腿不好。」
  - 期待抽取：`companions: [老人 ×2]` + `physical: [适合老人, 无台阶]` + `distance_max_km: 3`
- 「下午临时被叫去接个外地客户，对方是商务人士，帮我安排下。」
  - 期待抽取：`companions: [商务 ×1]` + `social_context: 商务接待` + `dietary: [高人均, 包间]`

详细 6-8 个演示场景及需的 Mock tag 见 `演示场景集.md`。
