# 小团 App 能力接入指南

> 这份文档是写给 **小团团队（命题方）** 的：晌午局如何作为「半日规划」技能，接入小团 App 主页 5 tab 中央入口。
>
> 阅读对象：小团 App 客户端工程师 / 后端 / 评委
> 配套文档：`docs/06-business/01-数据源切换路径.md`（切高德/大众点评的工作量评估）

---

## 一、定位：晌午局 = 小团 App 的第 5 个能力维度

小团 App 当前主页底部已有 4 个单点场景：

```text
| 已有场景    | 能力定位          | 调用工具                       |
|-----------|------------------|------------------------------|
| 自助餐     | 单点找店 + 领券   | 「找优惠」/「一键领券」        |
| KTV       | 单点找店 + 领券   | 「找优惠」/「立即购买」         |
| 电竞酒店   | 单点找店 + 预订   | 「确认订单」                   |
| 夜宵       | 单点找店 + 领券   | 「找优惠」                     |
```

它们都是「**单店 + 一键执行**」的执行类能力——用户已经知道想去哪类店，差临门一脚。

**晌午局补上「半日规划」这一面状能力**：

```text
| 输入                              | 小团已有                  | 晌午局补上                                     |
|----------------------------------|--------------------------|---------------------------------------------|
| "下午带娃出去玩"                  | 用户自己翻列表 → 点 KTV   | 一句话 → 出发/主活/用餐/返程串起来 4-6 段时间轴 |
| "5 岁孩 + 老婆减肥 + 距家 5km"     | 各自独立筛选漏斗          | 一次性吸收所有约束 + 多轮反馈重规划             |
| "餐厅 17:00 满了"                | 用户重新搜               | 自动换 17:30 / 备选店（异常自愈）              |
```

一句话定位：

> **晌午局 = 小团 App 的「半日规划」技能扩展。**
> 复用小团已有 Tool（「找优惠」对应 search_pois、「一键领券」对应 buy_ticket、「确认订单」对应 reserve_restaurant），多出来的是「一句话 → 多约束推理 → 串起来的行程 → 异常自愈」这条链路。

## 二、集成形态：3 个 HTTP API + 1 个 SSE 事件流

小团 App 客户端只需对接 3 个 HTTP 接口：

```text
| 接口                | 调用时机              | 用途                                          |
|---------------------|----------------------|---------------------------------------------|
| POST /chat/turn     | 用户在输入框发送一句话 | 主入口：自动识别新需求 vs 反馈，跨 turn 上下文持久 |
| POST /chat/confirm  | 用户点「确认预约」    | 派发预约 + 购票 + 转发文案 3 个执行类工具        |
| GET  /scenarios     | 主页加载              | 拉 8 个演示场景文案，渲染主页快捷按钮            |
```

**SSE 事件流**：所有 `/chat/*` 端点都返回 `text/event-stream`，事件类型见 `backend/api_contract.md` §3。

**评委可现场访问**：`http://localhost:8000/docs` 直接看 Swagger UI 自动生成的 API 文档（含「小团接入」tag 一组 4 端点详细签名）。

## 三、API 详细签名

### 3.1 POST /chat/turn —— 对话主入口（推荐）

**为什么选这个而不是 `/chat/stream`**：本端点会从 `ConversationStore` 取当前 session 状态，自动判断 `message` 是「新需求」还是「对已有方案的反馈」。小团 App 不必维护"现在该调哪个端点"的状态机。

**Request Body**：

```json
{
  "message": "今天下午陪老婆孩子出去玩，孩子 5 岁，老婆减肥",
  "session_id": "sess_abc123",
  "scenario_id": "S3",
  "user_id": "ut_user_xxx"
}
```

```text
| 字段          | 类型           | 必填  | 说明                                              |
|---------------|---------------|-------|-------------------------------------------------|
| message       | string ≤ 500  | 是    | 用户输入的一句话                                   |
| session_id    | string ≤ 128  | 是    | 会话 ID，跨 turn 复用同一个；新会话用 uuid4         |
| scenario_id   | string        | 否    | 演示场景 ID（S1-S8），点击快捷按钮时透传            |
| user_id       | string ≤ 64   | 否    | 用户 ID，缺省时按 `X-User-Id` header > "demo_user" |
```

**Response Headers**：

- `Content-Type: text/event-stream`
- `X-Planner-Mode: rule | llm`（解析自请求 header `X-Planner-Mode` > env > 默认 rule）
- `X-User-Id: <resolved>`

**SSE 事件序列（典型成功路径）**：

```text
intent_parsed       → 用户约束抽取完成
agent_thought       → "我先查一下亲子场所..."
tool_call_start     → search_pois（fan-out worker 1）
tool_call_start     → search_restaurants（fan-out worker 2）
tool_call_start     → get_user_profile（fan-out worker 3）
tool_call_end × 3   → 三路并发结果汇合
agent_thought       → "找到 5 个亲子场所..."
tool_call_start/end → check_availability（17:00）
replan_triggered    → "17:00 满 → 切 17:30"（异常自愈）
tool_call_start/end → check_availability（17:30）
tool_call_start/end → estimate_route_time × 3
itinerary_ready     → 完整六段行程 + agent_narration 文案
done
```

**curl 样例**：

```bash
curl -N -X POST http://localhost:8000/chat/turn \
  -H "Content-Type: application/json" \
  -H "X-Planner-Mode: rule" \
  -H "X-User-Id: ut_demo" \
  -d '{
    "message": "今天下午陪老婆孩子出去玩，孩子 5 岁，老婆减肥",
    "session_id": "sess_001"
  }'
```

### 3.2 POST /chat/confirm —— 确认下单

用户在小团 App 看到行程卡片后点「确认预约」按钮，App 把卡片中的合法 ID 集合回传给后端，后端按白名单派发执行类工具。

**Request Body**：

```json
{
  "session_id": "sess_abc123",
  "decision": "confirm",
  "user_id": "ut_user_xxx",
  "allowed_restaurant_ids": ["R001", "R005"],
  "allowed_poi_ids": ["P003", "P012"]
}
```

```text
| 字段                     | 类型         | 必填  | 说明                                  |
|-------------------------|-------------|------|-------------------------------------|
| session_id              | string      | 是    | 与 /chat/turn 同一个 session_id        |
| decision                | enum        | 是    | confirm / reject / modify             |
| user_id                 | string      | 否    | 同 /chat/turn                          |
| allowed_restaurant_ids  | list[str]   | 否    | 防 hallucination 白名单：reserve 仅能在此集合内派发 |
| allowed_poi_ids         | list[str]   | 否    | 同上：buy_ticket 仅能在此集合内派发     |
| modifications           | object      | 否    | reject/modify 时的额外说明              |
```

**SSE 事件序列**：

```text
reserve_restaurant       → 餐厅预约下单（mock 返成功 + 订单号）
buy_ticket（如有 POI 票）→ 门票购买
generate_share_message   → 生成转发文案（mock LLM 模板化）
itinerary_ready          → 含 orders + share_message 的最终行程
memory_persisted         → 用户偏好画像跨 session 持久化
done
```

### 3.3 POST /chat/refine —— 独立反馈（兼容老接入）

**用法**：用户在卡片内说「太远了，希望 3 公里以内」时，小团 App 调本端点。**新接入推荐用 `/chat/turn`**——本端点保留是为了让分阶段集成的客户端有渐进路径。

**Request Body**（schema 见 `backend/schemas/refine.py:RefinementInput`）：

```json
{
  "session_id": "sess_abc123",
  "feedback_text": "太远了，希望 3 公里以内"
}
```

### 3.4 GET /scenarios —— 拉演示场景文案

主页加载时调一次，拿到 8 个快捷按钮的 `id / title / input / icon`。

**Response**：

```json
{
  "scenarios": [
    {
      "id": "S1",
      "title": "学生党 KTV 局",
      "input": "周五晚上和室友 4 个人想去 K 歌，预算别太贵",
      "icon": "🎤"
    },
    {
      "id": "S2",
      "title": "兄弟撸串夜宵",
      "input": "今晚和兄弟出来撸串喝点酒，人均 50 左右就行",
      "icon": "🍢"
    }
  ]
}
```

## 四、部署形态：两条路径

### 形态 A（推荐 · 中长期）：晌午局服务部署到小团内网

```text
小团 App 客户端
       ▼
小团 App 网关
       ▼ （内网转发上述 4 个端点）
晌午局服务（FastAPI + Redis + LLM 配置）
       ▼
小团已有 Tool 服务（找优惠 / 一键领券 / 预订 / ...）
```

**优势**：

- 小团 App 用户感知不到外部跳转
- 共享小团已有的鉴权、风控、监控基建
- 切高德 / 大众点评后台数据时不影响 App

**对接工作量**：

- 容器镜像：`backend/Dockerfile` 已就绪（multi-stage / FC Custom Container 兼容）
- 健康探针：`/health`（liveness）+ `/ready`（readiness，含 LLM/Redis/mock_data 三层）
- 部署文档：`docs/06-business/02-阿里云FC部署.md`
- env 配置：`backend/.env.example` 列出全部 5-7 个必备 env

### 形态 B（短期 · 联调）：本地 docker compose 起 demo

```text
git clone <repo>
cd 美团AI\ Hackathon
docker compose up --build
访问 http://localhost:3000  # 前端 demo
访问 http://localhost:8000/docs  # API Swagger UI
```

**用法**：

- 评委两分钟跑起来，亲手点 8 个场景按钮 + 输入框即兴扔输入
- 小团 App 工程师把 base_url 切到 `http://<demo-host>:8000` 联调

```text

|--------------------|------------------------|------------------------------------|
```

剩余 5 个 Tool 是「半日规划」独有的能力：


## 七、安全与防护

```text
| 风险                       | 防护                                                 |
|---------------------------|----------------------------------------------------|
| LLM 编造餐厅 ID            | confirm 端点 allowed_restaurant_ids / poi_ids 白名单 |
| LLM 抽取出 D9 禁止字段     | schemas/intent.py 用 ConfigDict(extra="forbid")    |
| 词典外 tag                 | router/critic 双层兜底，自动降级到词典内最近邻        |
| 异常导致 SSE 流挂死         | _safe_stream wrapper 兜底 → 推 stream_error + done  |
| 跨用户偏好串号             | user_id 解析三层兜底（body > header > demo_user）   |
| 多人协作下单数据竞争        | 仅 owner 可触发 confirm；其他成员只能投票/反馈        |
```

```text
1. [ ] 起本地服务：docker compose up
```

## 九、FAQ

**Q1：评委两分钟内能看到什么？**

A：访问 `http://localhost:8000/docs`，左侧「小团接入」tag 下展开 4 个端点，每个端点有 summary + 完整 schema + curl 样例。
访问 `http://localhost:3000`，主页 8 个快捷按钮（首两位是青年向 KTV / 烧烤夜宵——小团 App 主力用户群）+ 输入框，点任一按钮即可看到 SSE 事件流 + ToolTracePanel + 行程卡片。

**Q2：和现有「找优惠」「一键领券」会冲突吗？**

A：不冲突。晌午局调用的执行类工具（reserve_restaurant / buy_ticket）就是包装小团已有的预订/领券能力，
切真后台时映射到小团内部服务即可，不引入新交易通道。

**Q3：上线工作量怎么估？**

A：见 [`01-数据源切换路径.md`](./01-数据源切换路径.md)。三层抽象都在 demo 阶段就位，切换属于「构造函数注入」级改造，工作量是中等量级而非重写。

**Q4：跨用户偏好画像数据放哪？**

A：当前 demo 阶段写到 `mock_data/user_profile.json`（单进程文件）；MVP 阶段切 Redis（`SESSION_STORE=redis`），
真产品阶段切 PostgreSQL + Redis。详见 [`02-持久化演进.md`](./02-持久化演进.md)。

**Q5：小团 App 现有 4 场景之外的需求，需要扩 mock 数据吗？**

A：不需要扩 demo mock。`backend/agent/v2/tool_provider.py` 已暴露 `ToolProvider` 抽象，
切真后台时换实现，业务代码零改动。
