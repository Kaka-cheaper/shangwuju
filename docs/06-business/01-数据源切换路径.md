# 数据源切换路径

> 从 Demo 的 mock JSON 到真产品的商家直签数据，分三阶段、不重写业务代码。

---

## 一、为什么需要切换路径

题目硬性要求 **不接真实美团 / 高德 / 支付 API**（赛题原文）。所以 Demo 阶段我们用 `mock_data/` 静态 JSON 跑通完整闭环。

但评委会问：「比赛完之后呢？这套架构是死路一条吗？」

**回答**：不是。我们在 Tool 层下方留了一道抽象，叫 `ToolProvider`——业务代码（Tool 实现）只面对抽象接口，底层换数据源不用改业务代码。

```text
┌────────────────────────────────────┐
│  Tool 层（不变）                    │
│  search_pois / search_restaurants  │
│  / check_availability / route ...  │
└──────────┬─────────────────────────┘
           │ 业务代码只调抽象方法
           ▼
┌────────────────────────────────────┐
│  ToolProvider 抽象（一道接口）       │
│  query_pois(filters) → POIList     │
│  query_restaurants(filters) → ...  │
└──┬─────────┬──────────┬────────────┘
   │         │          │
   ▼         ▼          ▼
mock JSON  高德 API   商家直签 + 大众点评
（Demo）   （MVP）    （真产品）
```

> 该抽象层由 `backend/agent/v2/tool_provider.py` 提供（已就绪：`ToolProvider` 抽象 + `MockToolProvider` 默认实现 + `GaodeToolProviderStub` / `DianpingToolProviderStub` 接入位）。Demo 阶段直接跑 `MockToolProvider`；MVP 切 `GaodeToolProviderStub` 落地实现；真产品切 `DianpingToolProviderStub` + 直签数据源。

> **附近搜索 NearbySearchProvider（Phase 0.21 新增，2026-05-22）**：在 `ToolProvider` 之外，单独抽出「以用户位置为中心、按 max_km 半径找附近 POI/餐厅」的语义。原因：高德 `v3/place/around` 与美团 POI 接口签名不同，需要独立映射层。位置：`backend/data/nearby_provider.py`（`MockNearbyProvider` 用 haversine 实时算距离 + `GaodeNearbyProvider` / `MeituanNearbyProvider` stub）。env 切换：`NEARBY_PROVIDER=mock|gaode|meituan`。
> 
> **一键部署叙事**：评委 docker compose up → `NEARBY_PROVIDER=mock` 直接演示；切 `NEARBY_PROVIDER=gaode` + 配置 `AMAP_REST_KEY` 立刻接入真附近搜索（接入步骤见本文 §四 章节）。

## 二、当前状态（Demo · 2026-05）

### 2.1 数据规模

```text
| 类型     | 数量 | 字段密度                                     |
|----------|------|---------------------------------------------|
| POI      | 39   | 名称 / 标签 / 距离 / 客群 / 价位 / 失败埋点    |
| 餐厅     | 45   | 菜系 / 人均 / 容量 / 营业时段 / 失败埋点       |
| 路线     | 56+  | 起终点对 × 出行方式（步行/打车/公交）         |
| persona  | 5    | mock 用户画像（user 维度）                    |
```

### 2.2 关键约束（已在 mock 里覆盖）

- 9+ 处失败埋点（餐厅满 / 门票售罄 / 候选 0）
- 8 种 social_context 走向（家庭 / 朋友 / 情侣 / 父母 / 闺蜜 / 商务 / 独处 / 跨代际）
- D9 调性匹配：mock 数据本身不打 `scene_type` 枚举标签，靠 tag 词典（`physical_constraints` / `dietary_constraints` / `experience_tags` / `suitable_for`）描述

### 2.3 切换准备度

`backend/tools/` 下的 8 个 Tool 实现都已以 **过滤器函数式** 写法对接数据，没有硬编码 mock 文件路径——更换 ToolProvider 实现是一次「构造函数注入」改动。

## 三、MVP 阶段（1-2 月）：高德 Web Service API

### 3.1 为什么先选高德

```text
| 候选        | 月免费额度          | POI 数据完整度  | 接入难度 |
|-------------|--------------------|----------------|----------|
| 高德 Web    | 30 万次（个人开发者）| 全国覆盖、星级齐 | 低（HTTP）|
| 大众点评开放 | 需商务谈判          | 评分 / 评价完整 | 高       |
| 美团开放    | 仅商家入口          | 商品 / 团购详细 | 高（合规）|
```

> 数据来源：[高德开放平台 · Web 服务 API · 配额说明](https://lbs.amap.com/api/webservice/quota)。免费额度数字以官方实时为准，本文 2026-05 数据点供规划参考。

高德 Web Service API 是个人开发者注册即用，30 万次/月足够 MVP 验证 PMF（按每用户 5 次 POI 查询估算可支持 6 万 DAU 内）。

### 3.2 申请流程

```text
Step 1  [https://lbs.amap.com](https://lbs.amap.com) 注册账号
Step 2  控制台 → 应用管理 → 创建新应用 → 添加 key（类型选 Web 服务）
Step 3  实名认证（个人 / 企业，企业额度更高）
Step 4  pip install amap-py 或直接 httpx 调 https://restapi.amap.com/v3/...
Step 5  本地写 .env 注入 AMAP_KEY
```

### 3.3 schema 映射

高德 POI 输出 → 我们的 schema（D-SoT §5.7）：

```text
| 高德字段              | 我们的字段                | 映射规则                   |
|----------------------|--------------------------|---------------------------|
| poi.id               | poi_id                   | 直接拷贝                   |
| poi.name             | name                     | 直接拷贝                   |
| poi.location         | lat / lng                | 拆 "lng,lat" 字符串        |
| poi.distance         | distance_km              | 米转公里 / 1000           |
| poi.type             | tags / suitable_for      | 字典映射（详见下表）        |
| poi.business[*]      | rating / pricing         | 部分字段                   |
| poi.photos           | image_url                | 取首张                     |
```

类型字典映射示例（高德 type → 我们的 tag）：

```text
| 高德 type            | suitable_for       | experience_tags |
|---------------------|--------------------|-----------------|
| 餐饮服务;中餐厅      | 家庭日常 / 朋友聚会 | 中餐 / 家常       |
| 体育休闲服务;公园广场 | 家庭日常 / 独处放空 | 户外 / 拍照友好   |
| 科教文化服务;博物馆   | 情侣亲密 / 独处放空 | 安静 / 文艺       |
```

完整字典在切换实施时维护在 `backend/data/amap_type_mapping.json`（待 Phase 0.11）。

### 3.4 频率限制策略

```text
- L1 LRU 缓存：城市级 POI 查询缓存 1 小时（Redis），命中率预估 60%+
- L2 流控：单用户 IP 每分钟 ≤ 5 次（防爬虫被高德封 key）
- L3 降级：触发限流 → 回退 mock 数据 + agent_thought 提示「数据源热备中」
- L4 多 key 池：注册 3 个开发者账号轮换（合规边界，单账号月度 30 万次）
```

### 3.5 切换工作量估算

```text
| 工作                                  | 人时        |
|---------------------------------------|-------------|
| 写 AmapToolProvider 实现（5 接口）     | 4 h         |
| schema 映射字典 + 单测                 | 3 h         |
| Redis 缓存 + 流控装饰器                | 4 h         |
| 灰度切换开关（env: TOOL_PROVIDER）      | 1 h         |
| 端到端回归（沿用现有 8 场景测试）        | 2 h         |
| 总计                                  | **14 h**    |
```

## 四、真产品阶段（3-6 月）：商家直签 + 大众点评开放 API

### 4.1 为什么不只用高德

高德有「点」（POI）但没「面」（评价 / 实时容量 / 可预约时段）。真产品想做「对话即下单」，必须能拿到：

```text
- 餐厅实时空位（用户对话期间不能存在「订上后台才发现满了」）
- 大众点评级评价数据（用户决策需要「我朋友会不会觉得我品位差」）
- 商家直接的优惠 / 套餐数据（差异化定位 §05 关键武器）
```

### 4.2 双数据源混合策略

```text
| 数据维度       | 来源                      | 备注                    |
|---------------|--------------------------|-----------------------|
| POI 基础数据   | 高德 Web Service         | 全国覆盖兜底           |
| 餐厅评分 / 评价 | 大众点评开放 API          | 头部品类优先           |
| 实时容量       | 直签商家 webhook         | 我方主动维护           |
| 路线规划       | 高德 Web Service         | 不动                   |
| 套餐 / 优惠    | 直签商家管理后台          | 商业模式 §04 抽佣抓手   |
```

### 4.3 商务谈判要点

向商家推销「晌午局接入」时的话术框架：

```text
价值点 1  「我们带的是有意图的用户，不是随便看看」
          → 平均预算 / 客单价高于行业基线（待 PMF 数据填充）
价值点 2  「Agent 优先推荐有套餐 / 有优惠的商家」
          → 等价于 SEO，但比关键词搜索精准度高
价值点 3  「失败案例（满 / 售罄）我们会帮你引流到同档备选」
          → 不是流量黑洞，是流量调度
合规边界  「我们不替商家承诺履约，下单后责任在商家系统」
```

### 4.4 数据合规架构

```text
- 用户 PII：仅存 user_id / 偏好画像 / 消费历史；不存身份证 / 银行卡
- 商家数据：直签合同明确「评晋数据使用范围 = AI 推荐展示」；不二次倒卖
- LLM 输入：用户原话发给 LLM 前过滤手机号 / 地址（regex + 关键词词典）
- 日志保留：用户对话日志保留 30 天后脱敏归档（仅留 intent + tag，不留原文）
- 数据出境：所有 LLM 选型以国内 base_url 为主（DeepSeek / 通义 / 智谱）
```

详见 [`02-持久化演进.md`](./02-持久化演进.md) 数据合规章节。

## 五、切换里程碑

```text
| 时间        | 数据源                               | 验收标准                        |
|-------------|--------------------------------------|--------------------------------|
| 现在（Demo）| mock_data/ JSON                      | 8 场景跑通 + 异常显式触发        |
| +1 月（α）  | AmapToolProvider                     | 8 场景沿用相同测试全过           |
| +3 月（β）  | Amap + 大众点评开放 API（只读）       | 评分数据真实 + 50 商家直签       |
| +6 月（v1）| Amap + 直签 webhook + 大众点评       | 实时容量真实 + 头部 5 商圈覆盖   |
```

**切换成本可控的理由**：8 场景的端到端测试（`backend/tests/test_8_scenarios.py`）走的是 ToolProvider 抽象接口，不绑定任何具体数据源——MVP 切 Amap 时这套测试直接复用，回归成本极低。

## 六、风险与规避

```text
| 风险                          | 规避策略                                  |
|------------------------------|------------------------------------------|
| 高德 API 改字段              | schema 映射函数集中维护，字段漂移单测捕捉   |
| 高德额度耗尽                  | 多 key 池 + L1 缓存 + 降级 mock              |
| 大众点评接入门槛高            | β 阶段先只读评分，不阻塞 α 上线              |
| 商家不愿直签                  | 优先头部 5 商圈头部品类，10 家起步         |
| 数据合规审查                  | 上线前过等保二级 + 法务 review            |
```

详见 [`03-观测性骨架.md`](./03-观测性骨架.md) 错误率监控章节。
