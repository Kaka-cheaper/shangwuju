## 一、为什么选 FC（阿里云函数计算）

```text
┌─ 备选方案对比 ──────────────────────────────────────────────────┐
│ 方案              | 启动速度 | 成本    | 弹性   | 运维成本      │
│ FC Custom Container | 3-5s 冷启 | 按调用 | 自动   | 0（无服务器） │
│ ECS + Docker      | 永远在线 | 月固定  | 手动   | 高（监控/扩容）│
│ K8s ACK           | 永远在线 | 节点贵  | 自动   | 中-高         │
│ Serverless k8s    | 类似 FC  | 中     | 自动   | 中             │
└──────────────────────────────────────────────────────────────────┘
```

我们选 FC Custom Container 的核心理由：
- **零运维**：单租户 SaaS 早期 QPS 不稳定，按调用计费比月费便宜 5-10x
- **自动横向扩展**：一句 LLM 阻塞不会影响其它请求
- **冷启动可控**：配置预留实例 = 1 即可消除冷启动；按需付费

## 三、FC 函数创建（首次部署）

### 3.1 控制台创建（推荐第一次走这条）

```text
1. 进 fc.console.aliyun.com → 创建函数 → 选「自定义运行时」（Custom Container）
2. 函数名：shangwujue-backend
3. 镜像：选 ACR → 选刚推的 :0.1.0
4. 实例规格：1C 1G（demo 阶段够用；生产建议 1C 2G）
5. 端口：9000（FC 默认；与我们 Dockerfile 的 EXPOSE 9000 对齐）
6. 启动命令：默认（用 Dockerfile 的 CMD）
7. 内存超时：60s（SSE 流式规划最长可能 35s，留 buffer）
8. 触发器：HTTP 触发 → 鉴权方式选「无鉴权」（demo）/「Function 鉴权」（prod）
9. 高级配置 → 预留实例：1（消除冷启动；按月计费）
```

### 3.2 环境变量配置

进函数详情 → 配置 → 环境变量，按下面填：

```text
LLM_PROVIDER=deepseek
DEEPSEEK_API_KEY=sk-...
DEEPSEEK_BASE_URL=https://api.deepseek.com
USE_LANGGRAPH=1

# 持久化：必须切到 Redis（FC 单实例 InMemory 会丢数据）
SESSION_STORE=redis
REDIS_URL=redis://:password@r-bp1xxx.redis.rds.aliyuncs.com:6379/0

# 数据源：比赛阶段保持 mock；上线时切到 gaode
DATA_PROVIDER=mock
NEARBY_PROVIDER=mock

# 可观测：配 Logfire token 自动上报
LOGFIRE_TOKEN=lf_...
LOGFIRE_ENVIRONMENT=production

# 日志：JSON 格式便于 SLS 自动解析
LOG_FORMAT=json

# 高德 jscode（前端代理用）
AMAP_REST_KEY=
AMAP_JS_CODE=
```

### 3.3 健康检查配置

进函数 → 触发器配置 → 实例健康检查：

```text
路径：/ready
端口：9000
启动延迟：30s（避免冷启动期间被打死）
间隔：30s
失败次数：3
```

我们的 `/ready` 端点会探活 LLM 配置 / Redis 连接 / mock 数据加载，**任何一项失败返 503**——FC 健康检查会自动剔除问题实例。

## 五、前端部署（多选一）

```text
方案                  | 优势                 | 劣势                  | 推荐度
Vercel                | 零配置 / 全球 CDN     | 国内访问偶尔慢         | ★★★★★（出海）
阿里云 OSS + CDN      | 国内速度快            | 需手动 build 上传      | ★★★★★（国内）
Cloudflare Pages      | 全球 CDN / 免费       | 国内访问不稳           | ★★★
FC Custom Container   | 与后端同套部署        | Next.js SSR 冷启动慢   | ★★
```

**我们的推荐**：国内运营 → OSS + CDN；海外或测试 → Vercel。

### 5.1 OSS + CDN 部署（国内主推）

```bash
# 1. next.config.mjs 已设 output: "standalone"；改加 output: "export" 走纯静态（如果只用 SSG）
#    保留 standalone 时可用 oss-deploy 工具静态部分 + 动态部分分别处理
# 2. 构建：
NEXT_PUBLIC_API_BASE=https://api.shangwujue.com pnpm build
# 3. 上传 .next/static + public 到 OSS bucket（用 ossutil / sdk）
# 4. CDN 配置回源到 OSS
```

## 六、上线核查清单

```text
□ FC 函数 + 健康检查 OK：curl https://fc-domain/ready 返 200
□ Redis 连接 OK：从 /ready 看到 redis: ok
□ LLM 调用通：POST /chat/turn 跑完整规划，5-10s 内出方案
□ Logfire 看板看到 trace：每次规划完整链路
□ CORS：前端域名加白名单（CORSMiddleware allow_origins）
□ HTTPS：FC 默认提供 *.fc.aliyuncs.com 证书；自定义域名需配 ACM 证书
□ 域名备案：ICP 备案（首次申请 ~20 工作日）
□ 隐私协议 / 用户协议：律师审核签字
□ 个人信息保护影响评估（PIA）：法务出具报告
□ 监控告警：Logfire 配 LLM 失败率 > 5% 邮件告警
```

## 七、运维常见问题

### 7.1 冷启动慢？

```text
现象：首次调用 /chat/turn 5-8s 才有响应
缓解：
1. 配预留实例 = 1（彻底消除）
2. import 重的模块（langgraph / pydantic_ai）放函数级懒导入
3. /ready 提前预热（健康检查会触发实例预热）
```

### 7.2 SSE 在 FC 上断流？

```text
现象：评委演示中 SSE 偶尔断开
排查：
1. FC 内存超时设 60s（SSE 主链路最长 35s 内必须出第一个 event）
2. 我们的 sse_adapter 已经在 8s 内推第一个心跳事件
3. nginx / 阿里云前端 LB 不影响（FC 自带）
```

### 7.3 Redis 连接耗尽？

```text
现象：/ready redis: ok 但 /chat/turn 偶尔抛 ConnectionError
排查：
1. RedisRepository 已设 max_connections=10（够单实例并发）
2. 多实例时按 FC 函数实例数 × 10 估算 Redis 连接数
3. Redis Tair 需要选「连接数 ≥ 100」的规格
```
