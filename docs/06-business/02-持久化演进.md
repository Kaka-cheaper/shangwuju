# 持久化演进

> 从 Demo 的进程内 dict 到真产品的 PostgreSQL，三阶段切换不动业务代码。

---

## 一、当前状态：InMemoryRepository

Demo 阶段所有跨 turn 状态（conversation / memory）存在 **单进程 Python dict** 里。

```text
┌────────────────────────────────────┐
│  Agent 编排层（业务代码不变）        │
│  router → intent → planner →       │
│  refiner → executor → memory       │
└─────────┬──────────────────────────┘
          │ 业务代码只调抽象方法
          ▼
┌────────────────────────────────────┐
│  ConversationRepository 抽象       │
│  get_session / save_message /      │
│  get_memory / update_memory ...    │
└──┬─────────┬──────────┬────────────┘
   │         │          │
   ▼         ▼          ▼
InMemory   Redis     PostgreSQL
（Demo）   （MVP）   （真产品）
```

> 该抽象层由 `backend/agent/v2/conversation.py` 提供（已就绪）。Demo 阶段使用 `InMemoryRepository` 实现；MVP 切 `RedisRepository`；真产品切 `PostgresRepository`。

## 二、为什么 Demo 用 dict 够了

```text
| 指标          | Demo 期望                | dict 表现 |
|--------------|-------------------------|-----------|
| 用户数        | 1 名评委 + 现场 demo     | ✅ 单进程 |
| 进程数        | 1 个 uvicorn worker     | ✅ 命中率 100% |
| 数据持久化    | 重启丢光也无所谓          | ✅ 不需要 |
| 跨设备同步    | 不需要                  | ✅ 不需要 |
| 数据分析      | 不需要                  | ✅ 不需要 |
```

但**只要任一项变 ✗**，就必须切持久化方案。

## 三、MVP 阶段（1-2 月）：Redis

### 3.1 触发条件

- 上线多实例后端（≥ 2 个 uvicorn worker）→ 进程间状态不共享
- 用户跨设备打开（手机 → PC）→ 上下文丢失
- 用户重新登录（24h 后）→ 偏好画像应保留

### 3.2 数据布局

```text
| key 模式                    | 类型     | TTL     | 内容                       |
|----------------------------|----------|---------|---------------------------|
| user:{user_id}:profile     | hash     | 永久    | persona + memory          |
| session:{session_id}       | hash     | 24h     | turn 列表 + intent + itinerary |
| session:{session_id}:trace | list     | 1h      | 最近一次 SSE 事件流        |
| stats:daily:{date}         | hash     | 7d      | 当日 active users / 完成数 |
```

### 3.3 关键防御

- **降级链**：Redis 不可达 → 自动 fallback InMemoryRepository（业务不中断 + 告警）
- **数据迁移**：MVP 启动时不迁移历史数据（Demo 数据不重要）
- **memory 衰减**：Redis 侧定时任务每天截断 `accepted_tags / rejected_tags` 到最近 20 条

### 3.4 切换工作量

```text
| 工作                            | 人时   |
|--------------------------------|-------|
| 写 RedisRepository（实现 5 方法）| 6 h   |
| 序列化层（Pydantic ↔ Redis hash）| 3 h   |
| 降级开关 + 健康检查             | 2 h   |
| 端到端回归                      | 2 h   |
| 总计                           | **13 h** |
```

## 四、真产品阶段（3-6 月）：PostgreSQL

### 4.1 触发条件

- 用户行为分析需求（哪些 tag 接受率高 / 哪些场景转化率高）
- 跨月份留存数据（产品决策需要纵向数据）
- 个性化模型训练数据（输入 → 决策 → 反馈的全链路样本）
- 法律合规要求审计日志

### 4.2 表设计骨架

```text
users                  user_id, persona_id, created_at, last_active_at
user_memory            user_id, accepted_tags(jsonb), rejected_tags(jsonb), distance_history(jsonb)
sessions               session_id, user_id, started_at, ended_at, turn_count
turns                  turn_id, session_id, seq, kind, payload(jsonb), created_at
itineraries            itinerary_id, session_id, status, payload(jsonb), confirmed_at
trace_events           event_id, session_id, turn_id, kind, payload(jsonb)
analytics_daily        date, dim_persona, dim_scene, count, avg_tags_accepted
```

完整 schema 在 Phase 1.0 之后维护。

### 4.3 Redis ↔ PG 双层

```text
Redis  ←→  缓存层（hot session、最近 1h trace）
PG     ←→  存储层（user / memory / itinerary 全量历史）
```

读写策略：

```text
- 写：先写 PG → 异步 invalidate Redis（保证持久化）
- 读：Redis 命中走缓存 → miss 回源 PG → 写回 Redis（TTL=1h）
```

### 4.4 数据合规

```text
| 合规项               | 实现策略                                |
|---------------------|----------------------------------------|
| GDPR 类「被遗忘权」  | DELETE FROM users WHERE user_id = ?     |
| 等保二级日志保留     | trace_events 保留 180 天后归档对象存储   |
| PII 脱敏            | turn.payload 序列化前过滤 phone/邮箱     |
| 数据出境            | 所有数据库部署在国内可用区               |
```

### 4.5 用户行为 analytics

PG 上加 ETL 任务（dbt / Airflow）派生：

```text
| 指标               | 计算方式                                  | 用途              |
|-------------------|------------------------------------------|------------------|
| tag 接受率         | accepted_tags / (accepted + rejected)    | 哪些标签真受欢迎  |
| 场景转化率         | 完成 confirm / 进入 planner              | 哪个场景产品力强  |
| 反馈重规划次数     | refine 次数 / session                    | 一次出方案的能力  |
| persona 切换收益   | 切换 persona 后的 confirm 率提升          | 个性化是否有效    |
```

详见 [`03-观测性骨架.md`](./03-观测性骨架.md) 关键指标章节。

## 五、切换里程碑

```text
| 时间        | 持久化方案                       | 验收标准                          |
|-------------|---------------------------------|----------------------------------|
| 现在（Demo）| InMemoryRepository（单进程 dict）| 重启清空也无所谓                  |
| +1 月（α）  | RedisRepository                  | 多实例后端 + 跨设备同步 OK         |
| +3 月（β）  | Redis + PG 双层                  | 留存 + 行为分析齐全                |
| +6 月（v1）| 同上 + ETL + 等保二级合规         | 头部 5 商圈数据稳定运营            |
```

## 六、为什么这个抽象层 Demo 阶段就值得做

评委可能会问：「Demo 用 dict 就行了，为什么要先抽象？」

```text
理由 1：Demo 跑通 ≠ 切换成本低；上线前再加抽象 = 重写业务代码
理由 2：抽象层让单测可以 mock Repository 直接跑（不依赖 Redis）
理由 3：商业评委要看「这个项目可以工程化」的证据，而不是「能跑就行」
```

抽象层的代码量是 50 行（5 个方法 + 一个 ABC），而切换成本节省的是 13 h × 多次切换。这是显然的杠杆。
