# 观测性骨架

> 从 Demo 的 stdout 日志到真产品的 OpenTelemetry + Sentry，渐进可观测。

---

## 一、当前状态：structlog 文本日志

Demo 阶段所有 Tool 调用、LLM 请求、异常都通过 `structlog` 输出到 stdout，给评委现场看「Agent 在做什么」。

```text
2026-05-08T13:45:22  info   tool_call_start  tool=search_pois args={...}
2026-05-08T13:45:22  info   tool_call_end    tool=search_pois duration_ms=12 result_count=8
2026-05-08T13:45:23  info   replan_triggered reason=restaurant_full from=R001
2026-05-08T13:45:23  info   tool_call_start  tool=check_availability time=17:30
```

> 该抽象层由 `backend/agent/v2/observability.py` 提供（已就绪：`get_logger` + `trace_span` + `bind_session_context` + `LOG_FORMAT=text|json` 环境变量切换）。Demo 阶段直接走 structlog text；MVP 切 json + Sentry；真产品在 `trace_span` 内对接 OpenTelemetry。

---

## 二、structlog 与企业级方案的兼容性

```text
| 兼容方向            | 实现路径                                       |
|--------------------|----------------------------------------------|
| → OpenTelemetry    | structlog + opentelemetry-instrumentation 直接桥接 |
| → Sentry           | sentry-sdk 自带 structlog 集成                 |
| → Logfire          | logfire 与 structlog 完全兼容（Pydantic 系出生）|
| → Datadog / 阿里 SLS | json 格式 + log agent 抓取 stdout              |
```

structlog 选型本身就是为「往后好换」做的——业务代码 `log.info("tool_call_end", tool=..., duration_ms=...)` 不动，下游 sink 替换是配置项。

## 三、Demo 阶段：text format

```python
# 当前配置（backend/main.py 启动时）
structlog.configure(
    processors=[
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.add_log_level,
        structlog.dev.ConsoleRenderer(),  # 人类可读
    ],
)
```

**为什么 Demo 用 text 而不是 json**：现场演示时评委希望直接看到日志输出，json 太密集；text format 加颜色更直观。

### 4.1 Sentry 接入

错误监控用 Sentry（个人开发者免费，月 5000 errors）。

> 数据来源：[Sentry 定价 · Developer 计划](https://sentry.io/pricing/)。免费额度数字以官方实时为准，本文 2026-05 数据点供规划参考。

```python
# 一行启用
import sentry_sdk
sentry_sdk.init(
    dsn=os.getenv("SENTRY_DSN"),
    traces_sample_rate=0.1,  # 10% 采样
    environment="production",
)
```

报警规则：

```text
- LLM 调用 5xx > 5% / 5min  → 立即报警
- planner fallback 率 > 10% / 1h → 报警（说明 LLM mode 不稳）
- empty_candidates / 总查询 > 20% / 1h → 报警（说明 mock 数据或高德数据有问题）
- pytest 在 CI 中失败 → 立即报警
```

### 4.2 关键指标

```text
| 指标             | 计算方式                       | 阈值（MVP）          |
|-----------------|------------------------------|---------------------|
| 错误率（5xx）    | 5xx 请求数 / 总请求数          | < 1%                |
| P95 延迟        | /chat/stream 完整响应耗时      | < 15s               |
| LLM token 成本  | 总 token × 单价                | < ¥0.50 / session   |
| Tool 命中率     | 非 empty_candidates / 总调用   | > 70%               |
| confirm 率      | confirm / planner 完成数       | > 30%（北极星）      |
| refine 率       | refine / planner 完成数        | < 50%（高即体验差）  |
```

## 五、真产品阶段（3-6 月）：OpenTelemetry

### 5.1 接入路径

```python
# 切到 OTel 桥接
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter()))
trace.set_tracer_provider(provider)

# structlog 自动桥接 trace_id / span_id
```

### 5.2 关键 span 设计

```text
session（root）
├─ router_classify      input=...  result=planning
├─ intent_parse         persona=u_dad  memory_top5=...
├─ planner_run mode=llm
│   ├─ llm_call_1       tokens_in=850  tokens_out=120
│   ├─ tool: search_pois        duration=12ms  count=8
│   ├─ tool: search_restaurants duration=15ms  count=5
│   └─ llm_call_2       tokens_in=...  tokens_out=...
├─ assemble_itinerary
└─ done
```

每个 span 携带：

```text
- session_id / user_id / persona_id
- intent_id / itinerary_id
- input/output 大小（token 数）
- 失败原因（如有）
- 耗时
```

### 5.3 后端 sink 选项

```text
| 方案              | 月成本（10k DAU 估算）| 优点                       |
|------------------|--------------------|--------------------------|
| Logfire（自建）   | ¥0（自托管）        | Pydantic 原生 + UI 友好     |
| Sentry（云）      | ~ ¥600              | 错误 + 性能一体             |
| 阿里云 SLS        | ~ ¥1500             | 国内合规 + 大数据查询       |
| Datadog           | ~ ¥3000             | 功能最全，但海外            |
```

> 成本估算基于公开定价 2026-05 数据点，实际以官方价为准。

推荐路径：β 阶段先 Logfire（免费 + 上手快），DAU 上来后切 SLS（国内合规 + 大数据）。

### 5.4 业务大盘

```text
| 大盘             | 维度                              | 目的                |
|------------------|-----------------------------------|--------------------|
| 漏斗大盘         | 输入 → 意图 → 规划 → 行程 → confirm | 哪一步流失最严重    |
| 错误大盘         | 错误率 × Tool / persona / scene    | 谁拖累整体可用性    |
| 体验大盘         | refine 率 × persona / scene       | 哪些场景出方案最差  |
| 成本大盘         | token 成本 × persona / scene       | 哪些场景太贵需要降本|
| 商业大盘         | 抽佣 GMV × 商家 / 品类             | 哪些商家产生收入   |
```

## 七、为什么这个抽象层 Demo 阶段就值得做

评委关心：「日志写到 stdout 跟评分有什么关系？」

```text
理由 1：评分项 4「Agent 行为可视化」直接受益——评委现场看到完整调用链
理由 2：单测可以断言日志事件序列，比断言函数返回更稳
理由 3：上线后不用重写业务代码加埋点
```

观测性是产品级项目的「肌肉记忆」——商业评委一眼看出「这队懂工程」。
