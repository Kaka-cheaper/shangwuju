## 1. 执行摘要

**一句话病根**:用户显式点名的诉求(「我想吃个烧烤」→ `preferred_poi_types=["烧烤"]`)在工具层被**从推断出来的软场景信号**(`experience_tags=["独处舒缓"]`、`social_context="独处放空"`)**当成硬性、不可放宽、且静默的过滤器删掉了**;而那个显式诉求本身**连工具入参都不是**,只是编排层一个事后重排提示——救不回已被删除的候选。**推断的软偏好压过了用户的硬性显式请求,还删得无声无息。**

**最伤系统主题的一条(H1/H2)**:系统主题是「诚实、接地、尊重用户显式诉求」。H1/H2 同时违反三条——删掉用户明说的(不尊重显式)、换成推断的场景(不接地)、且**告知机制(`relaxed_tags`/`exit_audit`)结构上看不见这次删除**(被删的候选没有节点可审计),所以连"诚实"层都无法讲出来。系统会自信地、静默地给用户一个他没要的东西且从不提及——正是主题的反面。

**三档路线图速览**:

| 档 | 名称 | 内容 | 触发信号 |
|---|---|---|---|
| **Tier-1** | hackathon 必修 | 检索层 anchor-escape(治 H1)+ 拆遮蔽测试 + 过度过滤埋点 + 诚实空态 | 就是当前烧烤 bug,路演在即 |
| **Tier-2** | 真实门槛 | 幂等+审计账本、确定性确认闸+saga、provider 契约+本体映射、信任边界 | 任一写工具不再是纯函数(接真支付/真商家)那天 |
| **Tier-3** | 架构债 | 领域策略层、语义检索、god-object 拆分、离线评估、演进纪律、反馈闭环治理 | 不阻断 demo,不还迟早爆 |

## 2. 审计:约 49 条发现

> 严重度:**严重**(概念完整性破坏 / 上线阻断)· **高** · **中** · **低**。出处链接见[第 8 章](#8-sources)。

### 2.1 四类最佳实践对照

**类别一 · 工具内部设计**

| 编号 | 命名范式 | 晌午局现状 / 差距 |
|---|---|---|
| 1.1 | 返回高信号信息(Anthropic) | `search_*` 返回完整对象含 id;`relaxed_tags`/`effective_distance_max_km` 标为纯调试;后者**从不被 set**(死字段) |
| 1.2 | 可执行错误(Anthropic) | `FailureReason` 枚举是好基线,但 `EMPTY_CANDIDATES` 不说是哪个约束打空的(无 MFS) |
| 1.3 | token 高效返回 | 有 `limit`,但截断只按 `rating`(与相关性正交) |
| 1.4 | 严格结构化输出(OpenAI strict) | 有 `extra="forbid"`,但未用 `strict:true` + 全 required/null 表可选 |
| 1.5 | 读/写行为标注(MCP) | `ToolSpec` 无 read/destructive/idempotent 标注,读写不分 |
| 1.6 | 幂等键(Stripe) | 写工具 `order_id` 用 wall-clock 派生 → 非幂等,重试双订 |
| 1.7 | 确认/dry-run 闸(MCP/LangGraph HITL) | 「绝不自动下单」仅 prompt 纪律,无 prepare/commit |
| 1.8 | 确定性/纯净 | 写工具 wall-clock id → 非确定,破坏幂等+可复现 |
| 1.9 | 可观测性 | `tracer.emit` tool_call + duration —— **相对强项**;但降级只对 dietary/physical 有痕 |

**类别二 · 工具与工具联动**

| 编号 | 命名范式 | 现状 / 差距 |
|---|---|---|
| 2.1 | 单工具不互调(机制/策略分离) | 显式强约束,原则正确;但跨工具关切全推给编排层而编排层写侧没建 |
| 2.2 | 工具合并(Anthropic consolidation) | `check_availability`+`reserve` 拆两低层 op,reserve 内部又二次重查 |
| 2.3 | Saga + 补偿事务(Richardson) | 三个独立写、无 saga、**无 cancel/refund 工具** → 部分失败孤儿订单不可回滚 |
| 2.4 | 跨工具契约一致性 | party_size 三处口径不一;距离曾双世界(已用 `venue_distance_km` 收口) |
| 2.5 | 状态传递显式化 | `exclude_visited` 由调用方喂入(对);但 memory **写**路径是未审计副作用 |

**类别三 · 整体工具架构**

| 编号 | 命名范式 | 现状 / 差距 |
|---|---|---|
| 3.1 | 两阶段:召回 vs 排序 | 单工具 filter+rank+truncate 三合一,相关性重排 bolt 在三个编排点 |
| 3.2 | 工具清单(读/写/查证/取详情/地理) | 缺:按 id 取详情、cancel/refund、why-not、geocoding |
| 3.3 | 命名空间(Anthropic) | 扁平名,无 read/write 前缀 |
| 3.4 | 契约统一 vs 分裂 | `SearchPoisInput` vs `SearchRestaurantsInput` 分裂,而 ILS 同池竞争 |
| 3.5 | 注册与发现 + 自动 spec | `to_openai_spec`/`all_specs` 已做(强项);无标注/版本 |
| 3.6 | 版本与演进 | `extra="forbid"` 严但脆(redis checkpoint 删字段崩) |
| 3.7 | 深工具 vs 薄机制 | 偏薄单一职责,把协调成本推给编排层(bug 聚集处) |

**类别四 · 边界情况**

| 编号 | 命名范式 | 现状 / 差距 |
|---|---|---|

### 2.2 当前工具批判性审计 H1–H11

| 编号 | 命名范式 | 晌午局现状 | 严重度 |
|---|---|---|---|
| **H1** | 优先级倒置 / RAG over-filtering | 推断 `experience_tags`/`social_context` 在 `_non_tag_filter` 硬删显式烧烤候选(`search_restaurants.py:79,81`),`preferred_poi_types` 只事后重排救不回 | **严重** |
| **H2** | 诚实层盲区 | experience/social 在 `additional_filter` 恒定生效,永不进 `relaxed_tags`;`exit_audit` 不审 social、也审不到被删候选 → 静默删除处诚实层什么都说不出 | **严重** |
| **H3** | 修复层错位 + 测试遮蔽 | anchor 修复在 `_utility`/critic 下游,盖不住工具删除;回归测试 `_patch_tool` monkeypatch `invoke_tool` 绕过真过滤 → 绿测红演 | **严重** |
| **H4** | 无候选生成 vs 排序分离 | 工具 filter→rank→truncate 只返幸存者,不返"谁被过滤为什么" | 高 |
| **H5** | 纯 rating 排序 + 硬截断 | 相关候选排 15 名外即丢;`fetch_limit=15` 扩池只缓解不根治 | 高 |
| **H6** | 双真源 | 工具 `social_context not in suitable_for`(二值)vs critic 四级 `social_compat` 矩阵 | 中 |
| **H7** | 双真源 | party_size:`search_adapter` 自算 vs planner 直传 `capacity_requirement` | 中 |
| **H8** | 检索/预订职责拆分副作用 | 检索无可用性/营业信号 → 选中闭店/售罄店到 critic 才拦 → repair 空转 | 中 |
| **H9** | 工具清单缺口 | 无 lookup-by-id / why-not / geocoding;POI/餐厅契约分裂;route 双系统 | 中 |
| **H10** | function-calling 接口面 | 死字段 `effective_distance_max_km`;`EMPTY_CANDIDATES` 无 MFS;无 namespace | 中低 |
| **H11** | 静默复合过滤 | `exclude_visited` 等 AND 复合、零可见性 | 低 |

### 2.3 工具↔外部世界 N1–N18

| 编号 | 命名范式 | 晌午局现状 | 严重度 |

## 3. 目标架构:4 个接缝

**收敛判断**:约 49 条发现不是 49 个补丁,而收敛成 **4 个接缝**——这是"一套连贯架构"而非"待办清单"的关键。

### 3.1 三层分工 + 依赖图

**三层各自的权威(治 H1/H4/H5/N8 的根)**——由一个 grounded 事实定死:`plan_hybrid` 步骤 1 先召回,步骤 2/3 才算 `weights_llm` + `semantic_scores`,**检索结构上拿不到规划权重**。

| 层 | 打什么分 | 用什么信号 | 权威于 |
|---|---|---|---|
| **Retriever(粗排)** | 相关性/入池顺序 | anchor 命中 + tier + rating(无 LLM、确定性) | 谁进池、谁不被截断 |
| **规划 `_utility`(精排)** | 多目标 plan-fit | PlanningWeights + semantic_scores + `_PREFERRED_ANCHOR_BONUS` | 池内谁进行程、排哪槽 |
| **critic(校验)** | 硬/软合法性 | DomainRules(social_compat 等)+ 预算 | 方案合不合法/该不该告知 |

**依赖图(无环、依赖倒置)**:

```
                    schemas/              (纯数据，无依赖)
                       ▲
                 domain_rules/            (纯策略：social_compat / capacity / distance / age_caps / meal_windows)
              ▲            ▲            ▲
   ┌──────────┴──┐  ┌──────┴──────┐  ┌──┴─────────┐
   │  Retriever  │  │   critic    │  │ Ledger+Saga│
   └─────────────┘  └─────────────┘  └────────────┘
     (三 planner 只调 Retriever.search)   (confirm 节点驱动)
   domain_rules 谁都不 import ⇒ 无环；策略是稳定内核（DIP + SDP）

   ToolContract(registry/schema)横切四者：MCP 标注 / strict schema / selectable-receipt 分离 / C1 覆盖断言
```

**四接缝与主题的映射(一个思想的四个面)**:Retriever = 尊重显式 + 对过滤诚实;DomainRules = 策略单源(两把尺子判本身是不诚实);Ledger+Saga = 副作用诚实/安全;ToolContract = 让前三者可强制、且对 LLM 诚实。

### 3.2 接缝一 · Retriever

**对外契约**(三 planner 只调它,取代散在工具 + 6 构造点的召回):
```python
def search(intent, kind: Literal["poi","restaurant"], *, limit, session_id, user_id, provider=None) -> SearchResult

SearchResult:
    selectable: list[Scored]     # 唯一可进规划；check_tool_consistency 只校它
    receipt: RetrievalReceipt    # 证据/诚实层用；与 selectable schema 硬分离(治 4.6)

Scored:  entity; tier: Literal["anchor","soft_fit","baseline"]; anchor_matched: bool; rating
         # 刻意不带 plan-utility —— 那是下游 _utility 的职责
RetrievalReceipt:  requested_anchor_terms; funnel: list[FilterStage]; anchor_status; relaxations; dropped_sample(仅证据)
```

**五阶段内部**:
- **S0 source**:provider 一页 / mock 全量。
- **S1 硬门(删)**:距离 + dietary/physical **HARD** ALL-match + capacity + exclude_visited + social **BLOCKING**(经 social_compat)。experience_tags 不在此;social 非 BLOCKING 不在此。
- **S2 anchor 分区**:`anchor_terms` 非空 → 切 anchor-matched / rest。匹配 = 词法 `*_desire_match` 保底 OR(配语义)`sim ≥ τ`。
- **S3 分层排序(非加权和)**:tier(anchor > soft_fit > baseline),tier 内按 rating。soft_fit = experience 重合 OR social_compat ACCEPTABLE+。**刻意不算 plan-utility**(落实自审:不臆造无法校准的权重公式)。
- **S4 配额选取**:填 `limit`,先为 anchor tier 留 `q` 席,再 soft_fit,再 baseline → anchor 命中者即使 rating 低也不被截断(治 H5/H1)。

**anchor 横跨两层**:检索层用**配额保证进池**(新,必需);规划层保留 `_PREFERRED_ANCHOR_BONUS` 让它**赢得槽**(已存在、已正确)。不冗余——一个管在场、一个管胜出。`over_budget` 从检索移除(纠早期公式之错:预算是方案级总额,归 `_utility`/critic)。

**迁移(characterization 逐步钉)**:①快照现状 → ②第一节脊椎(删两硬删 + anchor 配额,逻辑留工具内) → ③抽 `Retriever` 类(逐字节复现) → ④删 `_rerank_by_preferred_*` → ⑤receipt schema 化 + 接 exit_audit → ⑥D8 provider + 语义 anchor + soft_fit 改用 DomainRules。

**prior art**:两阶段候选生成 vs 排序;bi-encoder 召回 + cross-encoder 重排;RAG over-filtering;HARPO 硬/软约束集。

### 3.3 接缝二 · DomainRules

**接口形态**:中立顶层包 `backend/domain_rules/`,一包**彼此独立的纯函数小模块**(非门面):
```
domain_rules/social_compat.py   evaluate(social_context, suitable_for) -> (CompatLevel, reason)
             capacity.py         required_party_size(intent)->int; table_fits(cap,party)->bool; needs_large_table(party)->bool
             distance.py         within(km, max_km, mode) -> bool     # 四档容差(tool 0/critic 0.5/grounding 1.0/relax 2.0)集中一处
             age_caps.py / meal_windows.py                            # 已单源，归位
```

**防上帝模块四纪律**(配 import-linter 测试):① 无门面(import 叶子,不 import 一个万能对象);② 包内零跨模块 import;③ 零共享可变状态(全纯函数);④ 只依赖 `schemas`(出现 agent/tools/planning 即 CI 红)。

**依赖方向**:Retriever/critic/reserve 都 import domain_rules,domain_rules 谁都不 import → 无环。这是依赖倒置(DIP)+ 稳定依赖原则(SDP):策略是最稳定的内核。**放顶层不放 `agent/`**——否则 `tools/reserve` import `agent/` 是底层依赖上层的倒挂。

**Strangler 迁移**(每步 a=纯搬迁 / b=行为改变,分开提交):
1. **social_compat**(收 H6/N8):1a 从 `critic/` 搬到 `domain_rules/`(现有 critic 测试全绿=不变);1b 工具 social 门从二值升"只删 BLOCKING",新 golden。
2. **capacity/party_size**(收 H7):2a 抽 `table_fits`/`needs_large_table`;2b 引 `required_party_size` 作规范,**同一 commit 同时**改 search 与 `check_capacity` 指向它(保"搜索与 critic 用同一数"不变量)。
3. **distance**(四常量合一):3a 抽 `within(mode)`,各调用点保各自 mode(不变);3b 显式摊开后再评估该不该统一。

**隐藏耦合归属**:补偿映射(reserve↔cancel)是**写工具拓扑知识**,归 Saga 节点,**不归 DomainRules**(否则破"只 import schemas")。

**缺字段两档策略(接 N10)**:
- **偏好维度(social/experience):缺 = 宽(不删)**——social_compat 已"缺 suitable_for → ACCEPTABLE";编码成 `PREFERENCE_ON_MISSING=LENIENT`。缺字段删候选正是 over-filtering。
- **安全硬维度(无障碍/忌口):缺 ≠ 满足**——`SAFETY_ON_MISSING=KEEP_BUT_FLAG`:不静默通过、不静默删,保留 + 打 `safety_unverified` 标,诚实告知"无障碍信息没核实到,请确认"。
- **横切不变量**:absence of evidence ≠ evidence of violation。

### 3.4 接缝三 · OrderLedger + BookingSaga

> **生产门槛,今天不建**(判据见下)。给终态设计。

**OrderLedger**:
- 幂等键 `= sha256(session_id | plan_id | step_index | tool | entity_id | canonical_json(params))`,PREPARE 铸、COMMIT 复用;`order_id` 确定性派生自 key(杀 wall-clock)。
- schema:`{idempotency_key(PK), saga_id, session_id, plan_id, step_index, tool, entity_kind, entity_id, params_hash, status, result(冻结), error(冻结), attempt_count, hold_expires_at, authorized_by, ts}`;`status ∈ PREPARED|COMMITTED|FAILED|COMPENSATING|COMPENSATED|COMPENSATION_FAILED|EXPIRED`。
- durable 落点:**生产**要重启可存活(CloudBase 关系库/持久 KV);**demo** 镜像 `SESSION_STORE` 模式(默认 `memory`,重启即失——诚实:memory 模式幂等只进程内有效)。
- Stripe 语义:按 key 复现首次结果(成功/失败都冻结、都复现),per-key 锁防房间并发双执行。

**BookingSaga 状态机**:`DRAFT→PREPARED→AWAITING_CONFIRM→COMMITTING→(COMMITTED | COMPENSATING→COMPENSATED/COMPENSATION_FAILED)`;PREPARED 带 `hold_expires_at` + reaper 超时→`EXPIRED` 释放 hold(治 hold 泄漏 R3);补偿映射静态表 `{reserve_restaurant:cancel_reservation, buy_ticket:refund_ticket, order_extra_service:cancel_extra_service}`(**前置:补偿工具今天不存在,需先补**)。

**确定性确认闸**:ToolSpec `destructiveHint` 驱动图层——破坏性工具**必经** prepare→interrupt→commit,读工具直通(取代 prompt-only 的「绝不自动下单」)。`execute_finalize` 拆 `prepare_writes`(落 PREPARED + 报价 manifest)→ `interrupt(manifest)`(LangGraph 挂起,现有 checkpointer)→ `commit_writes`(approve 跑 saga);`_persist_memory_side_effect` 移到 commit 成功后。

**写侧信任边界(D9)**:`check_tool_consistency` 扩到写工具(`restaurant_id ∈ 方案已定实体`);单轮写预算 = 已批准 manifest 条目集(治 N16);房间授权链(C2)`authorized_by` + `PaymentContext`。

**"今天不建"判据 + 触发信号**:写工具是 mock 纯函数,无真副作用可幂等/补偿/多扣,且「绝不自动下单」今天结构已成立。**触发器二元清晰:任一写工具不再是纯函数(接真商家/真支付)的那一刻——上线前这套必须先在。**

**红队**:prepare↔commit TOCTOU(拿真 hold 或 commit 重验)、interrupt 永不 resume(reaper 兜)、`COMPENSATION_FAILED`(**唯一强制人工升级 + 诚实告知**的终态)、房间并发 commit(per-key 锁)。

**prior art**:Stripe 幂等;Saga/补偿事务(Richardson);LangGraph HITL interrupt;MCP 标注。

### 3.5 接缝四 · ToolContract

**registry 级升级**:
- **MCP 标注** `ToolAnnotations{read_only, destructive=True, idempotent=False, open_world=True}`(默认悲观:未标注=破坏性,fail-safe),驱动 3.4 的确定性门。
- **strict schema**:`to_openai_spec` 置 `strict:true` + 全 required/null 表可选 + `additionalProperties:false`;`parallel_tool_calls=false`。杀"漏字段/漂移"。
- **可执行错误**:`FailureReason` 补 TIMEOUT/RATE_LIMITED/STALE_DATA/PARTIAL_RESULT/AUTH_QUOTA;`ToolOutputBase` 加 `recovery_hint`/`retry_after_s`。

**selectable/receipt 硬分离(收口 4.6)**:输出声明 `__valid_pool_field__="selectable"`;`dropped_sample` 是 `DroppedRef` 异类型(非 `Candidate`);`check_tool_consistency` 读声明字段 → **结构上**不可能把被拒候选当可选。一次定义(契约)、三方遵守(Retriever 产、critic 校)。

**C1 覆盖纪律 CI 断言**:`schemas/intent_coverage.py` 声明表 + `tests/test_intent_tool_coverage.py`:每个非元数据字段须有声明消费者;标 `TOOL_INPUT` 者须真接线(今天跑即红 `preferred_poi_types`/`budget`)。

**工具描述即 prompt(P3)**:描述须暴露"哪些硬过滤/哪些字段不进检索/失败恢复";字段 `description` 是进 LLM 的 prompt,须写单位/范围/软硬;评估含 tool-selection 准确率。

**演进的非对称严格原则(治 N17)**:摄入端严(input:`extra=forbid`+strict,抓漂移)/ 演进端宽(output:`extra=ignore`,additive 容忍)/ 删除端 deprecate-不-delete(持久 schema 加 `contract_version` + 兼容 fixture)。→ 加字段不必全链路开刀。

## 6. 概念完整性与自我修正

**一个思想**:系统主题(诚实、接地、尊重显式诉求)一路贯到底的四个面(见 [3.1](#31-三层分工--依赖图))。**不是四个各自漂亮的补丁。**

**三处缝靠棘轮才不漂**:
1. **"同一把尺子"只靠纪律、无结构强制**:这仓历史恰恰是反复各写本地版(H6/H7)。DomainRules 减少但不消除漂移;**真正兜住的是棘轮测试**(import-linter + C1 覆盖断言)——没棘轮,缝漂回去。
2. **Retriever 粗排 vs `_utility` 精排的边界建立在时序偶然上**:权重在检索之后算,是 pipeline 排序契约而非硬墙;有人为效率把语义打分挪进检索,分离就塌。
3. **Ledger+Saga 是孤岛,风险是与 plan 状态失同步**:`saga_id/plan_id` 须与方案生命周期绑牢(C4)。

**自我修正(如实记录,不美化)**:
- **写侧脊椎(Ledger/Saga)、语义检索是生产门槛,不是 hackathon 工作**——第 8 轮我把它们与 bug 修复混在一个紧迫叙事里;mock 纯函数无真副作用可保护,50 条 catalog 不需要 dense 检索。
- **R2 的 blast radius 我夸大了**:grounding 显示 `build_route` 早有 `pinned`(保证进行程)+ `_PREFERRED_ANCHOR_BONUS`(优先级)两套机制、且已正确;bug 纯在"候选从没进池",最小修复纯检索层,**规划层一行不碰**。
- **三处未验证假设**:D5"durable store"(demo 默认 `SESSION_STORE=memory`,不满足)、D1"新权重公式"(无离线集可校准,改为 tier 分层不臆造权重)、D11"预烤 embedding"(真 provider 后 catalog 动态,不能预烤)。

## 9. 未竟的方向(元层诚实声明)

> **这一章是给未来自己/团队的诚实边界声明,不美化、不删。**

这次探索(十五轮)**从没离开过机器**。

整件事由一个**用户面的症状**触发——「我想吃个烧烤」给了个安静咖啡馆。然后我们用十五轮把它变成了纯**系统工程**:工具内部 → 接缝 → saga → 契约,越挖越深。但**两个方向从没走过**:

**没往上(paradigm)**——从没质疑"把模糊一句话塞进 21 字段刚性 `IntentExtraction`、再对字段硬过滤"这个**槽填充范式本身**是否适合这个域。整个"推断 tag 变硬过滤"的 bug 类,是这个范式的下游产物;换个范式(intent 作软上下文而非硬槽、或更端到端的检索增强)可能**从根上溶解**整类 bug。我们在槽填充管道**内**优化了十五轮,没问过槽填充对不对。

**没往外(user)**——我们设计的"诚实"几乎全是**后端诚实**(receipt / advisory / exit_audit)。从没问过:一个真人在 3 秒扫一眼方案时,**感不感知到**这份诚实?**我们把"系统内部正确/诚实"等同于"用户觉得被理解了"——这是工程师偏见。** 十五轮零轮谈那个真人:他能忍什么、根本不会注意到什么、什么才真让他信任。后端诚实 ≠ 用户被理解。

**元观察(不辩解)**:探索的**追问机制本身**——"继续挖新洞、加到 15 轮"——结构性地**奖励了深、惩罚了跳出**。激励产生了这个偏见:深度优先、边界零广度;向下钻(工具→接缝→saga→契约)钻得极深,却从不向上(范式)、向外(用户)。**把一个信任/UX 问题,当成了纯粹的正确性工程问题。**

**给未来的建议**:在照本文档动工之前,先花一轮做这两件本文档欠的事——(1) 问一次"槽填充是不是这个域的对范式";(2) 找一个真实用户,看他 3 秒扫方案时到底感不感知到我们设计的诚实。**如果这两个方向的答案动摇了地基,本文档第 3–5 章的接缝设计可能是在优化一个不该存在的管道。**
