# 团队分工方案

> 3 人团队 · 1 个月时间盒（2026-05-08 → 2026-06-08）
> 本文件 session 1 拍板，后续按实际情况微调。

---

## 一、角色定义

### A · Agent 主轴

> 最强 Python + LLM 经验的人

**职责范围**：

- `backend/agent/`：LLM 客户端 wrapper（`llm_client.py`）、意图解析（`intent_parser.py`）、Planner + Executor（ReAct 循环）、异常重规划
- `backend/prompts/`：system prompt、few-shot 示例
- `backend/schemas/`：§5.7 IntentExtraction 等 Pydantic v2 model
- `backend/main.py`：FastAPI 入口 + SSE 流式端点 `/chat/stream`
- 与 C 共同定义 Tool 调用契约（输入 JSON Schema + 输出结构）

**核心风险**：意图解析准确率 + 异常重规划——直接对应评分项 1（Agent 编排能力）和 3（异常韧性）

**交付检验**：8 个演示场景文案 → 每个输出正确 §5.7 JSON → 完成 ReAct 循环 → 异常分支至少触发 1 次并自动恢复

---

### B · 前端 + 演示

> 有 React / Next.js 实战经验的人

**职责范围**：

- `frontend/` 全部：Next.js 14 App Router + TypeScript + Tailwind + shadcn/ui
- 聊天框组件：SSE 消费 + 流式打字效果
- 行程卡片组件：时间轴形式呈现最终方案
- Tool 调用链路可视化：展示 Agent 每步决策过程（评委加分项）
- 转发文案预览 + 一键复制
- 8 个快捷输入按钮（演示场景集 6-8 个）
- Demo 录屏 + 现场演示 dry run

**核心风险**：视觉冲击力——评委 5 分钟内"哇"时刻大部分来自 UI 质量

**交付检验**：端到端 Demo 跑通「输入 → 流式输出 → 卡片 → 确认 → 执行 → 转发」完整闭环

---

### C · Tool + Mock 数据 + 测试

> 细心 + 有耐心的人

**职责范围**：

- `mock_data/` 全部：POI / 餐厅 / 路线 JSON
  - MVP-1：≥20 POI + ≥30 餐厅（家庭走向优先）
  - MVP-2.2：30-40 POI + 50-60 餐厅（覆盖 6-8 种 `suitable_for`）
- `backend/tools/` 全部：6-8 个 Tool 实现（查询 mock_data → 返回结构化结果）
- 埋失败案例：≥2 个（餐厅没位 / 门票售罄）——直接对应评分项「异常韧性」
- `tests/` 全部：Tool 单测 + 端到端 Agent 流测试
- 按 `演示场景集.md` §四 覆盖表自检 Mock 数据完整度

**核心风险**：Mock 数据"像不像真的"——字段密度 + 业务约束完整度是评分项 4

**交付检验**：8 个场景 → 每个在 mock_data 中命中 ≥3 候选 + 至少 1 次触发异常分支

## 二、并行时间线

```text
         Week 1           Week 2           Week 3           Week 4
A(Agent) 骨架+LLM客户端    意图解析+Planner   Executor+重规划   集成联调+打磨
B(前端)  Next.js骨架       聊天框+SSE消费     卡片+Tool可视化   Demo准备+录屏
C(工具)  Mock数据v1        6个Tool实现        测试+失败案例     覆盖表自检+扩数据
         ──────────────────────────────────────────────────────────
         ↑ W1末首次联调    ↑ W2末意图→Tool    ↑ W3末全闭环
```

## 三、关键协作节点

| 时间 | 节点 | 参与角色 | 标志 |
|---|---|---|---|
| Week 1 末 | 首次联调 | A + C | A 的 LLM 客户端 + C 的 Mock 数据 v1 → 意图解析能跑通 |
| Week 2 末 | 意图→Tool 跑通 | A + C | A 的 Planner + C 的 6 Tool → 调用链路跑通 |
| Week 3 初 | 前后端对接 | A + B | B 的聊天框消费 A 的 SSE 流 → 前端能看到 Agent 输出 |
| Week 3 末 | 全闭环 | A + B + C | 输入→规划→确认→执行→转发→异常重规划 全部跑通 |
| Week 4 | Demo 打磨 | 全员 | UI 打磨 + 数据扩充 + 录屏 + 现场 dry run × 3 |

## 四、跨角色硬规则

1. **A 和 C 共同维护 Tool 契约**：每个 Tool 的输入 JSON Schema + 输出结构必须在 `backend/tools/` 模块 docstring 里写明，两人都能看到
2. **B 不直接调 LLM API**：前端只消费后端 `/chat/stream` SSE 端点（AGENTS.md §4.1）
3. **C 写的 Tool 不调其他 Tool**：每个 Tool 独立查 mock_data，不链式调用（AGENTS.md §4.1）
4. **代码 grep 门禁**：任何人 PR 前自检 `scene_type` / `relation_type` / `if scene ==` 不出现
5. **每周五快速 sync**：15 分钟过一遍各自进度 + 卡点，更新 `progress.md`
