# 如何使用这套文档（人类协作者视角）

> 这份文档是写给**团队成员**看的——告诉你这个仓库为什么有这么多 `.md` 文件、它们之间什么关系、你该读哪些。

## 一、为什么搞这么多文档

晌午局是 hackathon 项目，时间盒紧，看似不该花时间写文档。但我们仍然搭了这套体系，原因有 3 条：

1. **AI 协作密集**——本项目大量使用 Cascade / Claude Code 等 AI 编程助手。AI 没有"上次会话的记忆"，每次新开 session 都需要重新建立上下文。`AGENTS.md` + `progress.md` 是 AI 的"记忆载体"，比让 AI 反复读代码省 5 倍时间。
2. **多人并行**——前后端可能并行开发，需求理解必须先在文档对齐，否则联调一定撞车。
3. **评委视角**——hackathon 评分不止看 demo，也看"项目组织能力"。一份组织清晰的设计文档（≤2 页）是题目交付要求之一。

## 二、文档体系一览

```text
美团AI Hackathon/
├── AGENTS.md                    ← AI 优先入口、编码铁律
├── 项目说明.md                  ← 产品逻辑说明（人类友好、非技术可读）
├── 比赛详情.md                  ← 赛题原文（冻结、不动）
├── chatgpt分析.md               ← 需求拆解参考（冻结、不动）
├── 技术架构.md                  ← 架构候选方案（D-SoT 后可随决议修订）
└── docs/
    ├── 00-overview/             ← 项目级
    │   ├── progress.md          ← session 衔接（每次开发都要更新）
    │   └── 如何使用这套文档.md  ← 你正在读
    ├── 01-requirements/         ← 需求级
    │   ├── 需求分析.md          ← 赛题拆解为可验收条款（§5.7 意图抽取 schema SoT）
    │   ├── MVP定义.md           ← 最小可演示闭环
    │   ├── 验收标准.md          ← 怎么证明做完了
    │   ├── 架构选型.md          ← 选型决策记录 （D1-D9 已拍板）
    │   └── 演示场景集.md        ← 8 个开放场景脚本 + Mock 覆盖表

```

注：`02-design/` 和 `04-development/` 暂时不建——hackathon 项目体量不需要那两层。等真的有跨模块设计要写时再补。

## 三、按角色读哪些

### 我是非技术队友 / 家人 / 潜在用户

读 1 份就够：`项目说明.md`（5 分钟）。不需要看任何其他文档。

### 我是 AI Agent（Cascade / Claude Code / Cursor）

按 `AGENTS.md` §五 的"上下文恢复快速通道"读 4 步即可。

### 我是项目负责人 / PM

读这 4 份：

1. `项目说明.md`——产品状态一眼看完（5 分钟）
2. `比赛详情.md`——赛题原文（5 分钟）
3. `docs/01-requirements/MVP定义.md`——我们到底要交付什么（5 分钟）
4. `docs/00-overview/progress.md`——当前进度（1 分钟）

### 我是后端 / Agent 编排开发者

读这 5 份：

1. `比赛详情.md`（赛道 06 段）
2. `chatgpt分析.md` §三 / §四 / §五 ——需求拆解 + 系统能力分类 + Demo 形态参考
3. `docs/01-requirements/架构选型.md`——LLM 选型 / Tool 边界
4. `docs/01-requirements/需求分析.md`——约束词典（人物 / 饮食 / 体验 / 社交上下文 等跨场景通用维度，D9 后重构）+ `演示场景集.md` 6-8 个开放场景
5. `AGENTS.md` §三 / §四——MUST / MUST NOT

### 我是前端 / UI 开发者

读这 4 份：

1. `chatgpt分析.md` §五——Demo 形态描述（含输入/中间过程/最终输出/转发文案样例）
2. `技术架构.md` §一 / §二.1——UI 层职责
3. `docs/01-requirements/MVP定义.md`——UI 必做项 vs 加分项
4. `docs/01-requirements/验收标准.md` §3——UI 部分验收点

### 我是 Mock 数据 / 测试 / 评测开发者

读这 3 份：

1. `技术架构.md` §二.4——Mock 数据字段最低要求
2. `docs/01-requirements/需求分析.md`——业务约束（决定 Mock 数据要怎么编才"像真的"）
3. `chatgpt分析.md` §四.4——异常处理列表（告诉你 Mock 数据要埋哪些失败案例）

## 五、文档更新纪律

- **比赛详情.md / chatgpt分析.md**：**永不修改**。这是冻结项目输入。
- **技术架构.md / 项目说明.md**：D-SoT [2026-05-08] 后解锁为活文档，可随决议修订；大改必须在 `progress.md` 决策记录里留底。
- **AGENTS.md**：选型敲定后更新 §3.3「4 层架构边界」与 §3.8「代码风格」。其他段落只在 MUST/MUST NOT 出现新条款时改。
- **docs/01-requirements/ 下 5 份**：选型阶段密集改；进入实现后冻结大改、只允许小修。
- **docs/00-overview/progress.md**：每次 session 必更（即使没进展，也要在末尾追加 "session N: 无新进展，讨论了 X" 一行）
- **docs/03-implementation/pitfalls.md**：踩到坑就追加，不要追溯重写。

## 六、给团队的建议

1. **不要把 ChatGPT 输出直接复制到 docs/**——`chatgpt分析.md` 已经原样保留，docs/ 下要写"我们决定怎么做"，不是"ChatGPT 怎么说"。
2. **不要在 `progress.md` 里贴大段代码 / 完整文件路径列表**——它是衔接文件，应该 < 200 行。
3. **不要让 docs/ 数量超过 8 个**——超过就说明你想写的不是文档而是代码注释，写到代码里去。
