# ADR-0013 · 节点级协商:局部重解 + 调整按钮 + 房间轻量成员体系

- **状态**:Accepted(2026-07-03 · 需求谈判 + grilling 会话,用户逐题拍板)
- **范围**:规划层新增「局部重解」引擎;节点交互三元素(具名备选/定向调整按钮/赞踩);会话上下文家族新增「诉求台账」;协作房间轻量成员体系与路由同权;前端节点行/台账面板/进房卡。承接 D-7(pins+advisory,6c3c65d)、ADR-0011 决策 3「底座无关」增补(f5204fb)、ADR-0012(投影端口/字段生命周期表)。术语权威定义在 `backend/agent/planning/CONTEXT.md`(局部重解)与 `backend/agent/CONTEXT.md`(诉求台账)。
- **产品动机**:多人聚会的核心难点是成员偏好冲突的调解,不是选一个地理合适的点。现状把冲突压扁成单一折中方案端上来;本弧把冲突摆上桌面:节点级换菜 + 记名台账 + 共情解释,让推荐过程变成可见的协商。

## 背景(2026-07-03 读码钉死)

1. **粗粒度反馈是现存病**:嫌一个节点 → 整个方案洗牌。房间点踩合成「不满意第 N 段,请换一个」触发全量重排(room.py:273);多立场方案层被否——立场差异实质是单节点选择的投影(「辣味优先」vs「清淡友好」= 那家火锅换不换),做立场层 = 五个方案四个半重复。
2. **房间的协商是半成品**:约束流已记名(room.py:61,前端 ConstraintFeed 按人渲染)但消费即压扁进 intent(room.py:462-500,refine_intent 揉合后归名与节点指向全丢);自由打字绕过路由直连重规划(说什么都硬改方案);身份半成品(RoomClient.tsx:25-41,URL 昵称 + localStorage 持久 id);房间对象无 TTL 永活(内存泄漏 +「用完即毁」是空话)。
3. **前端家底比预想厚**:房间渲染完整 HomeView(RoomClient.tsx:77);气泡列表已在房间共享快照(collab-store.ts:93/247/463);药丸 chip 视觉语言已立(ItineraryCard 的 IntentChips);VoteButtons 现挂 stage 右侧。
4. **引擎零件全部已在生产**:pins(D-7)/修复闭环黑名单/贪心插入构造/统一 critic/advisory 管道;缺的只是把它们按「换这一格」语义粘起来的新函数。既有 alternatives 字段仅 LLM 路径产出(planner.py:73)且未验证可行性,不能直接当具名备选。

## 决策(每条标注用户拍板点)

1. **粒度 = 节点,立场层砍除**(Q0):单方案 + 节点级调整;路线级取舍走既有全局反馈通道,不建第三种东西。
2. **局部重解引擎**(Q3/Q4):纯函数、底座无关(主聊天与房间共用同一实现)——钉住其余节点 + 拉黑当前目标 + 带结构化诉求在缺口重解。语义承诺:其他节点的**选择**不变、时刻允许微调(挤破营业窗走告知);降级序列「同子类满足 → 同大类异子类满足 → 近似满足+告知」;**大类(kind)永不跨**(用餐永远换餐厅、活动永远换活动;跨大类=结构变更,归全局反馈);**直接换,无预览**(点即生效,反悔=再点;房间留有单层上一版兜底)。
3. **诉求台账**(Q1/Q2):会话上下文家族第三器官(轮次日志=证据、版本志=方案史、台账=诉求索引)。与 intent 分工:台账=「谁要了什么」的不压扁记名史(每条指回来源轮次,不存原话),intent=「当前有效全局约束」合并视图;全局语义诉求(忌口/预算)两边都进,节点级只进台账。**失效语义**:诉求不随重排自动死——节点在场=硬约束,节点没了降级为「尽量满足+主动提及+不满则告知」;**顶替规则**:同节点同维度后者顶替前者(旧条目标记不删),跨成员矛盾共存并暴露交 LLM 调解。单人模式同样记账(SESSION_SCOPED,生命周期表强制登记);「重规划时主动提及」的 prompt 供给**等 E-2 打包器,不建临时管**(用户拍板)。
4. **节点交互三元素与布局**(Q5/Q7):**右侧=具名备选**(「换成 XX 店」,必须由引擎预验证可行——试插通过才展示,拒绝拿未验证的 alternatives 充数);**下方=定向调整按钮 + 赞/踩并排**(定向按钮=「微辣的/安静点的」,赞踩从右侧挪下);**点踩收编**为无方向局部重解(拉黑当前、不带诉求、只换这格),一人踩立刻换(不等多数,拉锯即可见协商,台账全程记账);三者点击殊途同归进同一引擎,且都是**结构化指令,不过 LLM 路由**。
5. **按钮生成**:narrate 的既有 LLM 调用搭车产出(同次 JSON 增列,零额外延迟),每节点 ≤3 个、只生成该节点管得了的调整(路线级按钮禁入节点下方);stub/rule 模式按节点 kind 走模板兜底。
6. **房间轻量成员体系**(Q6):进房卡片(昵称必填才可进,复用 RoomClient 现有壳态);身份与其全部数据(台账条目/点击记录)**归属房间,房散即毁**;localStorage id 降级为断线重连凭证,不跨房间携带任何东西;**房间空闲 TTL 50 分钟**(最后活动起算,清扫器连台账一起销毁——「用完即毁」由此成真,顺带治房间永活的泄漏);房间内调整**串行队列**执行,被处理节点全员可见锁定态(Shimmer+禁用),后到的基于前一个结果+全部台账再解。
7. **房间路由同权**(用户 2026-07-03 定,ADR-0011 决策 3 已增补):房间消息处理器成为 route_turn 的又一层薄壳——成员自由打字先过同一个路由脑子,判出义务再房内分发(反馈→带台账重排;提问/陪聊→气泡进共享快照广播;防御→壳1 同款;歧义→保守引导),不再「说什么都硬改方案」;E 系列升级脑子时房间自动跟进。成员身份进判定上下文;澄清按成员分身归 E-3。
8. **排期**(Q8):E-0 收官 → **E-1 → F 系列(本 ADR) → E-2** → 路演前一周留给演示可见遗留四件+真 LLM 冒烟;E-3/E-4 视余量,大概率路演后(代价声明:路演版歧义输入走 E-1 后的保守引导,不反问——可接受)。

## 边界(不做)

持久成员画像(用完即毁是反义词)/鉴权(临时身份是名牌不是锁;接真钱前必须补)/房间迁图底座(ADR-0012 路标不变;台账房间侧存 Room 状态,E-2 打包器按多来源消费)/多立场方案层/跨大类结构变更/为「主动提及」建 E-2 前的临时 prompt 管道。

## 备选与拒因

- **多立场方案层**——拒(用户):内容重复+屏幕臃肿+让成员填偏好不现实;立场=节点选择的投影。
- **预览后确认**——拒:demo 魔法变表单;「悬而未决方案」中间态在多人房间语义复杂(进不进版本志/别人点击算不算数)。
- **点踩维持全量重排**——拒:嫌一家店全盘洗牌是本弧要治的病根。
- **降级序列跨大类**——拒:用户心理模型是「换这家」不是「改结构」;「只动一格」承诺会被打破;结构变更另有通道。
- **台账塞进轮次日志硬承载**——拒:结构化查询要扫全量日志、裁剪保护语义纠缠、证据与状态混淆;正解=第三器官+指回日志的指针。
- **多数票才换**——拒:现状(一人触发)平移,演示活性优先,台账可追溯兜底。
- **10 分钟 TTL**——改 50 分钟:演示中场讲 PPT 回来房间没了=台上事故;计时从最后活动起算。
- **单人节点诉求用完即弃**(主代理原推荐)——被用户推翻:单人也记账,「懂人的系统」要在重规划时主动提及;成本由台账+E-2 打包器复用消化,不重复建设。

## 实施拆步(F 系列;TDD,每片独立 commit 可回滚)

- **F-1 局部重解引擎**:纯函数 `resolve_node_swap(itinerary, pools, ledger_slice, target_node, adjustment) → (new_itinerary, advisories)`;复用 pins/黑名单/critic;**具名备选的预验证候选生成同在此层**(展示与点击同一真相源)。**机制修正(2026-07-03 实施前设计核查,主代理裁决)**:重解不走 `build_route`(其涌现填充循环在替换品更短时会加塞额外活动,破坏「只动这一格」承诺),改基于修复闭环的 `_repair_route` 语义(腾格→只补该格→不加塞),将其从 ils_planner 模块私有提升为共享 seam;已知取舍:替换候选无本会话 LLM 语义分,utility 走中性默认,与保留节点存在打分基准漂移——引擎免 LLM 的必然代价,docstring 记录。
- **F-2 诉求台账**:schema + 状态机(生效/被顶替/已满足)+ 单一写入 helper(同 diff 写台账+轮次指针);单人=图状态 SESSION_SCOPED 字段(完备性测试强制登记),房间=Room 状态;全局语义诉求同步揉 intent。
- **F-3 按钮与备选生成**:narrate LLM 搭车(JSON 增列)+ 按 kind 模板兜底;SSE payload 形状按统一 agent 消息面家族设计;前端具名备选/定向按钮数据落节点行。
- **F-4 单人入口 + 前端节点行**:结构化调整端点(实现时定:独立轻端点 vs /chat/turn 结构化 payload,倾向前者);节点行改造(右备选/下按钮+赞踩挪位/Shimmer 锁定态/归名小字);台账面板=升级 ConstraintFeed(单人模式也显示,加状态徽标+节点指向)。**范围追加(F-6 实施时发现的既有前端缺口)**:`ChitchatBubble` 的 chip 点击与确认按钮硬连单人主 store、不判 collabMode——房间里气泡按钮点击打到单人接口而非房间 WS 通道(ChatDock 有正确先例可抄);F-4 改造交互元素时必须一并修,否则房间内一切按钮(地板 chips、未来的节点按钮)都是死的。
- **F-5 房间成员体系**:进房卡;身份房内归属+重连凭证;TTL 50min 清扫器(连台账销毁);串行队列+锁定态广播;点踩收编;台账房间侧接线。
- **F-6 房间路由同权**:消息处理器接 route_turn + 义务分发表;气泡入共享快照广播;成员身份进上下文;特征化测试先钉现状再切。

**总验收**:S1-S8 结构测试不回归;演示脚本联排三幕——单人点按钮换菜(只动一格+归名解释)、房间多人抢点(串行+台账+冲突告知)、房间自由打字(陪聊不炸方案);实跑探针肉眼过文案。
