## 一、总体架构

```text
┌─────────────────────────────────────────────────────┐
│  Web UI (聊天框 + 行程卡片 + 转发预览)                │
└────────────────────────┬────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────┐
│  Agent 编排层（Planner + Executor）                  │
│  - Intent 解析  - 规划循环  - 工具调用  - 异常重规划 │
│  - 基于 LLM Function Calling / ReAct                │
└────────────────────────┬────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────┐
│  Tool 层（必须严格按 Function Schema 定义）          │
│  search_poi / search_restaurant / check_availability │
│  reserve / buy_ticket / route_eta / share_msg ...   │
└────────────────────────┬────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────┐
│  Mock 数据层（JSON / SQLite，"像真的"的业务数据）    │
└─────────────────────────────────────────────────────┘
```

## 二、四层职责

### 1. Web UI 层

- **聊天框**：接收用户自然语言输入，展示 Agent 中间过程（"正在查询附近亲子乐园…"）。
- **行程卡片**：以时间轴形式呈现最终方案（出发时间、地点、活动、预约状态）。
- **转发预览**：生成可一键复制的转发文案，模拟"发给老婆/朋友确认"。
- **Tool 调用日志可视化**（加分项）：让评委看到 Agent 的思考过程，凸显"Agent"而非"聊天机器人"。

### 2. Agent 编排层

核心循环采用 **ReAct / Function Calling** 范式：

```text
解析意图 → 调用 Tool → 观察结果 → 判断是否满足约束
   ↑                                       │
   └────────── 异常时重规划 ────────────────┘
```

关键能力：

- **Intent 解析**：从一句话中抽取结构化约束 JSON——详细字段以 `需求分析.md` §5.7 schema 为唯一权威（含 `start_time` / `duration_hours` / `distance_max_km` / `companions` / `physical_constraints` / `dietary_constraints` / `experience_tags` / `social_context` 等）。按 D9 决议，**不**抽取 `关系类型` / `场景类型` 这类枚举字段。
- **分层规划**：目标解析 → 候选生成 → 约束过滤 → 行程组装 → 执行确认。
- **异常重规划**：Tool 返回失败（餐厅没位、票售罄）时，自动切换备选策略。
- **执行确认**：用户点击"确认"后，调用执行类 Tool 完成模拟下单。

### 3. Tool 层

所有 Tool 严格按 LLM Function Calling 标准定义（JSON Schema），分两类：

| 类别 | Tool 示例 | 作用 |
|---|---|---|
| **查询类** | `search_poi` / `search_restaurant` / `check_availability` / `route_eta` | 提供决策依据 |
| **执行类** | `reserve_restaurant` / `buy_ticket` / `order_extra_service` / `generate_share_message` | 模拟下单/预约/转发 |

设计原则：

- Tool 数量控制在 **8–10 个**，宁少勿滥。
- 每个 Tool 必须有 **明确的输入约束** 和 **结构化输出**，避免 Agent "瞎调"。
- 故意在 Mock 数据里埋"没位 / 售罄"案例，触发异常分支以体现 Agent 韧性。

### 4. Mock 数据层

数据"像真的"是评分关键，最低字段要求：

- **POI**：名称、类型、距离、营业时间、评分、适合人群、价格区间、标签（亲子友好/适合拍照/室内）、库存。
- **餐厅**：名称、菜系、距离、人均、可预约时段、当前是否有位、排队人数、标签（低脂/亲子椅/4 人桌）。
- **路线**：地点 A → 地点 B 的预估通勤时间（步行/打车/公交分开给）。

数量建议（以 `架构选型.md` D4 + `需求分析.md` §三 分档）：

- **MVP-1**：≥20 POI + ≥30 餐厅，优先填家庭场景走向
- **MVP-2.2**：30-40 POI + 50-60 餐厅，覆盖 6-8 种 `suitable_for` 画像（家庭 / 朋友 / 情侣 / 带父母 / 闺蜜 / 商务 / 独处 / 跨代际，具体见 `演示场景集.md`）

## 三、技术选型

| 层级 | 技术 | 理由 |
|---|---|---|
| **后端 / Agent** | Python 3.11+ + FastAPI + Pydantic v2 + sse-starlette | LLM 生态成熟、Function Calling SDK 完善、SSE 流式原生支持（D6） |
| **LLM** | **DeepSeek-V3 主 + 通义 Qwen-Plus 备**（`openai` Python SDK + `base_url` 切换） | 国内直连、Function Calling 稳定、价格便宜、保留多 LLM 对比加分点（D1） |
| **Agent 编排** | 手写 ReAct 循环（不强依赖 LangChain） | 更可控、更好讲故事、便于自定义异常处理 |
| **前端** | Next.js 14 App Router + TypeScript + Tailwind CSS + shadcn/ui | 快速做出现代感聊天 UI + Tool 调用链路可视化（D2） |
| **数据存储** | JSON 文件起步，可升级 SQLite | Mock 数据无需真实数据库 |
| **部署** | 本地 Demo + 录屏 | 不需要上线，跑通本地即可（D8） |

## 四、关键设计取舍

- **不做真实地图 API**：用 Mock 路线时间即可，评委不会扣分，反而省时间。
- **不做真实支付**：所有"下单/预约"都是 Mock 返回成功，用日志体现"已执行"。
- **不追求 LLM 多么强**：评委看的是 Agent 编排能力，不是模型能力。
- **优先 Web UI 而非 CLI**：视觉冲击直接决定演示效果。
- **意图解析层底层全开放 + 演示集保底**（D9 双轨）：不枚举关系类型 / 场景类型，Tool 与 Agent 对场景名称无感；但主压测场景集中在家庭画像跑透，演示现场以 6-8 个压测过的快捷输入作为安全网。
