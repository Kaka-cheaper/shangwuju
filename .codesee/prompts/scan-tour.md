# CodeSee · 导览生成（tour）

> 实验性 prompt。独立于 scan/sync 主流程——install 会注入到 .codesee/prompts/，
> 但只在用户显式要求时调用，绝不进自动扫描链路（这样主流程保持稳定，导览能单独迭代）。
> 用途：在 features.json 已存在且通过校验后，为它生成一条引导式导览（tours 字段）。
> viewer 会把导览播放成"逐步点亮"的旅程：每步先抛问题 → （可选）让用户预测 →
> 揭晓答案叙事 + 节点点亮 + 镜头移动。走完才解锁全图自由探索。
>
> 触发：用户说"生成导览""加个 tour""给新人做个引导"。

---

## 你在写什么

你不是在写文档，是在写**导游词**。读者是一个从没见过这个项目的人，
你的任务是让他走完 6-10 步后，能用自己的话说出系统的主线骨架。

认知约束（这些不是风格偏好，是人脑的硬限制）：

1. **工作记忆只有 4±1 个组块**。一步最多点亮 3 个节点，整条导览 6-10 步。
2. **理解靠序列，不靠全图**。每步的 reveal 要自然引出下一步的 gap，像剧集结尾的钩子。
3. **好奇心是缺口造出来的**（信息缺口理论）。先抛问题再给答案，顺序不能反。
4. **人只记得住自己生成过的东西**（生成效应）。在 1-2 个岔路口放 quiz 让用户预测。

## Schema

见 `_schema.md` 的 Tour 段。写进 features.json 顶层的 `tours` 数组。
所有面向人的文本用 `manifest.lang` 指定的语言。

## 写作流程

### 第 1 步：找主线

读 epics 的 `order` 和 `epic_flow`，找出"用户旅程主线"——
通常是 2-4 个 `importance: core` 的 epic 串成的因果链。
忽略 auxiliary epic（它们最多在尾部出现一步）。

### 第 2 步：写骨架步（steps[0]，最重要的一步）

- `focus`：主线上的 2-3 个核心 epic（**只写 epic id，不混 feature**——
  全 epic 的步会在概览视图播放，Epic 名字/摘要/主线边完整可见）
- `reveal`：**三段论主线**，模板："先 A（…），然后 B（…），最后 C（…）"

骨架步好坏对比：

```
✗ "系统包含认证、订单、支付、通知、管理五大模块。"
   （并列罗列不是 schema——没有方向，挂不住后续细节）

✓ "三段主线：用户先登录拿到身份（认证），挑商品下单（订单），
   最后钱货两清（支付）。其余模块都围着这条线转。"
   （有先后、有因果，后面每一步都知道自己挂在哪）
```

### 第 3 步：沿主线铺步子

每步 1-3 个 feature（写 feature id），顺着骨架的因果方向走。
每步先写 gap 再写 reveal：

**gap 必须是真问题**——制造"我居然没想过这个"的缺口，不是过渡句：

```
✗ "接下来我们看支付模块。"            （陈述句，没有缺口）
✗ "支付模块是怎么工作的？"            （太泛，等于没问）
✓ "订单创建后钱还没扣——系统怎么保证用户跑不掉？"
   （具体、有张力、答案非显然）
✓ "几十个文件的小项目和上千文件的大项目，扫描方式一样吗？"
   （邀请猜测，天然引出分流逻辑）
```

**reveal ≤60 字，有因果方向**：

```
✗ "支付模块负责处理支付相关的逻辑，包括调用第三方接口、
   处理回调、更新订单状态等多个功能点。"     （罗列，无方向）
✓ "下单先冻结库存，支付回调才真正扣减——两段式保证
   '钱没到账货不会少'。"                      （一句话讲清机制+理由）
```

### 第 4 步：放 quiz（1-2 个，不多不少）

放在**用户凭直觉会答错**的地方：条件分支、容错路径、异步行为、反直觉的设计决策。
不出背诵题（"以下哪个是 step 的合法 role"之类）。

- `options`：2-3 个，干扰项必须是"合理但错误"的直觉答案
- `wrong_note`：答错时一句话纠偏，句式"你以为 X，实际 Y——因为 Z"

```
✓ gap: "代码天天改，这张图不会第二天就过时吗？"
  options: ["改完所有代码后一次性同步", "每完成一个逻辑闭环就同步", "用户手动重新全量扫描"]
  answer: 1
  wrong_note: "你以为攒到最后一把同步，实际是检查点协议——图和代码的偏差永远不超过一个检查点。"
```

### 第 5 步：收尾步

最后一步通常落在"质量保障/兜底"类功能（校验、容错），gap 句式：
"如果 AI/用户/网络出了错，谁来兜底？"——给用户一个"这个系统是闭环的"的安全感收束。

## 硬规则

1. focus 只能引用存在的 epic id / feature id（校验器会把断裂引用报 error）
2. 每步 focus ≤3 个节点
3. 整条 6-10 步，第一步必须是骨架步
4. gap 必须含问号
5. reveal ≤60 字
6. quiz 1-2 个，answer 是 0-based 下标
7. 写完跑 `node .codesee/scripts/validate-features.mjs`，错误必须 0

## 自查清单（写完逐条过）

- [ ] 把所有 gap 连起来读一遍：是不是一条有悬念的故事线，而不是目录？
- [ ] 把所有 reveal 连起来读一遍：一个没见过项目的人能否复述出主线？
- [ ] 每个 quiz 的干扰项，是不是"合理的错误直觉"而不是凑数？
- [ ] 有没有哪步的 reveal 涉及了 focus 之外的功能？（有就调整 focus 或文案）
- [ ] 校验器 0 错误
