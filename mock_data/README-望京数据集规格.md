# 望京数据集规格(给数据同学)

目标:演示切换到望京真实感数据。**换数据不换架构**——系统的校验、词典、测试全部建立在下面这些约定上,违反任何一条都会在加载或校验层直接报错或让演示探针翻车。

## 0. 交付形态(最重要)

- **新建目录 `mock_data_wangjing/`,不要原地改 `mock_data/`**——全部单元测试钉着杭州数据的具体数值,原地改=测试全红。演示时用环境变量 `SHANGWUJU_MOCK_DIR` 指向新目录切换。
- 三个文件:`pois.json`、`restaurants.json`、`routes.json`,结构与现有文件**逐字段一致**(以现有文件为模板逐条替换)。

## 1. 保骨架换皮肤(强烈建议,省掉一切联动)

- **保留全部实体 ID**(P001-P0xx / R001-R0xx)和每个 ID 的 `type`/`cuisine` 类目——只换 `name`、`location`(名称+望京实地经纬度)、评论文案、价格微调。
- 这样做的收益:60+ 真实点火探针、演示场景脚本、前端引用全部零改动。

## 2. 硬红线(违反即报错)

- `tags`、`suitable_for`、`cuisine`、`type` **只能用现有词典值**——后端 schema 是白名单校验(schemas/tags.py + 类目等价表),新造词直接被拒。想加新标签先找 Claude 过词典。
- 每家餐厅保留 `reservation_slots` 结构(时段+available+queue_minutes);每个 POI 保留 `capacity`、`suggested_duration_minutes` 分档结构。
- 字段一个都不能少,不确定的字段照抄原值。

## 3. 坐标与距离

- `location.lat/lng` 用望京实地坐标(高德/百度拾取即可)。
- **home 锚点 = 会场**(定死一个,如望京 SOHO T1:39.9962, 116.4818——以实际会场为准),所有 `distance_km` 按"实体到锚点的直线距离"重算(保留 1 位小数)。可以找 Claude 生成校验脚本批量算。
- `routes.json` 里 `home` 相关的边按新锚点重算;实体间的边可以只保留主要组合(缺的边系统会自动用直线×1.3 估算,不会报错)。

## 4. 刻意保留的"坑"(演示的诚实告知桥段靠它们,别好心补齐)

- 烧烤类保持稀缺(触发"你要的烧烤没找到,换了最接近的"话术);
- 部分餐厅时段留缺口/满座(available=false)(触发预约吸附与告知);
- 部分实体不打无障碍标签(触发"没有明确标注,到店前问一下")。

## 5. 新要求(这次实测踩雷后加的)

- **独苗品类配双店**:现在密室逃脱/攀岩/桌游/真人CS 等 type 全库只有 1 家——每个至少配 2 家(用户"换个主题的密室"这类诉求在单店品类上数据性无解)。
- KTV 保持 ≥2 家,火锅/烧烤各 ≥2 家。

## 6. 商户名口味(建议)

用**虚构名+望京地缘感**(如"望京·麦霸欢唱""酒仙桥老王烧烤"),不要用真实品牌——真店名配虚构价格和空位数据,答辩时不好解释。地标类(望京 SOHO、凯德MALL)可以真实出现在 location.name 里。

## 7. 交付验收(做完跑这两条,绿了再交)

```
$env:SHANGWUJU_MOCK_DIR="<新目录绝对路径>"; LLM_PROVIDER=stub uv run --directory backend python scripts/smoke_final_llm.py --stub   # 83 探针应 FAIL 0
```
