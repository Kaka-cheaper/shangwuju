六、这个赛题的核心难点
难点一：不是单点推荐，而是多步骤组合规划
你不能只推荐一个餐厅或一个景点，而要组合成：

活动 + 餐厅 + 路线 + 时间 + 预约 + 转发
这是一个多步骤任务。

难点二：约束很多，而且互相影响
例如：

餐厅好吃，但太远；

活动适合孩子，但附近没有健康餐；

餐厅有位，但时间太晚；

展览适合朋友，但排队时间长；

路线合理，但总时长超过 6 小时。

系统需要综合权衡。

难点三：要体现 Agent 的“决策过程”
比赛不会只看你最终输出一段漂亮文案，更会看：

你有没有设计 Tool；

Tool 调用链路是否合理；

Agent 是否会根据工具返回结果调整计划；

异常时是否有备选方案；

最终是否能完成模拟执行动作。

难点四：Mock API 设计要像真实业务
虽然不接真实美团接口，但 Mock API 不能太假。

比较好的 Mock 数据应该包括：

地点名称、类型、距离、营业时间、评分、适合人群、价格、可预约状态、排队时间、库存、交通时间、标签。
例如餐厅数据不能只有名字，还要有：

是否有位、可预约时间、是否适合减脂、是否亲子友好、人均价格、距离活动地点多远。
否则 Agent 没有判断依据。

七、设计文档应该写什么？
题目要求设计文档不超过 2 页，重点写三件事。

1. Planning 策略
说明你的 Agent 如何规划。

可以写成：

系统采用“目标解析—候选生成—约束过滤—行程组装—执行确认”的分层规划策略。
首先解析用户自然语言目标，识别时间、人数、关系、距离、饮食偏好等约束；
随后分别检索活动和餐厅候选；
再根据距离、时间、适配人群、预约可用性进行过滤；
最后生成 1 条主方案和 1 条备选方案，并在用户确认后调用预订类工具完成执行。
2. 工具调用链路
说明工具怎么串起来。

例如：

parse_intent()
→ get_user_location()
→ search_activities()
→ filter_activities()
→ search_restaurants()
→ check_availability()
→ estimate_route()
→ build_itinerary()
→ reserve_restaurant()
→ book_ticket()
→ generate_share_message()
这部分最好配一个简单流程图或时序图。

3. 异常处理机制
说明如果工具失败怎么办。

例如：

当餐厅无座位时，系统优先尝试相邻时间段；若仍不可用，则切换同商圈备选餐厅。
当活动门票售罄时，系统保留餐厅安排，并替换为同距离范围内的室内活动。
当总行程超过 6 小时时，系统优先删除附加活动，保留核心活动与晚餐。
